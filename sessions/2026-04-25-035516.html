Agent Expert Kit — Design Document

Project: agent-expert-kit

Location: ~/dev/projects/agent-expert-kit

Status: Phase 4 complete (144/144 tests passing). Phase 5 scoped, not yet shipped.

Date: 2026-04-24

Agent: Principal Architect

1. Problem

Claude agents have no persistent memory of a domain across sessions. Each new conversation re-explains architecture, schemas, and conventions from scratch. This wastes tokens, leaks inconsistency between agents, and produces shallow answers when the agent doesn't know what it doesn't know.

Existing CLAUDE.md files conflate personality with knowledge. There is no durable, versioned, validated layer for domain knowledge that stays in sync with code.

2. Solution

Scaffold the Disler "agent expert" triad for any domain: a machine-readable mental model, a read-only Q&A interface, and an explicit re-sync workflow. The triad lives at ~/.pna/experts/<domain>/, is seeded from live code on first run, and stays valid YAML by construction.

Two skills ship the capability:

  • agent-expert-kit — scaffolder; creates a new triad from a domain slug + scope globs
  • self-improve-expertise — re-syncer; diffs current expertise against live code and updates in place

3.2 Scaffolder Flow (agent-expert-kit)

  1. Parse inputs: domain, scope (glob list), description, output_root, max_lines
  2. Slug collision check — if domain already registered in library.yaml, halt
  3. Overwrite gate — if triad exists and overwrite=False, raise; if True, save .bak with timestamp
  4. Expand scope globs to resolved file list
  5. Seed expertise.yaml body via inline seeder prompt (LLM reads scope files, produces YAML body)
  6. Render three templates with $VAR substitution
  7. Validate + trim loop: yaml.safe_load gate + line-cap enforcement, up to 3 retries
  8. Write triad to output_root/<domain>/
  9. Register entry in library.yaml under experts: key
  10. Report: domain, paths, line count, trim attempts, registration status

3.3 Re-Sync Flow (self-improve-expertise)

  1. Parse inputs: domain slug, optional --since=<ref>, optional --focus=<section>
  2. Resolve triad — locate three files; halt if missing
  3. Git diff scoping (if --since provided): identify changed files in scope via git diff --name-only <ref>..HEAD; validate only those files, not full scope
  4. Read current expertise.yaml
  5. Validate documented claims against codebase
  6. Identify discrepancies (schema drift, renamed functions, added/removed tables)
  7. Update expertise.yaml in place to fix them
  8. Enforce line cap (trim as needed)
  9. Compile check: yaml.safe_load must pass — till-done gate
  10. Write .last-sync.md drift report (only after successful compile)

4. Key Components

4.1 SKILL.md Files

agent-expert-kit/SKILL.md is the orchestrator. It carries:

  • YAML frontmatter: name, description (WHEN triggers), argument-hint
  • Inputs table: domain, scope, description, output_root, max_lines
  • 10-step workflow
  • Inline seeder prompt (LLM reads scope files, produces YAML body)
  • Trim prompt (for oversized expertise)
  • Validator code reference (PyYAML only, no extra deps)
  • Guarantees and known limitations

self-improve-expertise/SKILL.md carries the re-sync workflow:

  • Inputs: domain slug, --since=<ref>, --focus=<section>
  • 8-step workflow matching the self-improve.md.tmpl structure
  • Reference to syncer.py helpers
  • .last-sync.md canonical format

4.2 Templates

expertise.yaml.tmpl — canonical mental model structure:

  • Top-level keys: overview, core_implementation, one of key_operations or schema_structure, best_practices, known_issues
  • Seeded conditionally: LLM emits exactly one of the two operation/schema sections
  • All non-trivial claims cite a file path from scope
  • Validated by yaml.safe_load + line-cap loop before write

question.md.tmpl — read-only Q&A interface:

  • Reads expertise, spot-checks 2-3 claims against codebase, responds with citations
  • Permitted tools: Bash, Read, Grep, Glob, TodoWrite (task tracking only)
  • Hard constraint: NO file edits. Read-only.

self-improve.md.tmpl — drift detection + rewrite workflow:

  • Embeds SCOPE_GLOB and MAX_LINES as extractable vars (parsed by syncer.parse_self_improve_vars())
  • 7-step workflow: git diff, read, validate, identify, update, enforce cap, compile check
  • Till-done gate: compile check must pass before .last-sync.md is written

4.3 Python Modules

scaffold.py

  • ScaffoldError — loud failure, surfaces to SKILL.md workflow
  • ScaffoldResult — outcome dataclass with paths, validation metadata, registration status
  • scaffold() — main entry; Phase 2 additions: overwrite gate, registrar hook
  • _render() — $VAR substitution across template strings
  • _expand_scope() — glob expansion relative to repo root

validator.py

  • safe_load_check(text) — parse with yaml.safe_load, raise ValidatorError on failure
  • validate_and_trim(text, max_lines=1000, trim_fn=None, max_retries=3) — full loop
  • ValidationResult — (text, line_count, trim_attempts) dataclass
  • Design constraint: PyYAML only, no new dependencies

registrar.py

  • CollisionError — domain slug already registered
  • check_slug_collision(domain, library_yaml_path) — raise if slug exists
  • register_expert(domain, name, path, library_yaml_path, overwrite, last_sync) — append or update
  • get_registered_slugs(library_yaml_path) — return set of registered slugs
  • Design constraint: surgical string replacement, not full yaml.dump round-trip (preserves non-experts content)

syncer.py

  • Triad — resolved paths for domain triad (expertise, question, self-improve)
  • SelfImproveVars — extracted SCOPE_GLOB, MAX_LINES from self-improve.md
  • RepoRootResult — repo root inference outcome (three-step fallback, ADR-0002)
  • GitDiffResult — changed file list from git diff --name-only
  • LastSyncReport — drift report data model
  • Key functions: resolve_triad(), parse_self_improve_vars(), infer_repo_root(), get_changed_files(), write_last_sync_md(), read_last_sync_date()
  • No LLM invocation; all helpers are pure Python or shell calls

5. Key Decisions

ADR-0001: Domain Triad Pattern

Decision: Adopt the Disler triad as canonical agent memory. Three files, one function each.

Rationale: The pattern is proven in Disler's stack. Small, observable, knowable to work. Splitting into three files (canonical, read, write) keeps each function honest and testable independently.

Rejected alternatives:

  • Cramming into CLAUDE.md — conflates persona with knowledge; hard to validate
  • Free-form Markdown — no validation, no cap, no consistency guarantee
  • Vector stores — retrieval quality dependent, harder to reason about correctness

Key constraints baked in:

  • MAX_LINES=1000 enforced at write time and re-sync time
  • yaml.safe_load compile check at end of every write operation
  • Every seeded claim must cite a file path from scope (seeder constraint; enforced via prompt, not code)
  • Re-sync is explicit, never automatic (v1)

ADR-0002: --since=<ref> Partial Validation

Decision: Add optional --since flag to self-improve-expertise. Scopes validation to git-changed files; always operates on the full expertise.yaml.

Behavior:

  • Step 3: git diff --name-only <ref>..HEAD identifies changed files
  • Validate only the changed files; steps 5-7 still update the complete expertise
  • Unresolvable ref is a hard error, no silent fallback

Repo root inference (three-step fallback):

  1. Walk non-glob prefix of scope upward until .git found
  2. If fails: git rev-parse --show-toplevel from first resolved glob match
  3. If both fail: use cwd, note fallback in drift report

Rationale: Performance on large scopes without correctness impact. Validating only changed files is safe because the till-done compile gate still runs against full expertise.

## Step 0: Expertise Consultation
Before beginning research, check ~/.pna/experts/ for a relevant domain triad.

The shared research rubric gained a sixth dimension:

The standard research output schema gained an "Expertise Consulted" section. This enables A/B comparison of research quality with and without expertise consultation in the loop.

9. Phase 5 Scope (Upcoming)

T5.1: plan-build-improve Skill

A meta-skill that chains three agents in sequence:


planner agent
  → generates build plan from spec
parallel-build agents (N)
  → each implements one task from plan
self-improve-expertise
  → re-syncs domain expertise after build completes

Design notes:

  • Planner reads SPEC.md and TASKS.md from project directory
  • Build agents work on independent tasks (no shared state)
  • Re-sync runs once after all builds complete, not per task
  • Expertise re-sync uses --since flag pointing to pre-build git ref

This is scoped but not yet specced or implemented. Phase 5 opens after research_a/b A/B results are reviewed.

11. Guarantees

  • Output YAML is valid (yaml.safe_load passes) or skill fails loud
  • Output YAML is ≤ max_lines (default 1000) or skill fails after 3 trim passes
  • Every seeded claim cites a file path from scope (seeder prompt constraint)
  • expertise.yaml is only modified when drift is found AND compile check passes (till-done gate)
  • .last-sync.md is only written after successful compile check
  • Bad --since ref is a hard error — no silent fallback to full scan
  • library.yaml non-expert content is preserved (surgical append, no round-trip dump)

12. Non-Goals (v1)

  • Not a doc generator. Output is machine-readable YAML for agent Q&A, not human docs.
  • Not a replacement for CLAUDE.md. Personality stays there. This is the knowledge layer.
  • Not multi-repo. One domain = one repository.
  • Not auto-syncing. Re-sync is explicit, not a file watcher or git hook.
  • No daemon, watcher, or cron.
  • No web UI. All invocation through Claude Code principals.
  • Does not generate YAML from nothing. Expects source files as seeding input.
