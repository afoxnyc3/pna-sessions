NetSuite MCP OAuth — Comparative Analysis + Fix Guide

Overview

Two research agents independently investigated the NetSuite MCP "Your role does not support OAuth2 login" error. Both reached the same root-cause diagnosis via different analytical paths, with meaningful differences in depth, navigation specificity, and source breadth. This document compares their outputs, names a winner, and synthesizes the best combined fix guide.

SWOT Analysis

research_a

Strengths

  • Named the single most actionable root cause first and immediately: "Log in using OAuth 2.0 Access Tokens" is missing from all roles. This is the #1 gate and research_a stated it unambiguously in the Executive Summary rather than burying it.
  • Provided the most precise navigation paths of any output: exact menu trails like Setup → Users/Roles → Manage Roles → [role] → Permissions → Setup subtab → Permission dropdown → "Log in using OAuth 2.0 Access Tokens" are copy-paste ready for implementation tomorrow.
  • Identified a critical user-confusion trap: the similarly named but functionally different "OAuth 2.0 Authorized Applications Management" permission. This is a concrete mistake NetSuite admins commonly make, and calling it out saves a likely misfire.
  • Included a 7-step actionable fix list ordered by probability of resolving the issue, with verification steps after each.
  • Cited 15 sources including the NetSuite Professionals community thread (Source 12) which provides empirical confirmation that REST Web Services (Full) + Perform Search are required for tools/list completeness — a post-auth failure mode not documented by Oracle.
  • Included the Tim Dietrich Claude Code-specific bug citation as a valid exclusion criterion (rules out a confound if the user is on claude.ai web).
  • SAML SSO edge case noted — a rare but legitimate escalation path.

Weaknesses

  • The Feynman Bridge metaphor (three locks) is solid but requires more setup to grasp than research_b's nightclub bouncer metaphor.
  • Fewer sources cited from Oracle's error-specific sub-pages (e.g., subsect_161252543690 for the Login Audit Trail error catalog) compared to research_b.
  • Confidence gap on the literal error string mapping is acknowledged but the path from string to permission is slightly less explicitly grounded.

Opportunities

  • The dedicated role template is the most complete of the two outputs, including the _claude_mcp_integration internal ID suggestion and the explicit "Do NOT grant Create/Edit/Full" instruction per record type.

Threats

  • Marginal risk that the "Log in using OAuth 2.0 Access Tokens" fix alone is sufficient and the REST Web Services scope change is redundant — but both changes are safe to make together, so this is low stakes.

research_b

Strengths

  • Strongest root-cause framing: clearly articulated the two-pronged failure mode (scope mismatch on integration record + missing role permissions) as jointly sufficient to produce the error. The explanation of why Administrator fails despite its power is the most cogent of the two outputs.
  • The nightclub bouncer Feynman Bridge is the clearest plain-language explanation of the OAuth scope-to-role-permission matching mechanics.
  • Explicitly documented that the -System- auto-creation is intentional (not a wizard failure) and that no post-install wizard was missed — this directly addresses a specific anxiety from the problem statement.
  • Login Audit Trail diagnostic is more prominently positioned (Step 3 in the actionable list) and specifies the exact error codes to look for (ScopeMismatched, AuthorizationCodeGrantRequired, RoleNotAllowed, UserCredentialsRequired).
  • Explicitly noted that Administrator's permissions "are locked" when testing edits — a practical UI gotcha for anyone who tries to edit the Administrator role directly.
  • The raw authorize URL test at the end of the Confidence section is a useful diagnostic tool not present in research_a.

Weaknesses

  • Less granular navigation paths for the role permission fix. Step 2 says to navigate to Manage Roles and check "REST Web Services (Full) and Log in using OAuth 2.0 Access Tokens (Full)" but does not give the exact sub-tab path or warn about the similarly named permissions.
  • The dedicated role template is less detailed: omits the internal ID field, misses the Device ID restriction checkbox, and recommends unchecking 2FA initially without a re-enable instruction.
  • 13 sources vs. research_a's 15; missing the NetSuite Professionals community thread that empirically confirms the tools/list completeness issue.
  • Steps 4 and 5 are partially redundant (both address the integration record) and could be consolidated.

Opportunities

  • The explicit 2FA-disable-then-re-enable sequence (Step 6) is clearer than research_a's handling of the same issue.

Threats

  • The "two-pronged fix" framing could lead an implementer to batch both changes before testing, making it harder to isolate which change resolved the issue.

Combined Recommendation: Best Step-by-Step Fix

This guide synthesizes the strongest elements from both research outputs. Execute in order. Test after each step before proceeding.

Before you start: capture a baseline

Navigate to Setup → Users/Roles → View Login Audit Trail. Filter: Date = Today, Detail contains "OAuth". Make a note of the error code in the Detail column from the last failed attempt. The codes map as follows:

  • ScopeMismatched — integration record scope does not cover the role's permissions
  • AuthorizationCodeGrantRequired — Auth Code Grant checkbox off on the integration record
  • RoleNotAllowed / role rejection — the role lacks the OAuth 2.0 login permission
  • InvalidRedirectUri — redirect URI mismatch

Both research agents agree the current failure is most likely ScopeMismatched combined with a role lacking the login permission.

Step 1: Add "Log in using OAuth 2.0 Access Tokens" to the test role (highest likelihood fix)

This is the role-level gate that makes a role OAuth 2.0 eligible. It is separate from "OAuth 2.0 Authorized Applications Management" — that permission only lets users view/revoke apps, not log in via OAuth. Adding the wrong one is a common mistake; make sure you add the right one.

Navigation: Setup → Users/Roles → Manage Roles

For each role you want to test with (start with Administrator to prove the flow works, even though Administrator's permissions panel is read-only — use a custom role in production):

  1. Click Edit next to the role
  2. Click the Permissions tab
  3. Click the Setup subtab
  4. In the Permission dropdown, select "Log in using OAuth 2.0 Access Tokens"
  5. Leave Level blank (this permission has no level)
  6. Click Add
  7. Click Save

Repeat for CP Executive and any other role to be authorized.

Test: log out, log back in, go to claude.ai → Settings → Connectors → re-add NetSuite. If the role picker accepts the role, this was the fix. Stop here and move to Step 4 (dedicated role) for production hardening.

Step 2: Add REST Web Services scope to the integration record

Even after Step 1 succeeds, REST Web Services scope is required on the integration record for MCP tools (record retrieval, SuiteQL, saved search) to actually function. The auto-created -System- integration record only has "NetSuite AI Connector" scope by default; this is intentional but incomplete for tool execution.

Navigation: Setup → Integration → Manage Integrations

  1. Find the auto-created integration record (likely named "NetSuite AI Connector" or "MCP Standard Tools")
  2. Click Edit
  3. On the Authentication tab, confirm:

- State = Enabled

- Authorization Code Grant = checked

- Public Client = checked (PKCE flow — no client secret needed)

- Redirect URI = https://claude.ai/api/mcp/auth_callback (exact match, no trailing slash)

  1. Under Scope, check REST Web Services (in addition to NetSuite AI Connector that is already checked)
  2. Leave RESTlets unchecked unless you specifically need it
  3. Click Save

Test: re-authorize Claude and attempt a tool call (e.g., ask Claude to look up a customer record).

Step 3: Confirm "REST Web Services" feature is enabled at the account level

Navigation: Setup → Company → Enable Features → SuiteCloud subtab

Confirm both "REST Web Services" and "OAUTH 2.0" checkboxes are checked. If REST Web Services was not previously enabled, enabling it may prompt a page reload and require re-saving.

Step 4: Create a dedicated "Claude MCP Integration" role (do before production use)

Using Administrator or CP Executive long-term is overpermissioned. Create a minimum-privilege role:

Navigation: Setup → Users/Roles → Manage Roles → New

Configure the role header:

  • Name: Claude MCP Integration
  • ID: _claude_mcp_integration
  • Two-Factor Authentication Required: checked (keep 2FA; temporarily uncheck only if you need to isolate a 2FA-specific failure)
  • Web Services Only: UNCHECKED (critical: Web Services Only blocks the browser-based OAuth consent flow and also blocks RESTlets used by MCP tools)
  • Restrict this role by Device ID: UNCHECKED

Permissions tab — Setup subtab:

  • Log in using OAuth 2.0 Access Tokens — (no level)
  • REST Web Services — Full
  • User Access Tokens — Full (only if mixing TBA flows; safe to omit for pure OAuth 2.0)

Permissions tab — Lists subtab:

  • Perform Search — Full (required for Saved Search and SuiteQL MCP tools; View is minimum)
  • Documents and Files — View
  • Per record type (Customers, Vendors, Employees, Items, etc.) — View only. Do not grant Create/Edit/Full unless you specifically intend Claude to write records.

Permissions tab — Reports subtab:

  • Report Customization — View
  • Specific report types as needed — View

Permissions tab — Transactions subtab:

  • Transaction types Claude should read (Sales Order, Invoice, Purchase Order) — View only

Click Save.

Assign the role: Lists → Employees → Employees → [employee] → Edit → Access tab → Roles subtab → New Role → Claude MCP Integration → Save

Step 5: Verify 2FA enrollment

The OAuth 2.0 Authorized Applications Management permission requires 2FA (enforced since NetSuite 2018.2). If the user has not enrolled 2FA on the Claude MCP Integration role, the consent flow will interrupt with a 2FA setup prompt.

Log in as the target user with the Claude MCP Integration role. Navigate to the top-right user menu → Settings → Two-Factor Authentication. Confirm 2FA is enrolled (authenticator app or hardware token).

Step 6: Rule out "Web Services Only" on any tested roles

Navigation: Setup → Users/Roles → Manage Roles

Edit each role that was tested. Near the top of the form, confirm the Web Services Only checkbox is unchecked. This restriction routes the role to SOAP web services only, which is incompatible with both the browser OAuth consent flow and RESTlets used by MCP tools.

Step 7: Check for IP restrictions

Navigation: Setup → Company → Company Information — check "Allowed IPs" at company level.

Navigation: Setup → Users/Roles → Manage Roles → [role] → Edit → Restrictions tab — check for role-level IP restrictions.

If any IP allowlist is active, it must cover both the user's browser IP (for the consent page) and Claude's backend IPs for the token exchange callback. Anthropic does not publish a static IP range for claude.ai's MCP backend. The cleanest remediation is to remove IP restrictions from the Claude MCP Integration role only, not from production roles.

Diagnostic shortcut (if above steps don't resolve it)

Test the raw authorize URL directly in your browser:


https://<accountID>.app.netsuite.com/app/login/oauth2/authorize.nl?response_type=code&client_id=<consumer_key>&redirect_uri=https%3A%2F%2Fclaude.ai%2Fapi%2Fmcp%2Fauth_callback&scope=rest_webservices&state=test&code_challenge=<challenge>&code_challenge_method=S256

If this URL produces a consent screen that accepts your role, the problem is definitively scope-on-integration-record and Step 2 is the fix. If it still rejects, the problem is the role permission (Step 1).

If all steps pass but tools still fail

Check tools/list completeness. The following permissions on the Claude MCP Integration role are empirically required for the full tool catalog to appear (confirmed by NetSuite Professionals community, October 2025):

  • REST Web Services — Full
  • Perform Search — Full or View
  • View access to each record type you want Claude to query

Confidence

Both research agents rated this analysis high confidence based on Oracle primary documentation. The single acknowledged gap is that the literal UI string "Your role does not support OAuth2 login" is not reproduced verbatim in Oracle's public Help Center, making the mapping from string to error code to fix inferential rather than directly cited. The Login Audit Trail step will confirm the exact error code and validate the diagnosis.

If Steps 1 through 7 are completed and the issue persists, the escalation path is: open a NetSuite Support case referencing the Login Audit Trail error code, or verify whether your account uses SAML SSO (which routes OAuth consent through an IdP and adds a separate configuration layer).
