Summary

The first production A/B run of the research cluster (research_a + research_b) completed overnight on 2026-04-20 but returned results with no live sources. All three external tools failed silently: Exa web search threw a parameter error, and the feynman and alpha CLIs were invisible to the daemons because launchd was running with a stripped PATH. Both agents fell back to training-knowledge inference and self-reported low/medium confidence. The results were still structurally valid and contained useful reasoning, but they could not be considered cited research.

Detection was immediate on morning review of mission results. Resolution took approximately 30 minutes.

Root Causes

RCA-1: exa-py v2.12.0 removed use_autoprompt

The exa-py SDK breaking change dropped the use_autoprompt parameter from search_and_contents() in v2.12.0. The parameter was carried over from the original scaffold without a version pin. The call silently failed with a ValueError at runtime — not a build error — so it was undetected until the first live mission.

Affected: research_b (tools.py:web_research)

RCA-2: launchd PATH missing fnm bin directory

Both launchd plists had PATH set to /opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin. The feynman CLI binary lives at ~/.local/share/fnm/node-versions/v24.13.0/installation/bin/feynman — a path outside any standard bin and never included in the plist template. Processes spawned by launchd do not inherit the user's shell PATH, so feynman was a FileNotFoundError inside both daemons.

Affected: research_a (deepagents worker), research_b (Claude SDK daemon)

RCA-3: alpha CLI never symlinked

alpha ships bundled inside feynman's node_modules at a deeply nested path. It was never symlinked to a standard bin during feynman's install. Even with the correct fnm PATH, alpha would still have been invisible. This was a missing step in the install checklist — no code ever failed, it just was never done.

Affected: research_a (alpha-research skill), research_b (would have been affected if feynman PATH was fixed)

Impact

  • research_b result: confidence low — pure inference, zero live sources, no URLs in Source Ledger
  • research_a result: confidence medium — training-knowledge citations (canonical GitHub repos, arXiv IDs), no live verification
  • Downstream: Engineer mission (0fb10bbd) collected both results and published to pna-sessions + ran SWOT comparison. The comparison named research_a the winner on structural quality, which is valid, but neither result should be treated as live-cited research
  • No data loss, no service outage, no user-facing breakage. Impact was confined to result quality on a single research query

Fixes Applied

Fix 1 — Remove use_autoprompt from tools.py


# Before
response = _exa_client.search_and_contents(
    query,
    num_results=min(num_results, 20),
    type="auto",
    use_autoprompt=True,        # invalid in exa-py >= 2.12.0
    text={"max_characters": 800},
)

# After
response = _exa_client.search_and_contents(
    query,
    num_results=min(num_results, 20),
    type="auto",
    text={"max_characters": 800},
)

Verified: web_research("Claude Code skill files", num_results=2) returned 2 results successfully.

Fix 2 — Add fnm bin to PATH in both launchd plists

Prevention

P1 — Pin exa-py with upper bound in pyproject.toml


# research-agent-sdk/pyproject.toml
exa-py = ">=2.12.0,<3.0.0"

Add a test that calls web_research() with a live API key as part of the CI smoke test suite. Breaking changes will fail at uv sync time, not at runtime.

P2 — Standardise launchd PATH in plist design template

The fnm bin path must be first in every plist that runs Node.js CLIs. Update the plist template in agent-design/ with a comment block:


<!-- PATH must include fnm bin for feynman, alpha, and any fnm-managed CLIs.
     Update the fnm version segment when upgrading Node. -->
<string>/Users/__HOME__/.local/share/fnm/node-versions/v24.13.0/installation/bin:/opt/homebrew/bin:...</string>

P3 — Add feynman install checklist to CLAUDE.md

When feynman is installed or upgraded, the alpha symlink must be recreated. Add to setup instructions:


# After: npm install -g feynman (or upgrade)
FNM_BIN=$(fnm exec --using=default -- node -e "console.log(process.execPath)" | xargs dirname)
ALPHA=$(node -e "require.resolve('@companion-ai/alpha-hub/bin/alpha')" 2>/dev/null || \
  find "$FNM_BIN/../lib" -name alpha -type f | head -1)
ln -sf "$ALPHA" "$FNM_BIN/alpha"

P4 — Pre-flight tool-check before research missions start

Add a preflight_check() function to both research daemons. Before claiming a mission:


def preflight_check() -> list[str]:
    """Returns list of failed tool names. Empty = all clear."""
    failures = []
    if not shutil.which("feynman"):
        failures.append("feynman")
    if not shutil.which("alpha"):
        failures.append("alpha")
    if _exa_client is None:
        failures.append("exa")
    return failures

If preflight_check() returns non-empty, log a warning and fail the mission immediately rather than completing with low confidence. This surfaces infra issues as clear errors rather than degraded silent output.

P5 — Tag re-run results as "retry" in mission title

When re-running a failed mission, prefix the title with [retry] so the SWOT comparison can distinguish first-run from re-run results.

Lessons

The core failure mode here is silent degradation — both agents completed their missions, returned valid schema output, and self-reported low confidence, but nothing in the orchestration layer flagged this as an infrastructure failure requiring attention. A research agent completing with zero sources and low confidence should trigger an alert, not just a polite disclaimer in the output.

The fix for this is preflight checks (P4) and surfacing tool failure as a mission error rather than a quality flag in the result body.
