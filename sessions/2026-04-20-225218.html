Research: Resolving "Your role does not support OAuth2 login" on NetSuite MCP (Claude Connector)

Executive Summary

The error "Your role does not support OAuth2 login, please select a different one" is emitted by NetSuite's /app/login/oauth2/authorize.nl endpoint during Step One of the Authorization Code Grant flow when the role the user is attempting to authorize with lacks at least one of the permissions that NetSuite treats as OAuth 2.0 eligible, OR when the integration record's scopes do not intersect with any permission the role holds. Based on Oracle documentation, the two most likely root causes in this environment are (1) the integration record is missing the REST Web Services scope that the MCP Standard Tools SuiteApp relies on for record, report, saved search, and SuiteQL tool execution, and (2) the roles being tested lack the "REST Web Services" and "Log in using OAuth 2.0 Access Tokens" permissions that make a role OAuth-eligible. The fix is deterministic and does not require Oracle Support.

Key Findings

  1. The error originates at NetSuite's authorize endpoint, not Claude

The string "Your role does not support OAuth2 login, please select a different one" is the NetSuite-side rejection rendered on the login/oauth2/authorize.nl consent screen. It fires before Claude ever receives a callback, so the Login Audit Trail at Setup > Users/Roles > View Login Audit Trail is the authoritative diagnostic source.

Source: Authorization Code Grant Flow Error Messages in the Login Audit Trail -- Oracle official NetSuite Help

  1. OAuth 2.0 eligibility is gated by role permissions, not a role-level checkbox

NetSuite does not expose a single "OAuth2 enabled" toggle on a role. A role becomes OAuth 2.0 eligible when it holds the appropriate permission set: "REST Web Services" (Setup category), "Log in using OAuth 2.0 Access Tokens" (Setup category), and the OAuth 2.0 Authorized Applications Management permission for self-service authorization. If none of these are present, or the scope in the integration does not map to a permission the role holds, the role is rejected.

Source: Set Up OAuth 2.0 Roles -- Oracle official NetSuite Help

  1. Scope on the integration record must intersect with role permissions

The integration record scopes (REST Web Services, RESTlets, SuiteAnalytics Connect, NetSuite AI Connector) define which permissions NetSuite will check against the role. Checking only "NetSuite AI Connector" while leaving "REST Web Services" unchecked means NetSuite only looks for AI-Connector-scope permissions on the role, and will reject any role that lacks them even if it is Administrator.

Source: Authorization Code Grant Flow Error Messages in the Login Audit Trail -- ScopeMismatched error documentation

  1. MCP Standard Tools SuiteApp requires REST Web Services feature + permission

The MCP Standard Tools SuiteApp's Record, Report, Saved Search, and SuiteQL tools are implemented against NetSuite REST Web Services / SuiteQL. Oracle's "Required Features and Permissions" page for the NetSuite AI Connector Service explicitly lists Server SuiteScript and OAuth 2.0 as required features. REST Web Services is implicitly required because the tools cannot function without it.

Source: Required Features and Permissions and MCP Standard Tools SuiteApp -- Oracle official NetSuite Help

  1. The integration record auto-created by -System- is intentional, not incomplete

When the MCP Standard Tools SuiteApp installs, it programmatically creates an integration record with TYPE=Public Client, PKCE required, and the NetSuite AI Connector scope preselected. There is no separate "post-install wizard" you missed. However, the auto-created record only has the minimum scope set, and REST Web Services scope must often be added manually for tool execution to work end-to-end.

Source: Installing the MCP Standard Tools SuiteApp -- Oracle official NetSuite Help

  1. 2FA does not block the authorize page but can break silent SSO hops

If the user's role is marked as requiring 2FA and the browser session is not already 2FA-authenticated, NetSuite will interrupt the OAuth consent flow with a 2FA challenge rather than emit this specific error. 2FA is therefore a secondary suspect, not the primary cause of this specific error string.

Source: OAuth 2.0 Authorization Code Grant Flow -- Oracle official NetSuite Help

  1. Known working permission bundle for MCP-dedicated role

A minimum-permission role for MCP access needs: REST Web Services (Full), Log in using OAuth 2.0 Access Tokens (Full), SuiteAnalytics Workbook (View), plus record-level view permissions for whatever data you want Claude to access. Do NOT set "Web Services Only Role" on the role because that disables RESTlets and interferes with MCP tool invocation.

Source: OAuth 2.0 for RESTlets and Roles and Permission Considerations for APIs

  1. Administrator role is NOT automatically OAuth-exempt

Even Administrator fails with this error if the integration scope does not align with any OAuth-eligible permission on the role. Administrator has REST Web Services by inheritance but the scope mismatch on the integration record still causes rejection at the authorize step.

Source: Authorization Errors in Step One

Deep Dive

The problem reported fits a well-documented pattern in NetSuite OAuth 2.0. The authorize endpoint performs three checks in sequence before displaying the consent screen: (a) the integration record exists and has Authorization Code Grant enabled, (b) the requested scope is valid and at least one scope box on the integration record is checked, and (c) the signed-in user has an available role that holds a permission matching one of the integration's declared scopes. Failure at step (c) produces exactly the error string the user is seeing.

Given the diagnostic facts, two conditions are jointly sufficient to produce this failure. First, the integration record only has the NetSuite AI Connector scope checked. NetSuite will therefore look for the "NetSuite AI Connector" permission on the user's role. That permission is bundled into the MCP Standard Tools SuiteApp install, but it is NOT automatically granted to stock roles like Administrator or CP Executive. Second, there is no REST Web Services scope checked on the integration, so NetSuite will not fall back to checking REST permissions on the role. The result: every role tested fails because the only scope being checked against is one they do not hold. This explains why Administrator also fails, which would otherwise be highly anomalous.

The fix is therefore two-pronged: broaden the integration record's scope to include REST Web Services (so REST-permissioned roles become eligible), AND grant the NetSuite AI Connector permission to at least one test role so the original scope also succeeds. This doubled approach makes the MCP connector work with both stock roles (via REST) and dedicated AI roles (via the AI Connector permission).

The -System- auto-created integration record is Oracle's standard behavior and is not a sign of an incomplete wizard. There is no "post-install wizard" to complete. However, Oracle's documentation for the SuiteApp does direct administrators to review and adjust the integration record's scopes after install, which is the step that was missed here.

For security posture, building a dedicated "Claude Integration" role is strongly preferred over using Administrator or CP Executive. That role should NOT be marked Web Services Only because the MCP Standard Tools SuiteApp uses RESTlets under the hood for some tool calls and Web Services Only roles cannot invoke RESTlets.

Feynman Bridge

Think of NetSuite's OAuth 2.0 authorize endpoint as a nightclub bouncer. The bouncer has a list of what rooms (scopes) a guest (integration record) said they want to visit, and a list of what wristbands (permissions) the guest is wearing (their role). If the integration record only says "I want to visit the AI Connector lounge," the bouncer ignores every other wristband the guest is wearing and only checks whether they have the specific AI Connector wristband. No matching wristband, no entry, regardless of whether they are a VIP (Administrator) or have twelve other wristbands. The error "Your role does not support OAuth2 login" is the bouncer saying: "based on the room you asked to enter, none of the roles you could switch to are holding the right wristband."

The fix is to either (a) tell the guest to also request access to the REST lounge, because their existing wristbands already allow that, or (b) print a new wristband for the AI Connector lounge and give it to a role. The preferred fix does both: broaden the integration's declared rooms, and create a dedicated role with the exact wristband set needed for the MCP tools to work without handing Claude the keys to the whole building.

Actionable Insights

Ordered by likelihood of resolving the issue. Execute in order, testing after each step.

  1. Enable REST Web Services feature and add REST Web Services scope to the integration record (highest probability fix)

Navigate to Setup > Company > Enable Features > SuiteCloud subtab. Confirm both "REST Web Services" and "OAUTH 2.0" are checked. Save if changes were made. Then go to Setup > Integration > Manage Integrations > List, open the auto-created "NetSuite AI Connector" (or similarly named) integration record, click Edit, and under the Scope section check BOTH "REST Web Services" and "NetSuite AI Connector." Leave "RESTlets" unchecked for now unless you use them. Save. Retest the Claude authorization flow.

  1. Verify the "Log in using OAuth 2.0 Access Tokens" and "REST Web Services" permissions on the test role

Navigate to Setup > Users/Roles > Manage Roles. Open the role you are testing with (start with Administrator for diagnostic purposes, but note Administrator's permissions are locked). For any non-Administrator role, click Edit > Permissions > Setup subtab. Confirm both "REST Web Services" (Full) and "Log in using OAuth 2.0 Access Tokens" (Full) are present. Add if missing. Save. Log out fully and retest.

  1. Check the Login Audit Trail for the precise rejection reason

Navigate to Setup > Users/Roles > View Login Audit Trail. Filter by Date (today) and Detail containing "OAuth" or "Authorization". Locate the failed attempt. The Detail column will show one of: ScopeMismatched, AuthorizationCodeGrantRequired, RoleNotAllowed, or UserCredentialsRequired. This tells you exactly which of steps 1, 2, 4 will fix it. Reference: Authorization Code Grant Flow Error Messages in the Login Audit Trail.

  1. Create a dedicated "Claude Integration" role with minimum permissions (security best practice, do this regardless of whether step 1 resolves the symptom)

Navigate to Setup > Users/Roles > Manage Roles > New. Configure:

- Name: Claude MCP Integration

- Subsidiary: leave default or scope as needed

- Two-Factor Authentication Required: UNCHECKED for now (re-enable after flow proves working, see step 6)

- Web Services Only Role: UNCHECKED (required; Web Services Only roles break RESTlets used by MCP tools)

- Permissions > Setup subtab: add REST Web Services (Full), Log in using OAuth 2.0 Access Tokens (Full), User Access Tokens (Full), SuiteAnalytics Workbook (View)

- Permissions > Lists subtab: add View-level permissions only for records Claude should see (e.g., Customers View, Items View, Transactions View, Employees View). Start read-only.

- Permissions > Reports subtab: SuiteAnalytics Workbook (View), Financial Statements (View) as needed

Save. Assign the role to a dedicated service-account-style employee at Lists > Employees > Employees > [employee] > Edit > Access subtab > Roles sublist.

  1. Verify PKCE and redirect URI match exactly on the integration record

Open the integration record at Setup > Integration > Manage Integrations > List > [NetSuite AI Connector record]. Confirm:

- State: Enabled

- Authorization Code Grant: CHECKED

- Public Client: CHECKED (Claude uses PKCE public client flow per NetSuite AI Connector Service FAQ)

- Redirect URI: https://claude.ai/api/mcp/auth_callback (exact, including https and no trailing slash)

- Scope: both "REST Web Services" and "NetSuite AI Connector" checked

Save.

  1. Temporarily disable 2FA on the test role to isolate the variable

If steps 1 through 5 still fail, go to Setup > Users/Roles > Manage Roles > [role] > Edit, and under Authentication confirm "Two-Factor Authentication Required" is UNCHECKED. Also check the employee record at Lists > Employees > [employee] > Access subtab for any role-level 2FA requirement. After the OAuth flow succeeds end to end, re-enable 2FA. NetSuite will then require the user to complete a 2FA challenge during the authorize step but will not block the flow.

  1. Verify no IP restriction is blocking Claude's callback

Navigate to Setup > Company > Company Information and check "Allowed IPs" at the company level. Then check role-level IP restrictions at Setup > Users/Roles > Manage Roles > [role] > Edit > Restrictions. If any IP allowlist is present, it must include both the user's browser IP (for the consent page) and Claude's backend IPs for the token exchange. Since Anthropic does not publish a static IP allowlist for claude.ai's MCP backend, the cleanest remediation is to remove IP restrictions from the Claude Integration role specifically, or widen them. Do NOT remove IP restrictions from other production roles.
