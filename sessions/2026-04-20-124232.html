Key Findings

  1. Transcription is a near-solved commodity layer, but code-aware transcription is not

Whisper (large-v3 or distil-whisper), AssemblyAI, Deepgram Nova, and YouTube auto-captions all produce serviceable prose transcripts. None of them reliably transcribe spoken shell commands, file paths, or function names as code. For SKILL.md extraction this is the single biggest quality bottleneck: a developer saying "run pnpm dlx shadcn add button" gets rendered as prose mush.

Source: inference from public docs of Whisper, AssemblyAI, Deepgram (could not verify in this session).

  1. The task splits cleanly into two sub-problems that the research brief already identified

Transcript-to-procedure extraction is fundamentally different from transcript-to-explanation. Procedure extraction needs ordered steps, preconditions, commands, and decision points. Explanation extraction (Feynman-style) needs analogies and conceptual scaffolding. A SKILL.md wants procedure with just enough explanation to cue the agent on when to invoke the skill. These need different prompts and arguably different passes.

Source: inference from general LLM pipeline design patterns.

  1. Long-transcript handling has a dominant pattern: semantic chunking + map-reduce + refine

For a one-hour talk (roughly 9000-12000 words, well under Claude's context), chunking is unnecessary. For multi-hour podcasts or course series, the standard pattern is semantic chunking on topic-shift boundaries, a map pass that extracts candidate procedures per chunk, and a reduce pass that merges and deduplicates into a single skill (or splits into multiple skills). Timestamp anchors should be carried through every stage.

Source: inference from LangChain/LlamaIndex documented patterns.

  1. Claude Code skills have a published convention (as of late 2024/2025)

A SKILL.md is a Markdown file with YAML frontmatter containing at minimum name and description, placed in .claude/skills/<skill-name>/SKILL.md. The description is the trigger surface: Claude uses it to decide when to load the skill. The body is free-form Markdown, often including invocation conditions, a numbered protocol, example inputs/outputs, and constraints. Could not verify the exact current schema in this session.

Source: inference; needs verification against Anthropic's current skills documentation.

  1. No verified prior art for "agentic skill extraction from video"

There is substantial work on video summarization, lecture-to-notes, and meeting-minutes extraction. There is also growing work on skill libraries for agents (Voyager, SkillWeaver, etc.). I am not aware of a published tool that bridges the two specifically for Claude Code skills. Could not verify a null result without working search.

Deep Dive

Transcription layer. The realistic options fall into three tiers. Tier one: YouTube auto-captions via yt-dlp --write-auto-sub --sub-format vtt --skip-download. Free, instant, but no diarization, variable quality, and often missing for podcasts. Tier two: local Whisper (whisper.cpp, faster-whisper, or distil-whisper) on downloaded audio via yt-dlp -x --audio-format mp3. Zero marginal cost, good quality with large-v3, no diarization out of the box but whisperx adds forced alignment and pyannote-based diarization. Tier three: API (AssemblyAI, Deepgram Nova-2, Rev). Best diarization, word-level timestamps, custom vocabulary (critical for code terms like "pnpm", "tsx", "pydantic"), but costs roughly $0.25 to $1.00 per hour of audio. For skill extraction, tier three's custom-vocabulary feature is the key differentiator: you can seed it with the CLI tools and library names the speaker is likely to use.

Extraction layer. The prompt pattern that tends to work for procedure extraction is a two-pass structure. Pass one is a "scaffolding" extraction: given the transcript, list every distinct workflow, methodology, or protocol the speaker describes, each with a short description, rough start/end timestamps, and a trigger phrase. This produces a table of candidate skills. Pass two is a "densification" extraction run once per candidate: given the relevant transcript slice plus the scaffolding entry, produce a full SKILL.md draft. This beats single-pass because single-pass tends to over-generalise (turning specific tool protocols into bland advice) or over-specify (transcribing everything the speaker said verbatim). Chunking for long content: split on VTT paragraph breaks or on silence gaps longer than 2 seconds, target 2000-3000 token chunks with 200 token overlap, carry timestamps as metadata into every chunk. Map-reduce merges candidate skills by fuzzy name match; a final dedup pass asks the LLM to identify which candidates are actually the same skill described twice.

Skill structure. A defensible SKILL.md template derived from the brief: YAML frontmatter with name, description (trigger surface), source_url, source_timestamp_range, extracted_from, extraction_confidence. Body sections: "When to use" (trigger conditions in agent-voice), "Protocol" (numbered imperative steps), "Inputs required", "Outputs produced", "Examples" (pulled from the transcript verbatim where possible), "Constraints and gotchas" (the "watch out for" asides speakers give), "Source anchors" (timestamped quotes backing each non-obvious step). The "Source anchors" section is the validation mechanism: every non-trivial instruction in the protocol should be traceable to a timestamp, and a validation pass can check that every step has at least one anchor.

Voice preservation vs agent-executability. The tension is real. The speaker says "I usually just kind of vibe it and see what feels right"; the agent needs "If X then Y." The resolution used in similar pipelines: keep speaker voice in the "Examples" and "Source anchors" sections (verbatim quotes), rewrite ruthlessly in the "Protocol" section into imperative agent-executable steps. A final "voice check" prompt asks the LLM: does the protocol contradict anything the speaker said? This catches the common failure mode where the LLM invents plausible-sounding steps the speaker never described.

End-to-end pipeline. Given the stated stack (claudeclaw, Claude Code, feynman, deepagents), the simplest shippable version is a Claude Code skill that wraps a shell pipeline: yt-dlp for acquisition, whisperx or Deepgram for transcription, a deepagents subagent for scaffolding extraction, a parallel fan-out of extraction agents (one per candidate skill) for densification, and a final validation subagent that checks source-anchor coverage. feynman can be invoked on any concept the speaker references but does not define, to enrich the "When to use" section.

Feynman Bridge

Think of this pipeline as turning a cooking show into a recipe card. The video is a chef rambling about three dishes, tasting as they go, telling stories. Transcription is writing down everything they said. But a recipe card is not a transcript: it is ingredients, steps, times, temperatures. The extraction layer is the step where a careful listener watches the show and writes the recipe, throwing away the stories but keeping the one aside where the chef warns "do not skip resting the dough." The SKILL.md is that recipe card, written for a cook (the agent) who will never see the original show. The validation step is the equivalent of watching the show again with the recipe in hand and checking: does every instruction on my card actually come from something the chef did or said? If yes, the card is trustworthy. If some steps are invented, the card is a hallucination pretending to be a recipe.

Actionable Insights

  1. Ship a minimum viable pipeline first, then harden. Minimum viable: yt-dlp --write-auto-sub --skip-download + a single Claude prompt that outputs one SKILL.md. This lets you feel the failure modes before investing in transcription quality or multi-pass extraction.
  1. Use the two-pass scaffolding + densification pattern, not single-pass. Pass one enumerates candidate skills across the transcript with timestamp ranges. Pass two fills in each one from its local slice. This is the single biggest quality lever after transcription quality.
  1. Invest in transcription custom vocabulary if the content is technical. Seed AssemblyAI or Deepgram with a list of CLI tools, library names, and jargon from the expected domain. This eliminates the dominant error class (misheard code terms) that no amount of downstream prompting can recover.
  1. Enforce source-anchor coverage as your validation contract. Every non-trivial step in the protocol must cite a [mm:ss] timestamp from the transcript. Reject drafts where coverage drops below some threshold (start at 80 percent). This converts hallucination detection into a mechanical check.
  1. Proposed concrete pipeline:

- Input: URL (YouTube, podcast RSS item, arbitrary MP4/MP3).

- Step 1 Acquire: yt-dlp -x --audio-format mp3 --write-info-json <url>.

- Step 2 Transcribe: whisperx <audio> --model large-v3 --diarize --highlight_words True for local, or Deepgram Nova-2 API with custom vocabulary for higher-value runs. Output: timestamped, speaker-labelled transcript.

- Step 3 Scaffold: single LLM call with the full transcript (or map-reduced summary if over 150k tokens), prompt: "List every distinct workflow or protocol the speaker describes. For each: name, one-sentence description, trigger phrase, start/end timestamps."

- Step 4 Densify (fan-out): for each scaffold entry, one LLM call with the relevant transcript slice, prompt: produce a full SKILL.md matching the template in the Deep Dive. Parallelise via deepagents subagents.

- Step 5 Validate: one LLM call per draft, prompt: "For each numbered protocol step, cite the transcript timestamp that supports it, or flag as UNSUPPORTED." Reject or rewrite drafts with too many UNSUPPORTED flags.

- Step 6 Emit: write to .claude/skills/<kebab-name>/SKILL.md, with source_url, extracted_at, and extraction_confidence in frontmatter.

  1. Keep a "rejected scaffolds" log. Candidate skills that fail validation still carry signal about what the speaker talked about. Log them for a human review pass rather than discarding silently.
  1. Make the skill idempotent on re-run. Hash the transcript; if the hash matches a previously processed version, skip. If transcription changed (better model, custom vocabulary added), re-run extraction and diff the output.
