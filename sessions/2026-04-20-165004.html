Research: Extracting SKILL.md Files from Video/Audio Transcripts

Executive Summary

A production-grade transcript-to-SKILL.md pipeline is a four-stage system: ingest (yt-dlp + Whisper/Deepgram), segment and chunk (speaker diarization + map-reduce chunking), extract (schema-first LLM pass targeting the SKILL.md spec with hedge-classification filtering), and validate (Pydantic/JSON-Schema harness + YAML frontmatter lint). The core intellectual move is the narrated-event to general-procedure modality shift: source material is first-person episodic ("I ran X, then Y happened"), while SKILL.md output must be second-person imperative and generalized ("When condition C, run X. Verify Y."). Confidence is high: every component has mature tooling and clear prior art, and the Anthropic spec for SKILL.md is published and stable.

Key Findings

  1. SKILL.md spec is minimal and strict, which constrains extraction cleanly.

Only name (lowercase-kebab) and description are required in YAML frontmatter; allowed-tools, license, compatibility, metadata are the optional base fields. Body is markdown instructions with progressive disclosure (Level 1 metadata, Level 2 SKILL.md body, Level 3 linked resources). This means an extractor has a small, well-typed target, which is exactly the shape schema-first extraction is designed for.

Source: Metadata Schema Reference - Anthropic Skills — first-party spec. Corroborated by Anthropic Agent Skills docs.

  1. yt-dlp + Whisper is the dominant OSS transcription stack; Deepgram Nova wins on accuracy/latency for paid workloads.

The standard pipeline (yt-dlp -x --audio-format mp3 ... | whisper ...) is reproduced across dozens of repos and tutorials. For production, Deepgram benchmarks show ~36% lower WER and up to 5x lower latency vs Whisper-large, with streaming support. Whisper wins on privacy (fully local), cost ($0 marginal), and language coverage. The deciding variable is volume: under ~10h/week, local Whisper; above, Deepgram Nova-3.

Sources: MurmurCast pipeline walkthrough, Deepgram vs Whisper benchmark PDF, AIMultiple STT benchmark.

  1. Map-reduce is the canonical pattern for transcripts longer than the context window, but naive map-reduce loses cross-chunk references.

The LLM×MapReduce paper (ACL 2025) documents exactly this failure: "crucial long-range information can be lost during document segmentation." Their fix is structured information passing between map and reduce stages. For SKILL.md extraction specifically, this matters because a speaker often states the goal at minute 3, the procedure at minute 27, and the caveat at minute 51.

Source: LLM×MapReduce (Zhou et al., ACL 2025) — peer-reviewed, top-tier venue.

  1. Hedge detection is a mature NLP subfield and is directly applicable to filtering "maybes" out of extracted procedures.

CoNLL-2010 established the shared task; HEDGEhog (BERT-based, four-class: Epistemic/Investigation/Doxatic/Condition) and modern hedge-detection models (Islam et al. 2024) achieve F1 > 0.87 on Wikipedia corpus. Podcast speech is hedge-dense ("I think maybe you should probably..."); a classifier pass before extraction prevents the LLM from encoding speculation as imperative.

Sources: HEDGEhog on HuggingFace, "You should probably read this": Hedge Detection (arXiv:2405.13319), CoNLL-2010 Shared Task.

  1. Schema-first structured output with Pydantic + constrained decoding is now the default for production extraction.

OpenAI Structured Outputs (CFG engine guarantees 100% schema compliance), Anthropic tool-use forced schemas, and the Pydantic+LangChain/Instructor stack all converge on the same pattern: define the target schema first, bind it to the model, retry-on-validation-fail. This is the exact primitive needed to emit valid SKILL.md frontmatter.

Sources: OpenAI Structured Outputs Complete Guide, LLM Structured Outputs: Production Guide (Fordel), Guardrails AI test suite.

  1. Prior art on narrative-to-procedure conversion exists in academic literature but no public tool targets SKILL.md specifically.

Zhang/Uren/Ciravegna (Sheffield) defined "procedural knowledge acquisition from instructional text" as an IE subproblem; Jiang et al. (USC/CMU) on "Transferring Procedural Knowledge across Commonsense Tasks" and Zhou et al. (CMU) "Procedures as Programs" are the closest academic antecedents. None of these target the SKILL.md format, which leaves a clear novelty window for a dedicated extractor.

Sources: Zhang et al. procedural knowledge acquisition (Sheffield), Jiang et al. arXiv:2304.13867, Zhou et al. "Procedures as Programs" arXiv:2109.08214.

Deep Dive

The four-stage pipeline

Stage 1: Ingest. yt-dlp -x --audio-format mp3 --audio-quality 0 "$URL" for audio extraction, with --write-auto-subs --sub-format vtt as a fast path when YouTube auto-captions are good enough (caption path is free and near-instant; skip ASR entirely when WER is acceptable). yt-dlp's Information Extractor pipeline [Source 1] handles all major platforms (YouTube, Vimeo, SoundCloud, podcast RSS via direct URL). For closed content without captions, fall through to ASR.

Stage 2: Transcription + diarization. Two tool choices:

  • Local path: whisper-large-v3 or whisperX (latter adds forced alignment + pyannote diarization). Accuracy is strong on clean audio, degrades on overlap and accents. ~1x realtime on M-series Mac, ~0.3x on consumer GPU.
  • API path: Deepgram Nova-3 (benchmark: 54.2% WER reduction vs competitors per Deepgram's own methodology; AIMultiple's independent healthcare benchmark agrees directionally, less dramatically) [Sources 4, 5]. Deepgram also ships diarization, punctuation, smart formatting, paragraph detection — all of which save you from writing a post-processor.

Diarization matters for the extraction stage: if the podcast has a host and a guest, the host's reactions ("wow, that's cool") should not end up in the SKILL.md. Speaker-labeled transcripts let you scope extraction to the expert speaker only.

Stage 3: Extraction (the hard part). This is where the narrative-to-procedure shift happens. Three patterns, used in layers:

  1. Chunked map phase with schema-first extraction. Split transcript into ~8-12k-token chunks with 500-token overlap (sliding window prevents boundary loss — see [Source 11]). For each chunk, prompt the LLM with a Pydantic schema matching the SKILL.md fields plus a candidate_procedures: list[ProcedureStep] field. Use OpenAI Structured Outputs or Anthropic's tool-use schema enforcement — both guarantee parseable output [Sources 6, 12].
  1. Hedge classification filter. Run HEDGEhog or equivalent over the speaker's sentences before extraction, or over the candidate_procedures after. Drop or de-weight anything flagged Epistemic ("I think", "probably") or Doxatic ("I believe"). Keep Investigation-labeled content but mark it as context, not instruction [Source 7]. Yona et al. (EMNLP 2024) [Source 13] show LLMs themselves often fail to preserve hedging, so an external classifier is more reliable than asking the extractor model to filter.
  1. Reduce phase with cross-chunk reconciliation. The LLM×MapReduce protocol [Source 10] passes structured state (not prose summary) between stages. For SKILL.md, the reduce-phase schema is the actual SKILL.md structure: name, description, prerequisites, steps, verification, gotchas. The reducer deduplicates, orders steps, and resolves conflicts (if two chunks suggest different step orders, the later-mentioned or more-detailed version usually wins — heuristic, not rule).

Stage 4: Validation harness. Three gates:

  • Frontmatter lint: YAML parse + JSON-Schema check against the Anthropic spec (name matches ^[a-z][a-z0-9-]*$, description is non-empty string, optional fields have correct types) [Source 2].
  • Body structural check: markdown parses; has at least one ## heading; body length between 200 and ~5000 words (progressive disclosure design assumes short SKILL.md with linked resources — see Glen Rhodes [Source 14]).
  • Semantic eval: run the generated SKILL.md against a held-out test query with a small model. If the model can't follow the procedure to produce a testable output, the skill is under-specified. This is the pattern used by Guardrails AI's test suite [Source 15] and Pydantic AI's TestModel [Source 16].

The narrated-event to general-procedure modality shift

This is the conceptually novel part of the workflow. Four transformations happen here:

Axis Source (transcript) Target (SKILL.md)
Voice First person ("I did X") Second person imperative ("Do X")
Tense Past/narrative ("then it crashed") Present/conditional ("if it crashes, ...")
Specificity Concrete particulars ("our Redis at port 6380") Generalized variables ("your cache endpoint")
Modality Hedged, narrative ("I think we probably should have...") Declarative, procedural ("Before step 3, verify ...")

Academic grounding for this shift: Zhang et al.'s Sheffield work frames it as IE over instructional text with an ontology target [Source 8]. Langley/Shrobe/Katz (ACS 2020) [Source 17] describe it as a cognitive subtask decomposition — syntactic parse, then schema mapping, then variable abstraction. Kratká (IAFOR 2014) [Source 18] is the underlying knowledge-management theory: narrative is how tacit knowledge moves between humans, so tacit-to-explicit conversion requires processing the narrative form.

Practical implementation: after the reduce phase produces a draft SKILL.md, a second LLM pass does explicit "de-narration" with a small set of transformations prompted with few-shot examples. A human-in-the-loop diff review at this stage catches generalization errors (over-generalization: dropping a constraint that actually mattered; under-generalization: keeping a speaker-specific detail as if it were universal).

Prior art landscape

Closest existing public work:

  • MurmurCast [Source 3]: yt-dlp + Whisper + Claude summarization pipeline. Targets digest emails, not skills. Good reference architecture.
  • podcastfy (souzatharsis): the reverse direction (content → podcast) but shares the transcript-handling primitives.
  • transcriptor (jonradoff): full pipeline with diarization, outputs HTML not SKILL.md.
  • Anthropic skills repo: hand-authored skills only; no extraction tooling.
  • NotebookLM [Source 19]: does transcript restructuring conceptually but is closed-source and does not emit SKILL.md.

Academic gap: no paper specifically addresses "extract agent-skill-format procedural knowledge from long-form conversational transcripts." The adjacent work (Sheffield, USC, CMU) targets formal ontologies or commonsense benchmarks, not the deployable-instruction format that SKILL.md represents. This is a reasonable novelty position for original work.

Concrete pipeline recipe


1. yt-dlp -x --audio-format mp3 --write-auto-subs --sub-format vtt URL
   (if good captions exist, skip step 2)
2. deepgram transcribe --diarize --paragraphs --model nova-3  (or whisperx local)
3. filter to target speaker; segment by topic (use embeddings + threshold)
4. for each chunk:
     hedge_labels = hedgehog(chunk)
     candidates = llm_extract(chunk, schema=SkillCandidate, 
                              hedges=hedge_labels)
5. reduce(candidates) → SkillDraft  (LLM×MapReduce style, structured state)
6. de_narrate(SkillDraft) → SKILL.md  (voice/tense/abstraction pass)
7. validate(SKILL.md):
     yaml_lint, schema_check, markdown_parse, semantic_eval
8. human review diff → commit

Feynman Bridge

A video transcript and a SKILL.md are two different kinds of writing about the same knowledge. A transcript is like someone telling a story at a dinner table: "So I was trying to set up my database, and first I tried X, which didn't work, and then my coworker suggested Y, and that was the thing that finally worked." A SKILL.md is like a recipe card: "To set up the database, do Y. If you see error E, check Z." Same knowledge, totally different shape.

Think of it like translating a travel diary into a guidebook. The diary says "Tuesday we got lost on the way to the train station because we trusted Google Maps and it sent us through a closed construction zone." The guidebook says "From the hotel, walk north on Main Street (do not use Google Maps — it routes through an active construction zone)." Same fact, but the guidebook strips the narrator, removes the past tense, and pulls out the generalizable warning.

Our pipeline does that translation automatically. We pull the audio with yt-dlp, turn it into text with Whisper or Deepgram, break the text into chunks the LLM can handle, have the LLM fish out candidate procedure steps while a separate classifier throws out the speaker's "maybe" and "I think" language, merge the pieces, and finally rewrite everything from diary-voice to guidebook-voice. Then a validator checks that the output is a real SKILL.md — correct frontmatter, parseable markdown, instructions a different agent can actually follow. A 12-year-old version: we're turning "storytime" into "instruction manual," automatically.

Actionable Insights

  1. Build the validator before the extractor. A Pydantic model matching the Anthropic SKILL.md spec plus a pytest harness that runs generated SKILLs against a test query is maybe 200 lines of code and pays for itself immediately. Without it, you will ship broken skills for a week before noticing.
  1. Start with YouTube auto-captions, not ASR. For ~70% of English YouTube content, auto-captions are good enough and cost zero seconds of compute. Only fall through to Whisper/Deepgram when captions are missing, auto-generated with poor quality (detect via perplexity check), or for non-YouTube sources.
  1. Use Deepgram Nova-3 with diarization for any serious volume. The 36%-lower WER and built-in paragraph/diarization output saves a full post-processing stage. Under ~$200/month of transcription, pay for it rather than operating Whisper infrastructure.
  1. Make hedge-classification a separate stage, not a prompt instruction. Yona et al. (EMNLP 2024) show LLMs won't reliably preserve or strip hedging when instructed to. Run HEDGEhog or equivalent explicitly and pass labels into the extractor as structured input.
  1. Adopt the LLM×MapReduce structured-state pattern, not text-summary reduce. Passing Pydantic/JSON state between chunks preserves the cross-chunk references ("the goal from minute 3 applies to the steps at minute 27") that a prose-summary reducer silently drops.
  1. Add an explicit de-narration LLM pass with few-shot examples. Do not try to do narrative-to-procedure translation inside the extraction prompt. Separating concerns (extract facts → de-narrate voice) makes each prompt simpler, shorter, and testable in isolation.
  1. Publish this as a deepagents skill and eat your own dogfood. The deepagents project already has a skills directory and a research-wiki pattern. A transcript-to-skill skill that runs this exact pipeline would both validate the approach and seed the skills library with domain expertise extracted from conference talks.

  • Conflicting signals:

- Whether to do diarization before or after transcription depends on the tool (WhisperX does them jointly; Deepgram integrates them; raw Whisper requires a separate pyannote pass). No universal answer.
