Mission Control — Architecture and Current State

Overview

Mission Control (pna-sessions) is the static archive and operational dashboard for Principal and Agent. It serves three purposes simultaneously:

  1. Public artifact archive — research outputs, session logs, architecture diagrams, and brainstorms from all agents
  2. Live operational dashboard — mission kanban board, full backlog, and cost/performance tracing powered by real claudeclaw.db data
  3. Skill library browser — searchable catalog synced from the pna-library repository

Deployed at https://afoxnyc3.github.io/pna-sessions/ with Vercel auto-deploy on every push to main. The architecture is intentionally static: HTML files, a JSON manifest registry, and Python scripts that inject data from claudeclaw.db. No build step, no runtime server, no framework.

Core Architecture

Static HTML + manifest.json + Vercel auto-deploy

Every surface is a self-contained HTML file. Data flows in through two paths:

  1. Publish time (artifacts): publish.sh copies an HTML file to the appropriate subfolder, updates manifest.json, commits, and pushes. Vercel picks up the push and deploys in approximately 30 seconds.
  2. Refresh time (operational data): refresh.sh runs every 30 minutes via launchd (com.claudeclaw.pna-refresh). Three Python scripts query claudeclaw.db read-only and inject JSON into kanban.html, backlog.html, and tracing.html via regex replacement of window.__DATA__ placeholders. If content changed, it auto-commits and pushes.

Resource Strategy Reason
manifest.json Network-first Always fetch the fresh artifact registry; fall back to cache
Shell assets (icons, webmanifest, index.html) Cache-first These change rarely; fast load is the priority
HTML pages (sessions, designs, kanban, backlog, tracing, library) Network-only Always fresh from Vercel; avoids serving stale operational data

The result: the archive gallery is always current, operational dashboards never show cached-old data, and icons load instantly even offline.

Auth: Cloudflare Access Zero Trust and OTP

Cloudflare Access is configured at the proxy level in Cloudflare's dashboard and Vercel's project settings — it is not committed to the repository.

The internal domain (control.pna.agency) sits behind Cloudflare Access with one-time passcode (OTP) authentication. The public GitHub Pages URL (afoxnyc3.github.io/pna-sessions/) has no auth layer and is publicly readable.

Design intent: Session artifacts are public by design — the archive is a shareable record of P&A research work. Operational surfaces (kanban, backlog, tracing) contain mission data and cost metrics and are served from the Cloudflare Access-protected domain only.

No client-side auth or token validation exists in the static HTML. The protection boundary is entirely at the Cloudflare Access layer.

Surfaces

index.html — Artifact Gallery

Main archive browser. Loads manifest.json at runtime and renders a filterable list. Each artifact shows: type dot (green=session, blue=diagram, amber=brainstorm, purple=design, cyan=research, red=postmortem), date, agent, title, and tags. Filter tabs across all artifact types. Mobile-first with A2HS install banner.

kanban.html — Mission Board

Live mission kanban for the last 14 days of missions. Four columns: Queued, Running, Completed, Failed. Data injected as window.__KANBAN_DATA__ from the mission_tasks table in claudeclaw.db. Each card shows: task ID, title, agent, status, priority, result preview (first 200 chars), and timing. Updated every 30 minutes.

backlog.html — Mission History

Full mission history table (all non-cancelled tasks). Columns: Priority, Status, Title, Agent, Created, Completed. Shows error traces for failed missions. Same 30-minute refresh cycle as kanban.

tracing.html — Cost and Performance Analytics

Session-level token usage analytics (last 50 sessions) and mission timing (last 60 missions).

Per-session metrics: turn count, peak context tokens, total output tokens, cost in USD, compaction count, session duration.

Per-mission metrics: duration in seconds, error stacks for post-mortem analysis.

Primary tool for tracking spend and agent performance over time.

library.html — Skill Browser

Searchable, categorized skill catalog with tabs: Core Skills, Engineering Skills, Domain Expertise. Each card shows skill ID, description, trigger patterns, and source badges. Synced from pna-library (afoxnyc3/pna-library) and managed by principal_librarian.

sessions/, designs/, diagrams/, brainstorms/ — Artifact Folders

Self-contained HTML files. Every file embeds all CSS and font references so sessions work offline. Naming: YYYY-MM-DD-HHMMSS.html. Agent types represented: principal_*, main, clarke, research_a, research_b.

Publish Workflow

Standard artifact publish:


publish.sh --file /path/to/file.html \
           --title "Title" \
           --type session|diagram|brainstorm|design \
           --agent "agent_id" \
           --tags "tag1,tag2"

Pulls latest main, copies file to the correct subfolder, calls update_manifest.py to upsert the manifest entry, commits, and pushes. Vercel deploys in ~30 seconds.

Session generation from markdown:


python3 generate_session.py \
  --title "Title" \
  --agent "principal_architect" \
  --body-md path/to/summary.md \
  --type session \
  --tags "tag1,tag2"

Or pipe markdown via stdin. The script parses YAML frontmatter between --- delimiters, converts markdown to HTML (headings, lists, tables, code blocks, inline formatting), builds a self-contained styled HTML page, writes to sessions/YYYY-MM-DD-HHMMSS.html, and updates manifest.json.

Live data refresh (automated):


refresh.sh  # launchd: com.claudeclaw.pna-refresh, runs every 30 min

Calls export_kanban.py, export_backlog.py, export_trace.py sequentially. Each queries claudeclaw.db read-only and regex-replaces the window.__DATA__ block in the target HTML. Commits with message "data: auto-refresh YYYY-MM-DD HH:MM" only if content changed.

Current Gaps

Cloudflare Access not in repo: Configuration lives in Cloudflare dashboard and Vercel project settings. If the project is cloned fresh, the auth layer is absent. There is no infrastructure-as-code for the Cloudflare Access policy.

Local-only data dependency: The Python data scripts (export_kanban.py, export_backlog.py, export_trace.py) assume claudeclaw.db is accessible locally. There is no remote data path; the refresh loop must run on the machine where claudeclaw.db lives.

No retract or edit workflow: Published sessions are immutable. There is no mechanism to update or remove a published artifact short of manually deleting the file and updating manifest.json.

Font offline dependency: Google Fonts CDN is linked (preconnect headers in generated HTML). True offline mode would require inlining the fonts. Sessions work offline for content but degrade typographically without network access.

No real-time push: Operational dashboards have up to 30 minutes of staleness. There are no webhooks, WebSockets, or push channels. Acceptable given the use case — this is an async ops board, not a monitoring alerting tool.

Design Principles

Static-first: No build pipeline, no Node.js server, no database reads at request time. All complexity lives in the publish and refresh scripts, not in runtime infrastructure. Failure modes are simple and debuggable.

Self-contained artifacts: Every session HTML file stands alone. No dependency on CDN for content. Sessions can be emailed, archived, or opened from disk.

Data freshness via push: Operational dashboards are refreshed by a scheduled local job, not by client-side polling. Acceptable staleness: 30 minutes. Keeps the architecture simple and the Vercel usage minimal.

Public archive, protected ops: Session artifacts are public by design and shareable. Operational metrics (cost, mission status, agent performance) sit behind Cloudflare Access on the internal domain.
