Executive Summary

Winner: research_a — by a significant margin.

Both agents ran in degraded environments (Exa MCP, feynman, and alpha CLIs unavailable). Both were honest about it. That's where the similarity ends.

research_a produced a 10-step, tool-specific, validated pipeline with 15 cited sources (all canonical: GitHub repos, arXiv papers, official docs), a Pydantic schema for extraction, a concrete cost model, and a validation harness. Its confidence was correctly rated medium.

research_b produced a well-reasoned design sketch, the same two-pass idea (scaffolding + densification), and a solid Feynman Bridge — but zero external sources retrieved, low confidence declared, and shallower specificity throughout.

Single most important finding (from research_a): The narrated-event → general-procedure modality shift is the only truly hard engineering problem in this pipeline. Transcription is solved. Extraction tooling exists. The gap no existing tool covers is forcing the LLM to transform "I noticed the port was in use, so I killed the old process" into "Before starting the server, check port occupancy; if occupied, confirm with the user before killing the holder." This is where 60% of LLM budget and 80% of prompt-engineering effort should go. Both agents identified this; research_a named it precisely, gave it a hedge-classification taxonomy, and called it the key differentiator.

Winner

research_a — decisively.

research_a produced a publishable, source-backed, schema-specific, validated pipeline design with actionable tool choices, cost model, and a week-1 MVP definition. It correctly foregrounded the description field as the key product of skill extraction (not just the body), which is the most important architectural insight for building a high-quality skill library. Its hedge-classification taxonomy is the right abstraction for the modality-shift problem.

research_b produced a sound design sketch and is a good sanity-check that the two agents converged on the same core ideas independently. But it would not be publishable without the sources and schema that research_a provided.

The one legitimate addition from research_b that research_a missed: hash-based idempotency (skip re-extraction if transcript unchanged) and the rejected scaffolds log (preserve low-confidence candidates for human review). Both should be incorporated into the final pipeline.

Recommendation

Alex should build this in three phases, in this order:

Phase 1 (this week — ~2 days): Ingest + Transcription MVP

Run research_a's steps 1-3 on three test videos (one dense tech talk, one podcast, one short tutorial). Measure technical-term word error rate between WhisperX output and YouTube auto-captions. This validates the transcription layer before investing in extraction. Tools: yt-dlp, WhisperX large-v3-turbo, Claude Sonnet post-pass.

Phase 2 (next week — ~3 days): Extraction + Schema

Implement the Pydantic extraction schema from research_a. Start with the two-pass version from research_b (simpler to reason about), then add self-consistency voting if quality is insufficient. Incorporate research_b's hash-based idempotency from the start. Prototype hedge-classification on 5 hand-annotated segments before committing to the taxonomy. Tool: Instructor + Claude Sonnet.

Phase 3 (sprint 3 — ~2 days): Rewrite + Validation + Ship as video-to-skill

Build the rewrite pass with the explicit three-axis transformation (person, register, modality). Author description as a separate final pass using research_a's template. Add triggers.yaml validation harness. Ship the whole pipeline as a single Claude Code skill (video-to-skill) that accepts a YouTube URL and emits a ~/.claude/skills/<slug>/ directory with SKILL.md, reference/source.md, and tests/triggers.yaml.

Do not build the pipeline and the skill library in parallel. The extraction quality needs empirical validation on real content before the skill library accumulates junk. Ship the pipeline first, test on 10 videos manually, then open it to higher-volume use.

Tool recommendation (final):

  • Ingest: yt-dlp (no alternative)
  • Transcription: WhisperX large-v3-turbo self-hosted (if GPU available), else Deepgram Nova-3 with Keyterm Prompting
  • Extraction: Instructor + Pydantic + Claude Sonnet (match training distribution for Claude Code target)
  • Orchestration: single Python script exposed as a Claude Code skill and optionally as an MCP tool
