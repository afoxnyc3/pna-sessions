task-board — Spec

Problem

ClaudeClaw has a mission queue (mission_tasks) for one-shot agent dispatches and an action log (hive_mind) for what happened, but no persistent place to capture "things we want to do" that survives across missions. Alex captures ideas in chat, in vault notes, and in his head. There is no single source of truth that:

  • accepts intake from any agent or from Alex,
  • tracks an item from idea to done across multiple agent runs,
  • gives one Mission Control view that shows backlog and in-flight work side by side,
  • forces every meaningful artifact (design doc, diagram, ADR, retro) to be linked back to the work it came from.

This must be fixed without breaking the mission queue, the hive_mind log, or the Issue 6 session summary primitive currently in flight on feat/issue-6-session-summary.

Proposed Solution

A single tasks table in claudeclaw.db is the unified store. Status drives view membership: captured is the backlog, {todo, in_progress, review, blocked, done} are the kanban lanes. Missions reference tasks via a new task_id foreign key on mission_tasks (additive, nullable). Agents claim, update, comment on, and close tasks via a new task-cli that mirrors mission-cli patterns. Mission Control's dashboard-html.ts gains two sections: Backlog (intake form + filterable list) and Kanban (lanes by status, filterable by agent). Every agent's CLAUDE.md is amended with a small ritual: claim before work, comment as you go, link artifact and move to review at completion.

Non-Goals

  • Replacing the mission queue. Missions still exist as one-shot dispatches; tasks are the parent unit of work that may spawn 0..N missions.
  • Replacing Obsidian as the writing surface. Long-form planning still happens in vault and ~/dev/projects/<slug>/. Tasks are pointers, not prose.
  • Replacing GitHub Issues. This system is for cross-project, cross-agent work tracking inside the Principal & Agent stack. GitHub Issues remain the source of truth inside individual project repos like roadrunner-cli.
  • Multi-tenant or multi-operator access. Single-operator (Alex) only.
  • Real-time collaborative editing. Last-write-wins on update is fine for one operator.
  • Drag-and-drop kanban UI in v1. Status moves happen via buttons or CLI; drag-drop is post-v1 polish.
  • Auth on Mission Control endpoints in v1 — MC is already on localhost or behind PIN; same posture inherited.
  • Migrating historical work from hive_mind or mission_tasks into tasks retroactively. Forward-only.
  • Cross-database sync (no Linear, Jira, Notion mirror). Out of scope.

Acceptance Criteria

Storage and migration:

  • tasks table exists in claudeclaw.db after migration with columns and constraints from ARCHITECTURE.md, verified by sqlite3 store/claudeclaw.db ".schema tasks".
  • mission_tasks.task_id column exists, is nullable, and has an index on (task_id, status), verified by .schema mission_tasks.
  • Running migrations twice on a fresh DB and on a populated DB (60 missions, 39 hive_mind rows) produces zero errors and no data loss, verified by row counts before and after.
  • Existing rows in mission_tasks have task_id IS NULL and continue to behave correctly through the queue.

CLI:

  • node dist/task-cli.js add --title "X" --created-by alex --body "description" creates a row with status='captured' and prints the task id. Positional body argument also accepted (e.g. ... add --title "X" "description") to match mission-cli create ergonomics.
  • node dist/task-cli.js list --status backlog returns only rows with status='captured'.
  • node dist/task-cli.js list --status kanban returns rows with status in {todo, in_progress, review, blocked, done}.
  • node dist/task-cli.js move <id> in_progress updates status and writes a row to task_events.
  • node dist/task-cli.js claim <id> --agent principal_engineer sets assigned_agent, moves status to in_progress, and writes an event.
  • node dist/task-cli.js comment <id> "text" appends a comment row to task_events of type comment.
  • node dist/task-cli.js link <id> --artifact <path-or-url> appends an event of type artifact and stores the link.
  • node dist/task-cli.js show <id> prints the task plus its full event timeline plus all linked missions.
  • Every CLI subcommand exits non-zero on invalid input and prints a usable error.

Mission linkage:

  • node dist/mission-cli.js create --task-id <id> ... accepts --task-id, validates the task exists, and stores it on the mission row.
  • When a linked mission completes, the task automatically moves from in_progress to review (configurable behaviour, default on).
  • task-cli show <id> lists every mission with that task_id and their statuses.

Mission Control UI:

  • dashboard-html.ts renders a Backlog section listing all status='captured' tasks with title, created_by, age, and an "Activate" button that moves to todo.
  • dashboard-html.ts renders a Kanban section with lanes todo, in_progress, review, blocked, done showing tasks as cards with title, assigned_agent, age, and mission count.
  • Both sections refresh in under 500ms on a localhost DB with 1000 tasks (measured manually with the dashboard timer).
  • Backlog has an inline form with title, body, created_by; submit calls POST /api/tasks and the new task appears without a full reload.
  • Kanban cards have lane-move buttons (left and right arrow per status); clicking calls POST /api/tasks/:id/move and the card relocates without a full reload.
  • Filter by assigned_agent (dropdown of distinct agents from the table) works on the Kanban view.

Agent integration:

  • Every principal's CLAUDE.md (~/.claudeclaw/agents/principal_*/CLAUDE.md and the central Clarke prompt) gains a "Task discipline" section mandating: claim before work, comment per checkpoint, link artifact at completion, move to review.
  • At least one principal demonstrates the loop end-to-end: pick a task from backlog, move to in_progress, comment twice, link an artifact, move to review. Verified by running the loop and inspecting task-cli show <id>.
  • Mission completion handler updates the linked task's status to review and writes a mission_completed event.

Artifact publishing:

  • When an agent runs publish_session.sh against a task-linked artifact, the published URL is automatically captured as a link event on the task. Implementation: a small wrapper in pna-sessions/scripts/publish_session.sh that accepts --task-id and posts back to claudeclaw's task API on success.
  • task-cli show <id> displays artifact links inline.

Performance and durability:

  • With 1000 seeded tasks in the DB, the Kanban view's initial paint of the first 200 rendered cards completes in under 500ms p95 on Alex's M-series Mac (manual measurement, single user, three runs).
  • All writes go through the existing better-sqlite3 transaction wrapper; no raw db.exec outside migration code.
  • Existing tests in ~/dev/claudeclaw continue to pass; new tests cover task CRUD, status transitions, and mission linkage.

Tech Stack

Layer Choice Rationale
Storage SQLite via better-sqlite3 (existing claudeclaw.db) One DB, one process, one writer is already the model; adding a second store doubles backup and migration surface for no gain at single-operator scale.
Schema migrations Existing migrations.ts runner with a new numbered file Already used by hive_mind, agent_events, mission_tasks; reusing it keeps the upgrade path identical to every prior schema change.
CLI Node + TypeScript, distributed as dist/task-cli.js Mirrors mission-cli.js and schedule-cli.js exactly, including the DB_ENCRYPTION_KEY env contract and arg parsing style; zero new dependencies, instant familiarity for engineer.
Dashboard Extend src/dashboard-html.ts (single-file HTML/JS already serving Mission Control) Avoids spinning up a Next.js app for two new sections; current dashboard is 2289 lines of vanilla HTML/Tailwind/inline JS that loads in one round trip and works behind PIN auth. A separate web app would multiply deploy steps and break the localhost-first posture.
API Extend src/dashboard.ts HTTP handler with /api/tasks* routes Same reason — the dashboard server already vends /api/missions; adding /api/tasks is a 100-line patch, not a new service.
Test runner vitest (existing) Used by every other claudeclaw module.
Auth Inherit existing PIN gate on the dashboard Same posture as today; this is single-operator localhost.
Artifact links Plain text URL or absolute path stored on task_events rows No CMS, no thumbnails; Mission Control already renders pna-sessions URLs.

Real projects doing this exact pattern (single SQLite + single-file HTML dashboard for personal task tracking): simonw/datasette, TodoTxt desktop variants, and ClaudeClaw's own existing dashboard surface. Not novel.

Architecture

See ARCHITECTURE.md. High-level:

  • task-cli.ts is the write/read API surface for terminals and agents.
  • dashboard.ts API handlers expose the same operations to the HTML UI.
  • dashboard-html.ts renders Backlog + Kanban sections with vanilla JS calling /api/tasks*.
  • migrations.ts ships migration 00NN_tasks_and_events.sql.
  • mission lifecycle in scheduler.ts and orchestrator.ts updates linked tasks on mission completion.
  • agent CLAUDE.md prompts are amended with the task-discipline section.
  • publish_session.sh in pna-sessions/ is patched to accept --task-id and post back.

Risks

  1. Mission completion handler races with manual status moves. Likelihood: M, Impact: L — Mitigation: optimistic write that only auto-moves in_progress → review if status is still in_progress; otherwise no-op and write a mission_completed_no_move event for diagnosis.
  2. Migration runs against a live DB while bot is up. Likelihood: L, Impact: M — Mitigation: existing migration runner already takes a write lock; add a startup gate that refuses to start if migration version mismatches code version. Engineer documents the standard launchctl kickstart -k ritual in the migration's README header.
  3. Dashboard becomes slow when tasks grow past several thousand. Likelihood: L, Impact: L — Mitigation: paginate kanban lanes at 50 cards each with "show more"; backlog list paginates at 100; full-text search skipped in v1. Single-operator scale will not realistically hit this.
  4. Agent CLAUDE.md prompts get bloated. Likelihood: M, Impact: L — Mitigation: the task-discipline section is one short paragraph plus a 4-bullet ritual, total under 30 lines per agent. Librarian audits in the next routine grooming pass.
  5. Mission ↔ task linkage drift if a mission is created via DB insert outside the CLI. Likelihood: L, Impact: L — Mitigation: accepted. Single-operator system; if Alex inserts rows by hand, he owns the consistency.
  6. publish_session.sh patch needs to know the claudeclaw API endpoint. Likelihood: L, Impact: L — Mitigation: introduce CLAUDECLAW_API_URL env var with a localhost default; document in pna-sessions README.
  7. Kanban "review" lane fills up because no one closes tasks. Likelihood: M, Impact: L — Mitigation: dashboard shows age of cards; cards older than 7 days in review get a visual flag. Process problem, not architecture problem; flagged for retro discipline.
  8. Naming collision with tasks/ directory in ~/dev/projects/. Likelihood: L, Impact: L — Mitigation: project slug is task-board, table is tasks, CLI is task-cli. The existing ~/dev/projects/tasks/ is unrelated and untouched.

Open Questions

None. Defaults applied:

  • Linkage cardinality: one task → many missions, missions optionally link to one task. Bidirectional link via mission_tasks.task_id.
  • Sessions table interpretation: Alex said "agents logging missions to the sessions table." Interpreted as: every mission completion writes a task_event of type mission_completed against its linked task, AND the existing Issue 6 session summary primitive (engineer's branch) becomes the canonical narrative log. No new sessions schema needed; existing sessions table is left untouched.
  • Backlog vs idea capture: backlog accepts everything (ideas, bugs, initiatives, follow-ups). No separate "idea" status — captured covers it. Promotion to todo is the moment work is committed to.
  • Multi-task missions: a mission belongs to at most one task. If a mission spans concerns, split it.
  • Closed tasks: done is terminal. Reopening writes a new task_events row of type reopened and moves status back; immutable history preserved.
  • Default created_by: if CLI omits --created-by, default to the value of $CLAUDECLAW_AGENT_ID env var, falling back to alex.
