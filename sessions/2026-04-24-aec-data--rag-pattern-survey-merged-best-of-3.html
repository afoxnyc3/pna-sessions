Research: AEC Data & RAG Pattern Survey

Merged canonical output from 3 parallel research_a replicates. Support counts [k/3] annotate cross-replicate agreement.

Executive Summary

AEC platforms are actively decoupling project data from file formats, and this shift is what makes production RAG tractable. Today, AEC RAG is three loosely-joined stacks: text-RAG over specs/RFIs/submittals is production-grade (Autodesk Assistant, Newforma Konekt AI, Snaptrude, Nomic); drawing-aware retrieval via OCR + panoptic symbol spotting is production for navigation and beta for semantic reasoning (Procore, ACC Symbol Detection, CADTransformer, ArchCAD-400k, BLUEPRINT); and 3D/BIM retrieval is converging on IFC Graph-RAG (Neo4j + LLM-generated Cypher) as the emerging academic consensus, with geometric-similarity embeddings still research-only. Autodesk’s AEC Data Model API (GraphQL, public beta) and Speckle’s open object model (Apache 2.0) are the two realistic substrates for a multi-tenant, platform-agnostic firm-wide system. Speckle is the only stack explicitly designed to be self-hosted end-to-end.

Headline agreement across 3 replicates: YES. All three converged on the three-primitive architecture, the Speckle-vs-Autodesk substrate choice, the Graph-RAG-over-IFC recommendation, and the geometric-embedding whitespace. Confidence is medium across all three.

Key Findings

  1. [3/3] Files are losing to cloud-native data models. Autodesk’s AEC Data Model API exposes Revit elements as GraphQL-queryable objects with mutations in public beta (2025), explicitly framed as “from files to data.” Bentley iModels are self-describing, versioned, component-level, queryable via ECSQL. Speckle stores every object in a typed, versioned graph with GraphQL access. This cloud-graph shift eliminates the RVT/PLN parse step. Source: APS: From Files to Data — first-party Autodesk.

  2. [3/3] Graph-RAG over IFC is the emerging academic consensus for BIM Q&A. Iranmanesh et al. (EC3 2025, arXiv 2504.16813), Zhu et al. (Advanced Engineering Informatics 2026), Cypher4BIM (arXiv 2405.16345), BIMConverse (IAAC 2024), and IFC Whisperer (CISBAT 2025) independently show LLMs generating Cypher over IFC-to-Neo4j graphs outperform naive vector-RAG because BIM is inherently relational (Wall hosts Door, Room bounds Wall). Reported accuracy ceiling ~80% on NL-to-IFC queries; failures dominated by IFC’s multiple-valid-encoding problem. Sources: arXiv 2504.16813, Zhu 2026 AEI.

  3. [3/3] Drawing retrieval splits into text-OCR and CV-symbol-spotting primitives. OCR over vector-PDF text + callout geometry is production (Procore Drawings, ACC Sheets). Panoptic symbol spotting (FloorPlanCAD ICCV 2021, CADTransformer CVPR 2022, ArchCAD-400k 2025) is the SOTA backbone for countable-object retrieval (“how many GFCI outlets on sheet E-201”). BLUEPRINT (ORNL arXiv 2602.13345) combines region detection + VLM OCR + hybrid lexical+ANN retrieval for legacy archives. ColPali (PaliGemma + late interaction) offers an OCR-free alternative for intact digital sheets. Sources: CADTransformer CVPR 2022, ArchCAD-400k, BLUEPRINT.

  4. [3/3] Speckle is the only shipping open-source BIM data bus with a documented v3 object model. DataObjects, Collections, and Proxies form an AOT-free graph that receives from Revit, Rhino, ArchiCAD, IFC. Apache 2.0, self-hostable, multi-region. Snaptrude uses Speckle under the hood for its bidirectional Revit link. Source: Speckle Core Concepts, speckle-server GitHub.

  5. [3/3] Production AEC LLM assistants today do text-RAG only. Autodesk Assistant (Forma), Newforma Konekt Aggregate Search, Snaptrude AI, Nomic, and SpecGenius all operate on specs, RFIs, submittals, emails, and BIM metadata. None performs element-level BIM reasoning at GA. Autodesk Symbol Detection in Takeoff is a separate, narrow CV feature. Sources: Autodesk Assistant, Newforma Konekt, Snaptrude AI.

  6. [3/3] Geometric-similarity embeddings for BIM are research-grade, not productised. BIMSeek++ (Automation in Construction 2020), Lin et al. (Buildings 2025, GSTF embeddings fusing geometric + semantic + topological features, reported 780x speed-up over manual retrieval with 4-6% accuracy gain), and Gu et al. (2022 AAAI workshop, PointNet + graph-relational) all exist but no vendor surfaces “find elements like this” as a retrievable primitive. This is the clearest open whitespace. Source: Lin 2025 MDPI Buildings 15(6):951.

  7. [3/3] Hypar’s Elements library (MIT) is a rare neutral BIM code primitive. C# library modeling buildings as typed objects with IFC and glTF output, ~400 GitHub stars. Closest thing to a composable server-side building ontology available to a firm today. Source: hypar-io/Elements.

  8. [3/3] ConserveAI in the brief cannot be disambiguated. All three replicates independently reported: Conserve.AI (BLP Industry.AI) is an energy-efficiency product, not AEC RAG. Conserve Solutions is an India-based BIM automation services firm, not a RAG product vendor. Recommend clarification with Alex before citing ConserveAI in any downstream work.

  9. [2/3] Augmenta is generative, not RAG. Augmenta’s ACP (Autonomous Construction Platform) does agentic electrical raceway routing inside Revit — spatial AI for geometry synthesis, not retrieval. It does not belong in a RAG vendor matrix. Flagged by A1 explicitly and A3 in footnote.

Two observations, reinforced across all three replicates: (1) only Autodesk AEC Data Model and Bentley iTwin offer first-party element-granular cloud query APIs; the others still expect you to parse files. Speckle sits on top as a vendor-neutral overlay that exposes all of them through one object graph. (2) IFC remains the common denominator for vendor-neutral element data, but IFC’s permissive schema (multiple valid encodings of the same wall) is a documented RAG pain point.

Retrieval patterns by asset type

Text-class assets (specs, RFIs, submittals, emails, meeting minutes, contracts). Standard chunk-embed-retrieve. Chunking quality and metadata (project, phase, discipline, CSI MasterFormat section) dominate recall; AEC-specific gains come from CSI-aware chunking, RFI thread reconstruction, and sheet-coordinate anchoring for citations. Production at Newforma Konekt Aggregate Search, Autodesk Assistant, SpecGenius, Constructable, Snaptrude AI, Nomic.

2D drawings (PDF, DWG plots, scanned archives). Three stacked primitives: OCR for vector-PDF text and callouts (Procore, ACC Sheets — production); panoptic symbol spotting for countable-object queries (CADTransformer, ArchCAD-400k — beta); region/tile embedding for fuzzy visual match (ColPali, BLUEPRINT — emerging). Best retrieval is hybrid: symbol-spotting inventory + OCR text index + tile embeddings.

3D / BIM elements and relationships. Four primitives, ordered by maturity: - Property / parameter query via platform APIs (Revit, ECSQL, IfcOpenShell) — production, deterministic, exact-match only. - Graph-RAG on IFC/Speckle graph — LLM emits Cypher against Neo4j or GraphQL against Speckle. BIMConverse, IFC Whisperer, Cypher4BIM, simpleIfcAIAgentWithGraphRAG, mac999/BIM_graph_agent. Beta, reproducible. - Code-agent RAG — LLM writes Python against IfcOpenShell or Rhino.Compute and reads results. Text2BIM class, MCP4IFC. Higher compositional ceiling, harder sandboxing. - Geometric embeddings — BIMSeek++, Lin 2025 GSTF, Gu 2022. Research.

Point clouds. Out of scope for v1 firm RAG. Bentley iTwin has first-party Reality Data; Autodesk ReCap handles capture. No RAG product surfaces point-cloud semantic retrieval.

Three reference architectures for firm-wide AEC RAG

A. Speckle-centred open stack (lowest cloud lock-in, self-hostable). Speckle Server (Apache 2.0) as canonical object graph; specklepy to ingest Revit / Rhino / ArchiCAD; pgvector or Qdrant for text-RAG over SharePoint + project specs; per-project IFC → Neo4j mirror (nightly) for Graph-RAG; Poppler for vector-PDF OCR + optional CADTransformer microservice. Tenancy: one Speckle workspace per project, row-level security in Postgres. Trade-off: vendor-neutral, IFC-agnostic, data-residency control, but more glue code; Speckle’s schema is less MEP-rich than native Revit; Automate (code-agent layer) is Enterprise-only.

B. Autodesk-native stack (fastest for Revit+ACC firms). Ingest: Revit → ACC projects → AEC Data Model API (GraphQL). Text: ACC Docs + Model Derivative sheet PDFs → managed vector DB. BIM: GraphQL for element queries, Forge viewer for grounded citations. Tenancy: ACC hub/project IDs = firm tenants. Trade-off: zero file-parsing and mature sheet pipeline, but Autodesk lock-in at every layer; GraphQL still beta; pricing scales with ACC seats and APS usage tokens; non-Revit authoring becomes second-class.

C. iTwin-based federated twin (infra-heavy firms). iTwin connectors pull from Revit, OpenBuildings, DGN, IFC into per-project iModels. Text: SharePoint + iTwin Issues via iTwin Platform REST. BIM: ECSQL over iModel (SQLite-backed) + Graph-RAG for semantic questions. Tenancy: iTwin per client, Bentley Entra ID. Trade-off: strongest schema discipline (ECSchema), best for infrastructure-heavy firms, but Bentley-centric tooling, steeper learning curve, less community content, consumption-based opaque licensing.

Gap analysis: where text-RAG transfers cleanly vs. where new primitives are needed

  • Transfers cleanly: specs, RFIs, submittals, meeting minutes, emails, contracts, regulatory codes, drawing metadata (title blocks), OCR’d vector-PDF sheet text. Standard chunk/embed/rerank works.
  • Transfers with adaptation: drawing regions (need bbox as a first-class retrieval key), BIM property sets (category-aware chunking by IfcEntity), Revit schedules (tabular — better as SQL than vectors).
  • Requires new primitives:
    • Geometric similarity — needs learned shape embeddings; no production vendor ships this.
    • Relational BIM queries — graph traversal, not vector similarity; hence Graph-RAG.
    • Spatial containment / adjacency — needs IFC spatial structure tree (IfcRelContainedInSpatialStructure, IfcRelSpaceBoundary); vector stores cannot answer “which doors egress from assembly spaces.”
    • Cross-discipline clash reasoning — clash detection is a geometric engine problem (Navisworks, Solibri); RAG can explain clashes but cannot find them.
    • Countable-symbol queries on drawings — symbol-spotting CV service, not a generic vision embedding.

Feynman Bridge

A BIM model is not a document — it is a graph of typed objects (wall, door, duct) with relationships (contains, connects-to, is-hosted-by). Vanilla RAG, the kind that powers most chatbots, works by cutting text into chunks, turning each chunk into a vector, and finding the chunks whose vectors sit closest to your question’s vector. That works beautifully for a spec PDF. It fails for a question like “list every unrated fire door on floor 3,” because the answer is not a passage — it is the result of a query over structured relationships.

Think of an architecture firm’s project data as three shelves in a library. One shelf is novels — the specs, RFIs, and emails. Reading-comprehension AI handles those. One shelf is a card catalog — the BIM model as typed objects cross-referenced by host, level, and system. A smart research assistant reads the catalog index, not the novels, to answer “how many fire doors on level 3.” One shelf is an atlas — the drawings, full of maps studded with standardised icons. You need icon-recognition glasses (symbol spotting) to count fixtures, and you need OCR to read callouts. Production-grade AEC RAG is exactly this: three retrieval tools stitched together by a shared “which object are we talking about” identifier (IFC GlobalId, Revit UniqueId, Speckle object id) — not one magic vector search stretched over everything.

Actionable Insights

  1. [3/3] Design the firm-wide stack around three retrieval primitives, not one. Text-RAG for docs, Graph-RAG for BIM, symbol-spotting + OCR for drawings. A single vector DB alone will underperform on any real AEC question involving counts, relationships, or code compliance.
  2. [3/3] Anchor the substrate on Speckle + IFC → Neo4j, mirror from Autodesk AEC Data Model where licensed. Speckle is the only shipping vendor-neutral object graph; Neo4j mirror unlocks Graph-RAG. ACC integration is additive, not foundational.
  3. [3/3] Make stable object IDs a first-class concern. IFC GlobalId, Revit UniqueId, and Speckle object IDs are the glue that lets a chunked spec paragraph cite a specific element. AEC liability demands it; every LLM answer should link back to an IfcGUID or ElementId for audit.
  4. [3/3] Treat Graph-RAG on IFC as the highest-leverage investment. Peer-reviewed accuracy bounds (~80%), multiple independent reproductions (BIMConverse, Cypher4BIM, Iranmanesh, Zhu), and it answers the parametric questions architects ask daily. Budget for IFC normalisation — “same wall, five encodings” is a real problem.
  5. [3/3] Defer geometric-similarity embeddings to v2. Promising research (BIMSeek++, GSTF) but no vendor ships it. Build the other three primitives first; revisit when a vendor productises or the firm has labelled shape data.
  6. [2/3] Separate sheet search (OCR today, cheap win) from symbol search (CV service). Indices, latencies, and costs differ. Line up a CADTransformer / ArchCAD-400k benchmark against whatever Autodesk/Procore ship; this is where firm-specific differentiation is achievable.
  7. [3/3] Flag and avoid ACC-only and iTwin-only features in phase 1. Autodesk Assistant, Construction IQ, Bentley Grouping & Mapping are all cloud-locked; save them for tenants who already pay for the platform.
  8. [3/3] Treat Autodesk Assistant, Newforma Konekt, Snaptrude AI, and Nomic as complementary SaaS silos, not platform bets. Let them serve their native domains; the firm-wide AI aggregates via connectors.
  9. [3/3] Treat multi-tenancy boundary as project, not client. Matches Speckle workspaces, ACC projects, iTwins; lets joint ventures work without contortions.
  10. [3/3] Feed Mission 2 (Spector discovery) these probe questions:
    • Does Spector ingest via Speckle, ACC AEC Data Model, direct IFC, or a mix?
    • Is the BIM retrieval path vector-only or does it include a graph layer (Neo4j, Memgraph, or equivalent)?
    • How are drawing regions indexed — sheet-level only, or per-region symbol detection?
    • How is multi-tenancy implemented — project-scoped vector namespaces, row-level security, or separate graph DBs per tenant?
    • Do responses cite IfcGUIDs / ElementIds, and is there a human-verifiable audit trail?
    • Which features are cloud-locked (ACC, iTwin, Procore) vs. portable?
    • Does Spector offer any geometric-similarity primitive, or is it deferred like every other shipping vendor?
