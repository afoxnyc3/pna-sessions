Entra Triage — Project Brief

Overview

entra-triage is a single-tenant Microsoft Entra ID security triage and response platform. It monitors identity activity across a connected Entra tenant, surfaces security posture gaps and anomalies to an analyst, and proposes response actions with human approval gates.

The system ingests telemetry from Microsoft Graph (sign-in events, audit logs, risky users, risky sign-ins, directory changes, application and service-principal inventory), triages events through a rule-based pre-filter followed by LLM agents for ambiguous signals, evaluates security posture against Microsoft best-practice baselines, and maintains a full audit trail of every state change and proposed action.

Scope: Single-tenant only. V1 is advisory and read-only: agents propose actions, analysts approve, then execute manually in the Azure portal. No auto-execution in v1 — write permissions are gated until v2 (ADR 0006).

Architecture

Turborepo monorepo with three apps and five packages.

Apps:

  • apps/web — Next.js 14 dashboard (login, incidents, cases, baseline review, approvals)
  • apps/api — tRPC + Express API (read routes and action routes with JWT middleware)
  • apps/workers — BullMQ workers (IngestionWorker, TriageWorker, BaselineWorker)

Packages:

  • packages/shared — Zod schemas, Drizzle ORM definitions, enums, shared UI
  • packages/agents — Seven Anthropic SDK agents: BaseAgent, SignInTriageAgent, IdentityRiskAgent, DirectoryChangeAgent, BaselineReviewAgent, ResponsePolicyAgent, ResearchAgent
  • packages/connectors — Graph API client (real with MSAL auth; mock for P1/P2 dev)
  • packages/policy — assertActionPermitted() approval matrix with audit trail
  • packages/ui — Shared React components

Database (PostgreSQL, 10 tables, no tenant_id columns):

Table Purpose
sign_in_events Normalized sign-in records, indexed on created_at
directory_audit_events Role changes, app registrations, permission grants
risk_events P2 Identity Protection signals
cases Analyst-created incident groupings
incidents Agent-generated findings with severity and confidence
baseline_controls Security control catalog (versioned)
baseline_evaluations Per-control pass/fail with evidence
response_actions Proposed, approved, or executed actions (JSONB audit trail)
ingestion_cursors Per-stream checkpointing
audit_log Immutable append-only log of all mutations

Dependency order (acyclic): shared → policy → agents → connectors → api → workers → web

Events dedup on content_hash. Incidents dedup on hash(actor_id + category + target_id) within a 24-hour window.

Key Architectural Decisions

ADR 0002 — Anthropic SDK direct (no Pi framework): Full control over tool-use parsing. Confidence score is required on every agent result. No hidden prompt engineering from a framework layer.

ADR 0003 — Rule-based pre-filter (PASS/FLAG/AMBIGUOUS) before LLM: Approximately 90% of events bypass the LLM entirely, keeping cost and latency low. False negatives are the accepted trade-off; the pre-filter rules are auditable.

ADR 0005 — Single-tenant only: Simpler schema, no cross-tenant risk. Multi-tenant is a straightforward extension later — just add tenantId back where removed.

ADR 0006 — Read-only Graph permissions in v1: Smallest blast radius. All approved actions require manual execution in the Azure portal via markExecutedManually procedure. V2 write-permission expansion is gated on evidence: false-positive rate thresholds, operator training, rollback testing. Unlock path: RevokeSessionsAll first, then IdentityRiskyUser.ReadWrite, then User.ReadWrite, then Directory.ReadWrite.

Blocker: ROAD-010 (Resolved 2026-04-24)

ROAD-010 was an iteration counter bug in roadrunner-cli: the lifetime counter was used as the per-session runaway guard, so it could hit the session max after only a handful of actual session iterations across multiple Claude invocations, halting progress unexpectedly.

Fix: Split into two fields: iteration (lifetime audit trail, never resets) and session_iteration (per-session runaway guard, resets on SessionStart hook). The vendored roadrunner.py in entra-triage migrates transparently via setdefault on first load.

Impact on entra-triage: None blocking. The fix is live and the task loop is unblocked. ENTRA-011 is the next eligible task.

Success Criteria (V1 Definition of Done)

From SPEC.md acceptance criteria:

  • Connect to a single Entra tenant via env-configured Graph credentials
  • Validate Graph permissions at startup before ingestion begins
  • Ingest audit logs and sign-ins with deduplication and per-stream cursor checkpointing
  • Group incidents into analyst-friendly cases
  • Produce a baseline report mapping posture gaps to Microsoft best practices with scored findings
  • Gate all response actions through the approval matrix (v1: all writes require explicit approval)
  • Make every system action, agent action, and human override auditable
  • UI usable by a single analyst with exportable PDF reports
  • MOCK_GRAPH_API=true enables full local dev without real credentials
  • docker-compose + dev-setup.sh produces a working local environment from a clean clone

Target analyst workflow: Log in, view dashboard (risk posture, ingestion health, recent incidents), explore incidents by severity and category, drill into agent-generated narrative and recommended actions, approve a proposed response action, record manual execution in the Azure portal, view full audit trail.
