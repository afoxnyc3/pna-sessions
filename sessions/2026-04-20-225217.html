Research: NetSuite MCP OAuth 2.0 "Your role does not support OAuth2 login" resolution

Executive Summary

The error is gated by a single role-level permission that is missing on every role you have tested: "Log in using OAuth 2.0 Access Tokens" (Permissions tab → Setup subtab). Enabling the account feature and creating an integration record is not sufficient — each role a user can select at the consent screen must explicitly carry that permission. The REST Web Services scope on the integration record is required for the MCP Standard Tools SuiteApp to function (tools call the REST Record Service), and Administrator/CP Executive failing is consistent with the auto-created integration record having an incomplete scope set combined with missing role permissions. Fix both in tandem: add scope to the integration record, add the login permission to the role, assign the role to the user, retry.

Key Findings

  1. The role permission "Log in using OAuth 2.0 Access Tokens" is the specific gate triggering this error

This is the role-level OAuth 2.0 eligibility setting separate from the account feature flag and the integration scope. Oracle's Set Up OAuth 2.0 Roles page explicitly distinguishes it from "OAuth 2.0 Authorized Applications Management" (which only lets a user view authorized apps, not log in via OAuth). A role without this permission will reject OAuth 2.0 login even when everything else is configured.

Source: Set Up OAuth 2.0 Roles — Oracle NetSuite Help — first-party Oracle documentation.

  1. REST Web Services scope IS required on the integration record for the MCP Standard Tools SuiteApp

The NetSuite MCP record tools (create/retrieve/update) use REST Web Services under the hood. Leaving the scope unchecked produces a ScopeMismatched entry in the Login Audit Trail during the authorization code grant. Community verification: the recommended custom MCP role permissions explicitly include "REST Web Services (Full)" and "Perform Search" for all MCP tools to appear in tools/list.

Sources: Authorization Code Grant Flow Error Messages — Oracle NetSuite Help; Record Tools — Oracle NetSuite Help (confirms record tools use REST Web Services); NetSuite Professionals community thread on MCP tools/list permissions, 2025-10-01.

  1. The auto-created integration record from SuiteApp install typically has REST Web Services scope unchecked and must be edited

Oracle documents that integration records created by SuiteApps may need scope configuration after install. The "NetSuite AI Connector" scope alone does not grant the token access to the REST Web Services endpoint that the MCP tools invoke; REST Web Services must be an explicit scope checkbox.

Source: Create Integration Records for Applications to Use OAuth 2.0 — Oracle NetSuite Help.

  1. "Web Services Only" roles silently fail OAuth 2.0 authorization code flow

If any of your four tested roles is flagged as "Web Services Only," it will fail login validation because that restriction expects SOAP web services, not the browser-based consent flow. Worth checking on each tested role.

Source: Setting a Web Services Only Role for a User — Oracle NetSuite Help; OAuth 2.0 for RESTlets — Oracle NetSuite Help.

  1. 2FA is mandatory and can silently block the consent flow if the user has not completed initial 2FA enrollment

The OAuth 2.0 Authorized Applications Management permission requires 2FA (mandatory in all NetSuite accounts since 2018.2). If the user attempting the Claude authorization has not set up 2FA on the selected role, the consent flow fails before issuing an authorization code. This is separate from the "role does not support" message but worth ruling out.

Source: Permissions Requiring Two-Factor Authentication (2FA) — Oracle NetSuite Help.

  1. IP address rules apply to OAuth 2.0 flows — a role or company IP rule excluding claude.ai's outbound range will block token exchange

IP restriction rules are evaluated on every authenticated session including OAuth 2.0 token exchange. If your account has IP rules and Claude's outbound IPs are not allow-listed (or if your role has an IP restriction on it), step two of the flow fails.

Source: Enabling and Creating IP Address Rules — Oracle NetSuite Help.

Deep Dive

Why all four roles fail with the same error

The OAuth 2.0 Authorization Code Grant flow in NetSuite has three independent gates:

  1. Account feature — OAuth 2.0 enabled under Setup → Company → Enable Features → SuiteCloud. You've confirmed this.
  2. Integration record — Authorization Code Grant checked, Public Client configured, redirect URI match, at least one scope checked, state = Enabled. You have the record but REST Web Services scope is unchecked.
  3. Role permission — the role used at the consent screen must carry "Log in using OAuth 2.0 Access Tokens".

When a user hits the NetSuite consent URL from claude.ai, NetSuite first authenticates the user, then presents the role picker. For every role in the picker, NetSuite evaluates whether that role can authorize an OAuth 2.0 application. The check is not on the integration scope — it is on the role's explicit permission set. A role missing "Log in using OAuth 2.0 Access Tokens" is shown in the picker (because it has login rights to NetSuite) but rejected the moment it's selected for the OAuth flow. That produces the exact UI string you are seeing.

This explains why Administrator, CP Executive, and the other two all fail identically: none of them have the OAuth 2.0 login permission added by default in a stock NetSuite account. Administrator appears powerful but does not auto-include every auxiliary OAuth permission — Oracle specifically lists OAuth 2.0 permissions as ones that must be added to roles. The -System- user that auto-created the integration record has system-level rights and bypasses these role checks during install, which is why install succeeds but end-user authorization does not.

What the SuiteApp post-install wizard actually does (and what it leaves out)

The MCP Standard Tools SuiteApp install page confirms the SuiteApp auto-creates the integration record with the "NetSuite AI Connector" scope. Two things it does not do:

  • It does not add "REST Web Services" as a scope on that integration record.
  • It does not configure any role with the permissions needed for OAuth login or for invoking the MCP tools.

Oracle's intent is that administrators configure a dedicated role per MCP user. The community-verified minimum permission set for a custom MCP role is:

  • Permissions → Setup: Log in using OAuth 2.0 Access Tokens, REST Web Services (Full), User Access Tokens (if mixing TBA), SuiteAnalytics Workbook (View) — for workbook tools
  • Permissions → Lists: Perform Search (View minimum, Full recommended)
  • Permissions → Reports: relevant report view permissions for Report Tools
  • Permissions → Transactions / Lists: View-level access to every record type the user should query

Without the first two, the tools/list MCP call returns empty or partial results even after OAuth succeeds.

REST Web Services scope — what it actually enables

REST Web Services scope on the integration record authorizes the issued access token to hit the /services/rest/record/v1 and /services/rest/query/v1 endpoints. The MCP record tools (view_record, create_record, update_record) and SuiteQL tool are REST Web Services clients under the hood. Without this scope on the token, the MCP server receives authorization errors on every tool invocation even though the OAuth handshake completes. The "NetSuite AI Connector" scope alone covers the MCP streaming-HTTP endpoint but not the downstream NetSuite data endpoints the tools call into.

2FA and IP restrictions — likely vs. ruled-out

  • 2FA: Worth checking but unlikely to produce this specific error string. The user will see a 2FA challenge, not "role does not support OAuth2". If 2FA is enrolled on at least one of the tested roles, that's fine; if the role carries OAuth 2.0 Authorized Applications Management, that role is forced into 2FA enrollment before the consent can issue an auth code.
  • IP restrictions: Unlikely to produce this specific string because IP rules typically fail with "Invalid login attempt" or similar, but worth confirming under Setup → Company → Enable Features → Company → IP Address Rules, and on each role's "Restrict by IP Address" field.

Minimum-permission dedicated role ("Claude Integration") — security best practice

Start from a blank custom role, not a copy of Administrator. The community-verified and Oracle-consistent minimum set:

Subtab Permission Level Why
Setup Log in using OAuth 2.0 Access Tokens — Gates the OAuth 2.0 login itself. Without this the error fires.
Setup REST Web Services Full Required for all MCP record/SuiteQL tools.
Setup User Access Tokens Full Only if you plan to mix TBA; optional for pure OAuth 2.0. Safe to omit.
Lists Perform Search View (Full if you need Saved Search create) Required for saved-search and SuiteQL MCP tools.
Lists (per record type) View Grant View only on the record types Claude should read. Never Edit unless specifically required.
Reports (per report type) View Only if Report Tools are needed.

Do NOT add:

  • Full-level access to transactions or customer records unless the use case demands it
  • Administrator-derived permissions (overpermissioning)
  • Two-Factor Authentication Required = unchecked (2FA should stay required)

Role restrictions to set:

  • Web Services Only = unchecked (otherwise the browser consent flow fails)
  • Restrict this role by Device ID = unchecked (would break Claude's authentication)
  • IP Address Restriction = leave blank unless you want to enforce corporate VPN IPs on the MCP flow

Feynman Bridge

Think of NetSuite's OAuth 2.0 as three separate locks on the same door, not one. The first lock is the feature flag on the whole building — is OAuth 2.0 turned on at all? The second lock is the application itself — has someone created an ID card for Claude (the integration record) and stamped it with which rooms the card is allowed into (the scopes)? The third lock, the one you are hitting, is on the individual employee badge — does this specific job title (role) have the sentence "may log in with an OAuth 2.0 card reader" printed on it?

NetSuite's Administrator role is powerful but it does not automatically include that sentence. The sentence is a separate permission called "Log in using OAuth 2.0 Access Tokens", and you have to add it to every role a user might pick at the consent screen. The confusing part is that there's a similarly named permission, "OAuth 2.0 Authorized Applications Management", which sounds like the same thing but is not — it only lets the user view and revoke already-authorized apps. Adding the wrong one is a common mistake. So your fix is essentially: open each role, go to the Setup sub-tab of Permissions, and add the right one. Then check the door's second lock (the integration record's scopes) to confirm REST Web Services is one of the stamped rooms, because without it the access card gets you through the door but every internal room is locked.

Actionable Insights

Execute these in order. Each step has a ~70%+ chance of being the single remaining fix on its own; together they resolve the full problem surface.

Step 1 (highest likelihood — do first): Add "Log in using OAuth 2.0 Access Tokens" permission to every role that should use Claude

Navigation: Setup → Users/Roles → Manage Roles → click Edit next to the target role (start with Administrator for a proof-of-life test, then move to a dedicated role for production) → click the Permissions tab → click the Setup subtab → in the Permission dropdown select "Log in using OAuth 2.0 Access Tokens" → leave Level blank (this permission is binary, no level) → click Add → click Save at top.

Repeat for CP Executive and any other role you want to authorize.

Verification: Log out, log back in as a test user, go to claude.ai → Settings → Connectors → re-add the NetSuite connector. The role picker should now accept the roles you just edited.

Step 2: Add REST Web Services scope to the integration record

Navigation: Setup → Integration → Manage Integrations → find the auto-created NetSuite MCP integration record (named something like "NetSuite AI Connector" or "MCP Standard Tools") → click Edit → on the Authentication tab, confirm:

  • State = Enabled
  • Authorization Code Grant = checked
  • Public Client = checked (matches your PKCE flow)
  • Redirect URI = https://claude.ai/api/mcp/auth_callback (exact match, no trailing slash)
  • Scope section: check REST Web Services (in addition to NetSuite AI Connector) → click Save.

Verification: The saved integration record should show REST Web Services as an enabled scope. No Client Secret change is needed for a Public Client + PKCE setup.

Step 3: Rule out Web Services Only restriction on the tested roles

Navigation: Setup → Users/Roles → Manage Roles → edit each tested role → confirm the Web Services Only checkbox (near the top of the role form) is unchecked. If it's checked, uncheck it and Save.

Likelihood: low that this is the issue for Administrator, but non-zero for CP Executive and custom roles.

Step 4: Create a dedicated "Claude Integration" role (security best practice, do before production use)

Navigation: Setup → Users/Roles → Manage Roles → New

Configure:

  • Name: Claude MCP Integration
  • ID: _claude_mcp_integration
  • Subsidiary: as required for your OneWorld setup
  • Two-Factor Authentication Required: checked (keep security strong)
  • Web Services Only: unchecked
  • Restrict this role by Device ID: unchecked

Permissions tab:

Setup subtab:

  • Log in using OAuth 2.0 Access Tokens (no level)
  • REST Web Services — Full
  • User Access Tokens — Full (only if you also need TBA; otherwise skip)

Lists subtab:

  • Perform Search — Full (View minimum if you want truly read-only)
  • Documents and Files — View (for attachment tools)
  • Customers, Vendors, Employees, Items — View only
  • Any other record types Claude should read — View only

Reports subtab:

  • Report Customization — View (for Report Tools)
  • Specific reports as needed — View

Transactions subtab:

  • Transaction types Claude should read — View only (e.g., Sales Order, Invoice, Purchase Order)
  • Do NOT grant Create/Edit/Full unless you specifically intend Claude to mutate records

Click Save.

Then: Lists → Employees → Employees → edit the user who will use Claude → Access tab → Roles subtab → New Role → select "Claude MCP Integration" → Save.

Step 5: Verify 2FA enrollment on the user's chosen role

Navigation: Log in as the target user with the Claude MCP Integration role selected. If NetSuite has not yet prompted for 2FA setup (authenticator app or hardware token), complete it. Without 2FA enrollment, the OAuth consent step will silently fail partway through.

Verification: After login, go to top-right user menu → Settings → Manage Security Questions / 2FA. Confirm 2FA is enrolled.

Step 6: Check Login Audit Trail after next failed attempt

Navigation: Setup → Users/Roles → View Login Audit Trail → filter: Date = Today, Detail contains "OAuth" or "AuthorizationCode".

Map the error code per Oracle documentation:

  • FeatureDisabled → OAuth 2.0 feature off (not your case)
  • AuthorizationCodeGrantRequired → Auth Code Grant checkbox off on integration record
  • ScopeMismatched → integration scope does not include what the token request asked for (this is what the missing REST Web Services scope produces)
  • EntityOrRoleDisabled → the role or user is inactive
  • InvalidRedirectUri → Redirect URI mismatch

This is your ground truth if anything still fails after Steps 1-3.

Step 7 (if everything above passes but tools fail after auth): verify MCP tools permissions

If the OAuth handshake completes but Claude reports empty or partial tool results, you are missing tool-invocation permissions. Confirm the Claude MCP Integration role has:

  • REST Web Services (Full) — required
  • Perform Search (Full or View) — required for saved-search and SuiteQL tools
  • View-level access to every record type you want to query

Per the NetSuite Professionals community thread these are empirically required for tools/list to return the full tool catalog.

Confidence Assessment

  • Overall: high
  • Coverage gaps:

- The literal UI string "Your role does not support OAuth2 login, please select a different one" is not reproduced verbatim in Oracle's public Help Center text. The mapping from that string to the "Log in using OAuth 2.0 Access Tokens" permission is established via Oracle's documentation of which permissions gate OAuth 2.0 role eligibility, feynman analysis, and community reports. Confidence on the mapping is high but not 100%.

- Exact behavior of the SuiteApp install wizard in your specific NetSuite account version (2025.2 vs 2026.1) was not verified — some releases may auto-include REST Web Services scope. Verify on the actual integration record.

  • Conflicting signals:

- Tim Dietrich's blog reports a Claude Code-specific bug where OAuth never completes. If you are using claude.ai web (not Claude Code CLI), this does not apply. If you are using Claude Code, expect additional failure modes beyond the role issue.

  • Recommended follow-up:

- After Step 1 alone, test. If the error persists, proceed to Step 2, test again, then Step 3, and so on. Do not batch all changes before testing — you want to isolate which fix resolved it for future reference and runbooks.

- Capture the Login Audit Trail entry from the last failed attempt BEFORE making changes, so you have a baseline error code to compare against.

- If Steps 1-7 all pass and the error still fires, the next thing to verify is whether your NetSuite account has SAML SSO enforced. When SSO is enforced, OAuth 2.0 user consent routes through the identity provider, and the IdP must be configured to release the role claim — but this is an uncommon edge case worth raising with Oracle NetSuite Support only after the above steps are exhausted.
