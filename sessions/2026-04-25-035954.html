1. Platform Overview

Claudeclaw is a multi-agent orchestration platform that runs on a single Mac via macOS launchd, exposing AI agents over Telegram. It is not a hosted service or cloud product. It is a personal command-and-control system: persistent daemons on one machine, one operator (Alex), one shared SQLite database.

The platform wraps the official @anthropic-ai/claude-agent-sdk (v0.2.x). Each agent spawns the claude CLI as a subprocess. The OAuth token never leaves ~/.claude/; ClaudeClaw does not read it.

Core Characteristics

  • Single-machine: All agents run on the same Mac. Shared database, shared filesystem, shared MCP servers.
  • Telegram-native: Every agent has its own bot. Conversations are the primary interface. Voice notes are transcribed via Groq Whisper. Replies can be voiced back via ElevenLabs or fallback TTS.
  • Persistent: Sessions survive bot restarts via session ID resumption. Memory persists in SQLite with importance-weighted decay. Conversation history is logged per agent.
  • Async by design: Mission tasks and scheduled tasks run in the background. Alex fires a request and gets the result when it is done — the bot notifies him.

3. SDK: runAgent() Event Loop

3.1 How Claude Code Sessions Work

src/agent.ts is the core execution engine. It calls query() from @anthropic-ai/claude-agent-sdk, which spawns a claude subprocess and communicates over stdin/stdout via JSON-encoded events.


// src/agent.ts (simplified)
const messages: SDKMessage[] = [];
for await (const event of query({
  prompt: fullPrompt,
  sessionId: existingSessionId ?? undefined,
  abortController,
  options: {
    maxTurns: MAX_TURNS,
    cwd: PROJECT_ROOT,
    pathsToFiles: relevantFiles,
    mcpServers: mergedMcpServers,
  }
})) {
  switch (event.type) {
    case 'system':   // session init, session ID captured here
    case 'assistant': // text delta or tool use
    case 'tool_result': // result from a tool call
    case 'result':   // final usage stats
  }
  messages.push(event);
}

3.2 Session Resumption

On every agent run, src/db.ts looks up the last session_id for (chat_id, agent_id). If found, the session is resumed — Claude Code continues the prior context window. If the session has been compacted (context overflow auto-summary), a new session is started and the prior session ID is discarded.

Session IDs are stored in:


CREATE TABLE sessions (
  chat_id TEXT NOT NULL,
  agent_id TEXT NOT NULL DEFAULT 'main',
  session_id TEXT NOT NULL,
  created_at INTEGER DEFAULT (strftime('%s','now')),
  PRIMARY KEY (chat_id, agent_id)
);

3.3 MCP Server Loading

MCP servers are merged from two sources at session start:

  1. Project-level: /Users/alex/dev/claudeclaw/.claude/settings.json
  2. User-level: ~/.claude/settings.json

Both mcpServers objects are shallow-merged, with the project-level config taking precedence.

3.4 Tool Tracing: agent_events Table

Every tool invocation is traced in agent_events:


CREATE TABLE agent_events (
  id TEXT PRIMARY KEY,
  session_id TEXT,
  agent_id TEXT,
  event_type TEXT,      -- 'tool_use', 'tool_result', 'text', 'error'
  tool_name TEXT,       -- e.g. 'Bash', 'Read', 'WebSearch'
  tool_use_id TEXT,     -- matches tool_use → tool_result pairs
  input TEXT,           -- JSON: tool input parameters
  output TEXT,          -- tool result (truncated at 2000 chars)
  latency_ms INTEGER,   -- wall-clock time tool_use → tool_result
  success INTEGER,      -- 1 or 0
  created_at INTEGER DEFAULT (strftime('%s','now'))
);

Latency is tracked by recording tool_use_id → Date.now() when a tool_use event fires, then computing Date.now() - startTime when the corresponding tool_result arrives.

3.5 AbortController Timeout

Every agent run is wrapped in an AbortController. The default timeout is MISSION_TIMEOUT_MINUTES <em> 60 </em> 1000 milliseconds (30 minutes as of April 2026, recently extended from 10 minutes).


const abortController = new AbortController();
const timeout = setTimeout(() => {
  abortController.abort();
}, MISSION_TIMEOUT_MINUTES * 60 * 1000);

If the timeout fires, the in-flight Claude Code subprocess is killed, the mission task is marked failed with a timeout error, and the session ID is cleared (forcing a fresh session next time).

4.2 Encryption

WhatsApp and Slack message bodies are encrypted at rest using AES-256-GCM before storage. The encryption key is in .env as DB_ENCRYPTION_KEY (auto-generated on first run if absent). 3-day auto-purge removes expired records.

5. Mission Queue

5.1 Lifecycle

A mission task progresses through these states:


queued → running → completed
              ↘ failed
              ↘ cancelled (from queued only)

5.2 claimNextMissionTask

The scheduler polls for due missions every 60 seconds. Claiming is an atomic SQLite transaction to prevent double-assignment across concurrent processes:


// src/db.ts (simplified)
claimNextMissionTask(agentId: string): MissionTask | null {
  return db.transaction(() => {
    const task = db.prepare(`
      SELECT * FROM mission_tasks
      WHERE assigned_agent = ? AND status = 'queued'
      ORDER BY priority DESC, created_at ASC
      LIMIT 1
    `).get(agentId);
    if (!task) return null;
    db.prepare(`
      UPDATE mission_tasks SET status = 'running', started_at = ? WHERE id = ?
    `).run(Date.now() / 1000, task.id);
    return task;
  })();
}

5.3 completeMissionTask

On completion (success or failure), the result is written back:


completeMissionTask(id: string, result: string): void
failMissionTask(id: string, error: string): void

The principal's Telegram bot also DMs Alex the full output in its own chat. Clarke's job is to receive the result via polling, synthesise it, and reply in Clarke's own chat.

5.4 Polling Pattern

Clarke polls after creating a mission:


MID=$(node dist/mission-cli.js create --agent principal_strategy --title "..." "prompt" | grep -oE "[0-9a-f]{8}")
for i in $(seq 1 20); do
  RESULT=$(node dist/mission-cli.js result "$MID" 2>&1)
  echo "$RESULT" | grep -q "status: completed" && break
  echo "$RESULT" | grep -q "status: failed" && break
  sleep 30
done

Maximum polling window: 10 minutes (20 × 30s). Task timeout: 30 minutes (changed from 10 minutes in April 2026). This means a task can be running but not-yet-completed when Clarke stops polling. The scheduler continues to track it; Clarke should re-check with mission-cli.js result <id> if needed.

5.5 Auto-Assignment

If a mission is created without --agent, Gemini Flash classifies the prompt against the agent roster and assigns to the best match. The dashboard exposes unassigned tasks in a kanban view for manual override.

6. Scheduler

6.1 runDueMissionTasks Loop

src/scheduler.ts runs two loops on the same 60-second tick:

  1. Scheduled tasks: cron-based recurring prompts
  2. Mission tasks: one-shot async tasks (queued by mission-cli.js or dashboard)

Both loops share the same runAgent() execution path. The only difference is state management: scheduled tasks update next_run and stay active; mission tasks transition to completed or failed and are not retried.

6.2 Scheduled Task Execution


tick (every 60s)
  ↓
SELECT * FROM scheduled_tasks WHERE status='active' AND next_run <= now()
  ↓
For each due task:
  ├── UPDATE next_run = next cron occurrence (BEFORE running, to prevent double-fire)
  ├── UPDATE status = 'running'
  ├── Enqueue in per-chat message queue
  ├── runAgent(agentId, chatId, prompt, freshSession=true)
  ├── Store result in last_result
  └── UPDATE status = 'active'

Critical detail: next_run is advanced before execution begins. If the agent crashes or times out, the task will still fire again at the next scheduled time rather than being stuck in running.

6.3 Timeout Enforcement

The scheduler wraps every task execution in an AbortController with MISSION_TIMEOUT_MINUTES <em> 60 </em> 1000 ms (30 minutes). On timeout, the subprocess is killed, the task result is set to an error string, and the session ID is cleared.

7. Memory System

7.1 Five-Layer Context Injection

Before every agent run, src/memory.ts builds a [Memory context] block injected at the start of the prompt:

Layer Source Trigger
Semantic recall Vector cosine search (embedding similarity > 0.3) Always
Recent high-salience importance >= 0.5, ordered by accessed_at Always
Consolidation insights consolidations table, pattern clusters Always
Hive Mind Other agents' recent hive_mind entries Always
Conversation recall conversation_log, last N turns Phrases like "remember when"

7.2 Memory Extraction (Gemini Flash)

After every completed turn, src/memory-ingest.ts sends the assistant's response to Gemini Flash with a structured extraction prompt:


Extract:
- summary (1–2 sentences)
- entities (names, companies, tools — JSON array)
- topics (domains, themes — JSON array)
- importance (0.0–1.0)
  0.9–1.0: irreversible decisions, commitments, architectural choices
  0.7–0.8: significant preferences, project milestones
  0.5–0.6: useful context, recurring topics
  < 0.5: skip (not saved)

Only memories with importance >= 0.5 are persisted. Embeddings are generated via the OpenAI Embeddings API and stored as JSON float arrays in the embedding column.

7.3 Importance Decay

Decay runs daily via a scheduled task in the main process:

Importance Range Daily Decay Half-life
0.8–1.0 1% ~69 days
0.5–0.7 2% ~35 days
Pinned 0% Permanent

Memories that decay below 0.1 are soft-deleted (marked superseded = 1). Contradicted memories are also marked superseded by the consolidation pass.

7.4 Consolidation (Pattern Detection)

src/memory-consolidate.ts runs every 30 minutes in the main process only (to prevent race conditions). It:

  1. Groups recent memories by topic cluster
  2. Uses Gemini to identify patterns, contradictions, and thematic clusters
  3. Writes consolidations records (compressed insights)
  4. Marks superseded memories where contradictions are found

10. Multi-Agent Communication

10.1 Direct Delegation (Inline)

In any conversation, directing a message with @<agentId>: syntax triggers inline delegation via src/orchestrator.ts:


@principal_architect: spec out a caching layer for the pipeline enrichment API

The orchestrator:

  1. Loads the target agent's CLAUDE.md and memory context
  2. Constructs a full prompt (system + memory + user message)
  3. Runs runAgent() in-process (not as a subprocess)
  4. Returns the result inline to the calling conversation
  5. Logs the delegation to inter_agent_tasks and hive_mind

Inline delegation timeout: 5 minutes (separate from mission timeout).

10.2 Mission Tasks (Async)

For work that takes longer than a few minutes, Clarke creates a mission task via mission-cli.js. The principal picks it up on the next scheduler tick, runs it, and DMs Alex in its own Telegram chat.

Clarke also polls for the result and synthesises it back into the main chat.

10.3 Hive Mind (Passive Awareness)

All agents write significant actions to hive_mind. The memory context injection layer surfaces recent hive mind entries to every agent, so Clarke knows what Architect last shipped without being explicitly told.

11. Voice Pipeline

11.1 Speech-to-Text

Voice notes sent to any Telegram bot are downloaded as .ogg files and transcribed via the Groq Whisper API. The transcript is prepended with [Voice transcribed]: and handled as normal text.

Fallback if Groq is unavailable: macOS whisper CLI (if installed locally).

11.2 Text-to-Speech

Voice responses use a cascade:

  1. ElevenLabs (best quality, voice cloning) — if ELEVENLABS_API_KEY and ELEVENLABS_VOICE_ID are set
  2. Gradium AI (free tier, 45k credits/month) — if GRADIUM_API_KEY is set
  3. macOS say + ffmpeg (offline, no API required) — always available as final fallback

Voice mode is toggled per-chat via /voice command. When enabled, the agent's text response is also synthesized and sent as a voice note.

12. WhatsApp and Slack Integration

12.1 WhatsApp

Uses whatsapp-web.js (browser automation via Chromium DevTools Protocol). A separate wa-daemon.sh process manages the browser session at port 4242 (HTTP API) and 9222 (CDP).

Alex authenticates by scanning a QR code on first launch. The session is persisted in store/waweb/. Messages older than 3 days are auto-purged. Message bodies are encrypted at rest (AES-256-GCM).

WhatsApp mode is entered via /wa command in Clarke's chat. Alex can list conversations, read threads, and reply from within Telegram.

12.2 Slack

Uses the @slack/web-api client with a user OAuth token (xoxp-). Supports:

  • List conversations (sorted by unread)
  • Read messages with pagination
  • Send threaded replies

Slack mode is entered via /slack. Messages are encrypted at rest with 3-day retention.

13. Security Model

13.1 Auth Layers (outer to inner)

  1. Telegram chat ID restriction: only ALLOWED_CHAT_ID can interact. All other senders are silently ignored.
  2. Private chat only: group messages are rejected.
  3. PIN lock: bot starts locked after deploy or idle timeout. SHA-256 salted hash stored in .env. Idle auto-lock: IDLE_LOCK_MINUTES (default 30).
  4. Emergency kill phrase: if EMERGENCY_KILL_PHRASE is spoken, all agent processes immediately exit.
  5. Destructive confirmation: commands that match patterns (rm -rf, git push --force, DROP TABLE) require explicit confirmation before execution.

14. Known Gaps

14.1 ~/.claudeclaw is Not Git-Versioned

Agent config, CLAUDE.md files, and agent.yaml files live in ~/.claudeclaw/agents/. This directory is not under version control. If the Mac is lost or rebuilt, all principal system prompts must be reconstructed manually.

Mitigation options:

  • Symlink ~/.claudeclaw into the repo and gitignore only the secrets (.env files)
  • Back up via Time Machine (implicit, not explicit)
  • Add a scripts/export-agents.sh that commits agent configs to a private repo

14.2 principal_devops Not Yet Scaffolded

The DevOps principal (GitHub repo creation, CI/CD wiring, Vercel/Fly deploys) is referenced in the architecture but does not exist. Currently:

  • Architect produces plans and inits local git repos
  • Engineer implements and commits on feature branches
  • No agent pushes to GitHub or deploys to any infra

This is a conscious deferral, not an oversight. When Alex confirms the scope, architect will scaffold it and engineer will implement.

14.3 research_a / research_b Sub-Agents

These exist as launchd plists (or can be created via agent-create-cli.js) for parallel research missions but are not routinely active. There is no formal "fan-out and gather" API; Clarke manually creates two mission tasks and synthesises when both complete.

14.4 No Distributed Tracing

All agents share one SQLite database on one machine. There is no distributed trace ID linking a Clarke message → mission task → principal run → hive mind entry. Debugging a cross-agent chain requires manual correlation via created_at timestamps and session_id values.

14.5 Inline Delegation Timeout vs Mission Timeout

Inline delegation (@agent: syntax) has a 5-minute timeout. Mission tasks have a 30-minute timeout (as of April 2026). Tasks that routinely take 10–30 minutes should use mission tasks, not inline delegation. This distinction is not enforced by the system — it is a convention documented in CLAUDE.md.

14.6 Session Collision Risk

The per-chat message queue (src/message-queue.ts) serialises messages within a single process. However, if two launchd processes for the same agent ID were somehow started (e.g., misconfigured plist), they would both attempt to resume the same session ID. The KeepAlive + PID lock in src/index.ts mitigates this, but is not bulletproof across a hard kill + rapid restart.

15. Key Data Flows

15.1 Incoming Telegram Message (Normal Turn)


Telegram polling (grammy)
  → handleMessage()
    → [Auth: allowed chat ID check]
    → [PIN lock check]
    → [Kill phrase check]
    → [Delegation detection: @agent: syntax?]
      YES → orchestrator.delegateToAgent()
             → load target CLAUDE.md + memory context
             → runAgent(targetAgentId, ...)
             → log inter_agent_tasks + hive_mind
             → return result to caller
      NO  → enqueue in per-chat FIFO message queue
              → runAgent(mainAgentId, chatId, prompt)
                → build memory context (5 layers)
                → query() [Claude Code subprocess]
                  → [tool use loop]
                    → log each tool: agent_events (latency, success)
                  → stream text delta → Telegram (progressive update)
                → capture: tokens, cost, context_tokens, did_compact
                → log: token_usage, conversation_log
                → extract memories: Gemini Flash
                  → embed: OpenAI embeddings
                  → save memories (if importance >= 0.5)
              → format + send final response

15.2 Mission Task Lifecycle


mission-cli.js create --agent principal_strategy "prompt"
  → INSERT mission_tasks (status: queued)
  → print task ID

scheduler tick (every 60s)
  → claimNextMissionTask('principal_strategy')  [atomic transaction]
  → runAgent(agentId, systemChatId, prompt, freshSession)
    → [same execution flow as normal turn]
  → completeMissionTask(id, result)
  → principal_strategy Telegram bot DMs Alex

Clarke polls (or Alex checks later)
  → mission-cli.js result <id>
  → synthesises + replies in Clarke's chat

15.3 Scheduled Task Execution


scheduler tick (every 60s)
  → SELECT due scheduled_tasks
  → for each:
      UPDATE next_run = next_cron_occurrence  [atomic, before run]
      UPDATE status = 'running'
