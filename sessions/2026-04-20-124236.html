# Research: Extracting SKILL.md files from video/audio content

## Executive Summary

A working pipeline has three tractable layers (ingest, transcription, extraction) and one hard layer (the narrated-specific-event → general-procedure "modality shift"). The transcription layer is a solved problem: `yt-dlp` + WhisperX (self-hosted) or Deepgram Nova-3 (managed) deliver diarized, word-aligned, technically-adequate transcripts for roughly $0.20 per 45-minute video. The extraction layer is where the research frontier is: no published tool targets "reusable agentic skill extraction" specifically, and the genuinely novel work for the user's pipeline is forcing schema-first, hedge-classified, timestamp-grounded extraction that distinguishes the speaker's one-time story from the general rule the skill needs to encode.

## Key Findings

1. **Transcription is a commodity; technical-term accuracy is the only real ASR challenge.**
   No mainstream ASR handles spoken code well. The two practical mitigations are Whisper's `initial_prompt` / Deepgram Nova-3's Keyterm Prompting (pre-loaded jargon) plus an LLM post-pass that cross-checks auto-captions vs. Whisper output as adversarial sources.
   Source: [WhisperX GitHub](https://github.com/m-bain/whisperX) — canonical pipeline combining faster-whisper large-v3 + wav2vec2 alignment + pyannote diarization.

2. **WhisperX (self-hosted) or Deepgram Nova-3 (managed) is the clear tooling choice for 2025.**
   Local WhisperX on a 12GB GPU processes a 45-minute talk in ~5 minutes at zero marginal cost. Deepgram Nova-3 costs ~$0.19 for the same talk with better managed diarization. OpenAI's `whisper-1` is behind the frontier (no diarization, based on large-v2).
   Source: [Deepgram docs](https://developers.deepgram.com/) — Nova-3 model specs and Keyterm Prompting.

3. **Schema-first extraction with verbatim source spans is the load-bearing extraction pattern.**
   The dominant 2024-2025 production pattern is: define a Pydantic/JSON schema for the skill, require a `source_quote` (contiguous substring of transcript) and `timestamp` per step, validate by string search, then render Markdown deterministically from validated JSON. Asking the LLM to emit final Markdown directly loses validation.
   Source: [Instructor (jxnl/instructor)](https://github.com/jxnl/instructor) — dominant Pydantic-based extraction library with retries and validation.

4. **The `description` field is the product of skill extraction, not the body.**
   Anthropic Agent Skills use progressive disclosure: only `name` + `description` are always in context; the body loads on activation. The description is effectively a router rule. Video narrators almost never state the trigger condition explicitly, so the description must be *inferred* from the task context, not copied from narration. This is where most extraction effort should concentrate.
   Source: [Anthropic Agent Skills docs](https://docs.claude.com/en/docs/agents-and-tools/agent-skills) — progressive disclosure mechanism; [anthropics/skills](https://github.com/anthropics/skills) — reference implementations.

5. **The narrated-event → general-procedure shift is the most subtle extraction failure mode.**
   Speakers describe a specific session ("I noticed the port was in use, so I killed the old process"). Skills need the general rule ("Before starting the server, check port occupancy; if occupied, confirm with the user before killing the holder"). This is a modality transformation on top of the person/register shifts and is the step naïve prompts (`rewrite in imperative mood`) miss. Hedge detection ("usually", "sometimes", "I think") is the explicit signal.
   Source: No direct published prior art; closest analogs are runbook extraction from incident postmortems and wikiHow procedural-knowledge extraction. [HowTo100M paper](https://arxiv.org/abs/1906.03327).

6. **One video ≠ one skill.**
   A 45-minute developer talk typically contains 3-7 distinct workflows. Single-pass extraction tends to merge or drop sub-procedures. Segment-by-task-boundary first, then extract per segment, then dedupe is the pattern that recovers the full skill set.
   Source: [LangChain summarization tutorial](https://python.langchain.com/docs/tutorials/summarization/) — map-reduce/refine patterns for long documents.

7. **No existing tool targets agentic skill extraction specifically.**
   Meeting intelligence (Fathom, Supernormal, Otter, tl;dv, Granola) extracts summaries and action items. Video-to-summary tools (NoteGPT, Eightify, Mindgrasp) extract bullet notes and mind maps. None produce a schema'd reusable workflow artifact with hedge classification, timestamp grounding, and imperative-mood rewriting. The gap is real.

## Deep Dive

### The four-layer architecture

The pipeline decomposes cleanly into four layers. Three are commoditized and one is the frontier:

| Layer | Purpose | Commoditized? | Best 2025 choice |
|-------|---------|---------------|------------------|
| Ingest | Get audio + metadata off YouTube/podcast host | Yes | `yt-dlp` with cookies + info JSON |
| Transcribe | Audio → timestamped, diarized, technically-accurate text | Yes | WhisperX (local) or Deepgram Nova-3 (managed) |
| Segment + extract | Transcript → schema'd candidate skills with grounded source quotes | Partially | Schema-first LLM extraction with multi-sample voting |
| Rewrite to SKILL.md | Generalized procedure in imperative voice, good description | **No** | Custom two-pass: generalize-then-rewrite |

### Layer 1: Ingest (yt-dlp)

The canonical one-shot command [Source 1]:

yt-dlp \

-f "bestaudio[ext=m4a]/bestaudio" \

--extract-audio --audio-format wav --audio-quality 0 \

--write-subs --write-auto-subs --sub-langs "en.*,en-orig" \

--convert-subs srt \

--write-info-json --embed-metadata \

--cookies-from-browser chrome \

--output "%(id)s/%(title)s.%(ext)s" \

""



Three pieces of metadata matter:
- **`info.json` chapters** — pre-segments the talk into topic windows, reducing the LLM's segmentation burden.
- **Auto-captions** (`en-orig`) — not authoritative, but a free second ASR source. Use the disagreement between auto-caps and Whisper as a flag for likely transcription errors in technical terms.
- **Video title + channel name + chapter titles** — feed these as Whisper's `initial_prompt` to bias decoding on domain jargon.

Gotchas (2024-2025): bot-detection on cloud IPs requires `--cookies-from-browser`, and PO-token requirements occasionally force `--extractor-args "youtube:player_client=ios"` as fallback [Source 1 wiki].

### Layer 2: Transcription

Self-hosted winner: **WhisperX with large-v3**. It is the only mainstream project that bundles faster-whisper transcription + wav2vec2 forced alignment (real word-level timestamps, not Whisper's interpolated ones) + pyannote 3.1 diarization in one pipeline [Source 2]. For a 45-min talk on a 12GB GPU: ~5 min wall-clock, ~$0 marginal.

Managed winner: **Deepgram Nova-3 with Keyterm Prompting** at ~$0.0043/min (~$0.19 for 45 min). Nova-3's Keyterm Prompting is the most capable commercial feature for technical vocabulary — it supports multi-word phrases like `"npm install"` as atomic keyterms [Source 3].

Code-snippet detection remains an open problem. Mitigations in descending effectiveness:
1. Pre-load domain jargon via `initial_prompt` (Whisper) or Keyterm Prompting (Deepgram), built from video title + chapter titles + a hand-curated project vocab list.
2. LLM post-pass with both Whisper output and YouTube auto-captions as inputs, asking for a corrected transcript where they disagree.
3. Multimodal OCR of slides/terminal frames to cross-reference tool names (optional, higher-value content).

### Layer 3: Segment + Extract

This is where most pipeline failures happen. Two failure modes dominate:

**Failure 1: one-skill-per-video over-simplification.** A 45-minute talk usually contains 3-7 distinct procedures. Single-pass extraction flattens them. The fix is segment-first, extract-per-segment, dedupe-and-merge [Source 4].

**Failure 2: hallucinated steps and fabricated commands.** The model pattern-matches what a "typical" developer workflow looks like and fills in gaps with plausible-but-wrong content. The fix is schema-enforced verbatim grounding: every extracted step must carry a `source_quote` that post-hoc string-matches against the transcript, and rejected steps trigger regeneration.

The recommended schema (Pydantic/JSON):

class SkillStep:

order: int

action: str # imperative, rewritten

source_quote: str # verbatim transcript substring

timestamp_start: str # "HH:MM:SS"

timestamp_end: str

hedge_class: Literal["prescriptive", "default", "situational", "preference"]

commands: list[str] # any CLI/code mentioned

preconditions: list[str]

class Skill:

name: str # kebab-case

description: str # router trigger, <200 chars

when_to_use: list[str]

when_not_to_use: list[str]

inputs: list[str]

steps: list[SkillStep]

branches: list[dict] # conditional sub-flows

examples: list[str]

constraints: list[str]

source_video_url: str

source_segment: tuple[str, str] # start/end timestamps in video



**Extraction prompt pattern (three passes):**

1. **Pass 1 — skeleton extraction.** Per segment, extract an ordered list of steps with `source_quote` + `timestamp` + `hedge_class` only. Skip commands and preconditions. Temperature 0.3, N=3 samples.

2. **Pass 2 — self-consistency voting.** Cluster the 3×N steps by semantic similarity; keep only steps with ≥2/3 agreement. Flag disagreements as "low-confidence, review".

3. **Pass 3 — enrichment.** For each consensus step, enrich with commands, preconditions, and gotchas by re-reading the cited transcript window only. This is Chain-of-Density-style iterative enrichment [Source 5].

**Hedge classification is the key step** the rest of the ecosystem misses. The LLM explicitly tags each statement as:
- `prescriptive` ("you must", "always") → steps[] main path
- `default` ("usually", "I typically") → steps[] main path with note
- `situational` ("if X, then", "when Y") → branches[]
- `preference` ("I like to", "my style") → notes or discard

### Layer 4: Rewrite to SKILL.md

This is where the pipeline most differentiates from generic video summarizers. Three transformations happen together:

| Axis | Source form | Target form |
|------|-------------|-------------|
| Person/mood | "So what I do is..." (1st person, descriptive) | "Do X." (2nd person, imperative) |
| Register | Hedging, filler, backtracking | Terse, declarative |
| Modality | Narrated specific event | General rule |

The third is the one generic summarizers fail. "I noticed the port was in use, so I killed the old process" must become "Before starting the server, check port occupancy; if occupied, confirm with the user before killing the holder." This is a generalization step that must be explicit in the prompt, not hoped for as a side-effect of "rewrite in imperative mood".

**The `description` field authoring is separable and critical.** The description is the router; it gates activation. Write it *last*, *from the generalized rule*, not from the transcript. Structure: `<verb phrase of what skill does>. Use when <specific trigger>. Not for <negative scope>.` Example: `Fill fillable PDF form fields from a JSON spec. Use when the user provides an AcroForm PDF and a data file. Do not use for scanned PDFs (use ocr-extract) or layout generation (use pdf-generate).`

### Validation

Each extracted skill must ship with three cheap tests [Source 6]:

1. **Triggers manifest** — 3 prompts that should activate the skill and 3 that shouldn't. Install the skill alongside 10+ distractor skills and measure activation precision/recall. This tests whether the description is pulling its weight.
2. **Golden transcript** — one worked example from start to finish with expected intermediate steps. Replay after edits.
3. **Source-fidelity footer** — a `## Source` section at the bottom of SKILL.md linking to the video + timestamp range, so later reviewers can audit the extraction.

An LLM-as-judge pass comparing skill behavior against the source transcript [Source 7] is cheap insurance for newly extracted skills.

## Feynman Bridge

Think of a YouTube developer tutorial as a cooking show. The chef is making one specific dish, but you want to walk away with a recipe card you can cook yourself next week. The audio track tells you mostly what you need (ingredients, steps, timing) but the chef is constantly mixing two different things: "here's what I'm doing right now, with this fish" and "here's the general rule for cooking fish". A recipe card needs the second one, not the first.

Now imagine a factory that takes cooking show recordings and spits out recipe cards. The first three stations are easy: a machine rips the audio off YouTube, a speech-to-text robot types out everything that was said (with a note for each word saying whether the chef or a guest said it), and a sorter cuts the transcript into sections — one per distinct dish. The fourth station is the hard one. It has to read each section, notice the difference between "I added a pinch of salt because this particular fish was under-seasoned" and "always salt fish before searing", throw away the one-time story, keep the general rule, and rewrite it as an instruction card that tells you what to do, not what the chef did. That last station is the only part of the factory that isn't already built somewhere else, and it is the piece worth investing real engineering effort in.

## Actionable Insights

1. **Adopt a four-layer architecture explicitly: ingest → transcribe → extract → rewrite.** Do not conflate extract and rewrite into one LLM call. The extract step produces grounded, hedge-classified, schema-valid JSON. The rewrite step is a separate pass with a strict style contract (imperative, second person, no hedges). Conflating them is why generic summarizers produce unusable skill files.

2. **Concrete proposed pipeline (MVP, ~1 week to ship):**

Input: YouTube URL (or local audio/video file)

Step 1: yt-dlp → audio.wav + auto.srt + info.json (with chapters)

flags: --extract-audio --audio-format wav --write-auto-subs

--write-info-json --cookies-from-browser chrome

Step 2: WhisperX large-v3 with initial_prompt built from

{title, channel, chapter titles, project jargon list}

→ transcript.json (word-aligned, diarized)

Step 3: LLM post-pass (Claude Sonnet) — reconcile WhisperX vs auto.srt

for technical terms, output transcript.md with [HH:MM:SS] anchors

Step 4: Segment: LLM call with chapter hints → list of candidate skill

segments with start/end timestamps

Step 5: Per segment, extract with Instructor + Pydantic schema:

- 3 samples at T=0.5

- enforce source_quote substring check

- semantic cluster + majority vote

- hedge-classify each step

Step 6: Enrichment pass: for each consensus step, fill commands,

preconditions, gotchas by re-reading cited window only

Step 7: Rewrite pass: three-axis transformation (person, register,

modality) with explicit "generalize from narrated example" prompt

Step 8: Description synthesis: author description last, from generalized

rule. Template: ". Use when . Not for ."

Step 9: Render SKILL.md from validated JSON (deterministic, no LLM)

+ scaffold folder with reference/source.md containing

transcript excerpt + video URL + timestamp range

Step 10: Auto-generate a triggers.yaml manifest (3 should-fire, 3

should-not-fire prompts) and run a routing precision test

Output: directory at ~/.claude/skills// containing:

- SKILL.md

- reference/source.md

- tests/triggers.yaml



3. **Tool recommendation per stage** (ship-ready defaults):
   - Ingest: `yt-dlp` (only real choice)
   - Transcribe: `WhisperX` with `large-v3-turbo` if self-hosting, else `Deepgram Nova-3` with Keyterm Prompting
   - Extraction: `Instructor` + Pydantic + Claude Sonnet (Claude over GPT because skills target Claude Code; match training distribution)
   - Orchestration: single Python script in `deepagents/tools/` exposing as an MCP tool or a Claude Code skill

4. **Budget the extraction effort where it matters: the `description` field and the modality shift.** A weak description means the skill never activates; a weak modality shift means the skill encodes a story instead of a rule. Both fail silently. Allocate 60% of LLM spend to extract+rewrite passes, 20% to description synthesis, 20% to transcription.

5. **Ship this *as* a Claude Code skill immediately.** The natural first product is a skill called `video-to-skill` whose protocol *is* the pipeline above. This dogfoods the output format and gives the user a single-command workflow: `@video-to-skill <youtube-url>`. The skill folder contains `scripts/ingest.sh`, `scripts/extract.py`, `reference/schema.py`, and `templates/SKILL.md.jinja`.

6. **Include a source-fidelity footer in every generated SKILL.md.** Final section `## Source` with video URL, timestamp range, author attribution, and extraction date. This is cheap, makes every extracted skill auditable, and distinguishes the user's library from opaque generic summarizers.

7. **Validate every extracted skill with three cheap tests before committing it to the library.** Routing test (3 positive + 3 negative prompts), golden replay (one end-to-end session), and LLM-as-judge faithfulness check against the source transcript. Without these, skill-library quality degrades silently as the library grows.
