Roadrunner CLI — Design Document

Date: 2026-04-24

Author: principal_architect (via Clarke)

Status: Current — reflects ROAD-010 merge

1. What It Is

Roadrunner is a deterministic agentic loop harness for Claude Code. It solves the problem of a language model that doesn't stop: once you drop it into a project with a tasks.yaml, it will keep running, picking up tasks, validating its own work, and moving forward until the roadmap is done. Python owns control flow. Claude owns implementation. Bash hooks enforce boundaries.

The tool grew out of dogfooding: the roadrunner-cli roadmap is itself driven by roadrunner. Every ADR in docs/adr/ records a real defect found during that self-hosting run and the fix applied.

2. Architecture

2.1 The three-layer split


Layer 1: Control Plane (roadrunner.py, 2083 lines)
  Task selection, dep resolution, validation, state writes

Layer 2: Hooks (hooks/*.sh → roadrunner.py Python entry points)
  Stop, SessionStart, PreCompact, PostCompact, PostToolUse

Layer 3: Claude Code (implementation agent)
  Reads brief, writes code, runs validate/complete

No task advances without passing validation_commands. No loop exits without an explicit ROADMAP_COMPLETE signal or a hard iteration cap. Claude cannot bypass either — they are enforced at the Python layer, not by prompt.

2.2 roadrunner.py structure

The file is a single-module CLI dispatcher (2083 lines). Internal organisation:

Section Responsibility
Constants (lines 86-93) Tunables: timeouts, attempt caps, schema versions, log rotation
TypedDicts Task, RoadmapState, ContextSnapshot — typed state schemas
State I/O Atomic YAML/JSON read-write, schema migration, fcntl locking
Validation engine Subprocess executor with timeout, structured trace output
Log management Rotation, retention, per-task work logs, trace.jsonl
Hook implementations write_context_snapshot, cmd_session_start
Task lifecycle cmd_start, cmd_validate, cmd_complete, cmd_block, cmd_reset
Status and analysis cmd_status, cmd_next, cmd_analyze, cmd_health, cmd_watch
Loop control cmd_check_stop, iteration counter, auto-block logic
Scaffolding cmd_init — copies hooks, settings.json, creates tasks.yaml stub
Utility cmd_commit, cmd_snapshot, cmd_post_compact, cmd_reset_iteration
CLI dispatch argparse registration, main()

17 subcommands: status, next, start, validate, complete, block, reset, reset-iteration, commit, health, watch, check-stop, snapshot, post-compact, init, analyze, session-start

2.3 Hook system

All hooks are thin shell wrappers that pipe stdin to a Python entry point in roadrunner.py. The convention (ADR-010) is that no logic lives in the shell files — they exist purely to satisfy Claude Code's requirement for an executable at a registered path.

Hook Fires when Python entry Behaviour
stop_hook.sh Claude finishes a response turn cmd_check_stop Decides block/continue/halt; emits JSON decision to stdout
session_start_hook.sh Claude Code session initialises cmd_session_start Resets session_iteration to 0; injects context snapshot as additionalContext
precompact_hook.sh Before context compaction cmd_snapshot Writes .context_snapshot.json with active task, iteration counts, last events
postcompact_hook.sh After compaction completes cmd_post_compact Reads snapshot, verifies schema, logs post_compact_verify to trace.jsonl
post_write_hook.sh After file write (direct shell check) Lightweight guard; no Python dispatch

Hook paths are registered in .claude/settings.json using $CLAUDE_PROJECT_DIR/hooks/... (ADR-005). Relative paths silently fail when hooks fire from a different working directory.

Stop hook decision protocol — two distinct JSON shapes, never mixed:


// Loop continues: inject brief and resume
{"decision": "block", "reason": "Continue. Task ROAD-009 is eligible..."}

// Hard halt: iteration cap reached
{"continue": false, "stopReason": "Max iterations (100) reached. Roadmap halted."}

Per the Claude Code spec, {"continue": false} overrides any decision: block and terminates the session. The hook emits silent exit 0 (no JSON) in three cases: stop_hook_active=true (re-entrant guard), ROADMAP_COMPLETE detected, and post-halt cleanup.

2.4 YAML task queue

tasks/tasks.yaml is the source of truth. Every load validates the schema before use. Writes are atomic: write to a tempfile, fsync, os.replace (ADR-004). Rolling backups at tasks/tasks.yaml.bak[.1..5] (configurable via TASKS_BACKUP_KEEP).

Task schema (all fields):


- id: ROAD-001              # required; regex ^[A-Z]+-\d+$ enforced (ADR-008)
  title: "..."              # required
  status: todo              # todo | in_progress | done | blocked
  depends_on: []            # list of task IDs; checked before eligibility
  goal: "..."               # multi-line description; injected into task brief
  acceptance_criteria: []   # list of testable criteria; surfaced in brief
  validation_commands: []   # shell commands run by cmd_validate; exit 0 = pass
  validation_timeout: 300   # optional per-task override (default 300s, ADR-008)
  documentation_targets: [] # files the agent should update on completion
  files_expected: []        # scope guard; listed in the brief
  notes: "..."              # optional context injected into brief

2.5 Dependency graph

cmd_analyze runs a depth-first traversal to detect cycles and unreachable tasks before execution. is_eligible() (called by cmd_check_stop and cmd_next) checks that all depends_on entries are done before a task is surfaced.

3. State Model

3.1 State files

File Schema version Purpose
tasks/tasks.yaml - Task definitions and status; source of truth
.roadmap_state.json v2 (current) Active task, iteration counters, per-task attempt map
.context_snapshot.json v1 (current) PreCompact snapshot; read by SessionStart
logs/CHANGELOG.md - Append-only audit log; rotated at 10 MB
logs/ROAD-XXX.md - Per-task work log with validation output
logs/trace.jsonl - Structured event log; retained 7 days
.roadmap_state.lock - fcntl advisory lock; survives os.replace on state file
.reset_ROAD-XXX - Boundary marker written on task completion

3.2 Iteration counter model (ROAD-010)

Before ROAD-010, a single iteration counter incremented on every Stop-hook fire and persisted across all claude invocations. With 92 lifetime iterations against a cap of 50, the loop was halting at the start of every new session before any work could happen.

ROAD-010 split the counter:

Field Semantics Reset behaviour
iteration Lifetime audit counter Preserved across sessions; never decremented
session_iteration Per-session runaway guard Reset to 0 on every cmd_session_start

cmd_check_stop now gates on session_iteration. Default cap raised from 50 to 100 (approved 2026-04-24 in resolution plan: per-session is a runaway guard, not a budget cap; 100 gives headroom for legitimate multi-task runs while still catching true runaways). STATE_SCHEMA_VERSION bumped to 2; older state files without session_iteration load cleanly via setdefault(0) migration.

reset-iteration --soft resets session_iteration only. --hard zeros both. Both counters surface in cmd_status and in the task brief injected by the Stop hook.

3.3 Concurrency protection (ADR-009)

fcntl.flock() wraps the read-increment-write cycle in cmd_check_stop. The lockfile (.roadmap_state.lock) is a stable sibling to the state file — it survives the os.replace that atomically swaps in new state. This serialises concurrent Stop-hook fires (which can occur on rapid model responses) without data races.

4. Tunables

All configurable via constants at the top of roadrunner.py:

Constant Default Description
DEFAULT_VALIDATION_TIMEOUT 300s Per-task validation subprocess timeout; overridable in tasks.yaml
MAX_TASK_ATTEMPTS 5 Auto-block a task after N resume cycles without completion
TASKS_BACKUP_KEEP 5 Rolling .bak.N copies retained after each tasks.yaml write
LOG_ROTATE_BYTES 10 MB Rotate when a log file exceeds this size
LOG_RETAIN_DAYS 7 Delete rotated logs older than this
STATE_SCHEMA_VERSION 2 Current .roadmap_state.json schema; version mismatch exits 2
SNAPSHOT_SCHEMA_VERSION 1 Current .context_snapshot.json schema

Environment override: ROADMAP_MAX_ITERATIONS — passed as --max-iterations to check-stop by the Stop hook shell script, allowing per-project override without touching source.

8. Trust Model and Security Notes

  • shell=True is used in subprocess.run for validation commands. Pragmatic choice: validation_commands in tasks.yaml are project-authored, not user-supplied at runtime. Risk documented and accepted.
  • Task IDs validated against ^[A-Z]+-\d+$ before use in file paths (ADR-008), preventing path traversal.
  • All state writes use os.replace atomically (ADR-004). No partial state observable by a concurrent reader.
  • fcntl.flock on .roadmap_state.lock serialises concurrent Stop-hook fires (ADR-009).
