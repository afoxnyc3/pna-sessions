entra-triage

Status: 🟡 In development — Phase 1 (Roadrunner-driven; see tasks/tasks.yaml)

Microsoft Entra ID security triage and response. Single-tenant operations console (one Entra tenant per deployment; see ADR 0005).

What it does

  • Ingests Entra identity telemetry (sign-ins, audit logs, risky users, role changes) via Microsoft Graph
  • Runs specialized agents to triage identity risk and detect anomalies
  • Evaluates tenant posture against Microsoft best-practice baselines
  • Surfaces bounded response actions behind approval gates — v1 is advisory; analysts execute approved actions manually in the Azure portal (see ADR 0006)
  • Full audit trail for every state change

Quick start (local, mock mode)

entra-triage — Spec

Problem

Organizations running Microsoft Entra ID have fragmented visibility into identity security events: sign-ins, audit logs, risky users, role changes, application registrations, and service principals. Microsoft publishes detailed security operations guidance but customers need an opinionated tool that continuously ingests telemetry, triages events, checks posture against best-practice baselines, and proposes or executes constrained response actions — with a full audit trail and analyst-friendly UI.

Proposed Solution

A single-tenant identity security operations tool that connects to one Entra ID tenant via Microsoft Graph, ingests identity telemetry, runs specialized agents to triage risk and detect anomalies, evaluates posture against Microsoft best-practice baselines, and surfaces findings in a clean analyst UI. Response actions that are destructive or sensitive require explicit approval. Everything is auditable.

Initial deployment target: Chelsea Piers or a personal dev Entra tenant. Not an MSP platform. Not multi-tenant.

Non-Goals

  • No multi-tenancy, no tenant switcher, no MSP/MSSP features in v1
  • No support for on-prem Active Directory in v1
  • No offensive security, password spraying, or credential testing
  • No automatic disabling of users, deleting applications, or rewriting Conditional Access policies without explicit approval
  • Not a full SIEM
  • No Sentinel workspace dependency in v1

Acceptance Criteria

Tech Stack

Architecture

See ARCHITECTURE.md. High-level: - apps/web — analyst UI (dashboard, incident explorer, baseline, approvals) - apps/api — tRPC API, all data access, auth - apps/workers — ingestion + triage + baseline workers - packages/agents — 6 specialized Anthropic agents - packages/connectors — Microsoft Graph client with mock mode - packages/policy — action approval matrix and guardrails - packages/shared — types, zod schemas, Drizzle schema

Risks

  1. Graph API rate limits and eventual consistency — Likelihood: H, Impact: M — Mitigation: backoff, cursor checkpointing, dead-letter
  2. Licensing gaps (P1 vs P2 for ID Protection signals) — Likelihood: H, Impact: M — Mitigation: graceful degradation, surface warnings in UI
  3. Agent LLM cost at ingestion volume — Likelihood: M, Impact: M — Mitigation: rule-based pre-filter before agent invocation (ADR 0003)
  4. Secret management — Likelihood: L, Impact: H — Mitigation: credentials in .env only, never in DB or logs

Open Questions

  1. Auth for the web UI — defaulting to a single hardcoded operator login (username/password in .env) for v1; proper auth is v2
  2. Report export format — defaulting to PDF via headless browser; Excel is v2
  3. Baseline control evaluation cadence — defaulting to on-demand + daily scheduled run
  4. Approval notifications — defaulting to console/log alerts in v1; email/Slack in v2

Entra Connection (single-tenant)

This is a single-tenant app. The connected Entra tenant is configured entirely via environment variables — no DB table, no UI for managing connections.

GRAPH_CLIENT_ID=        # App registration client ID in the target tenant
GRAPH_CLIENT_SECRET=    # Client secret (never stored in DB)
GRAPH_TENANT_ID=        # The Entra Directory ID to monitor

On startup, the API and workers validate these credentials and confirm required Graph permissions before ingestion begins. If MOCK_GRAPH_API=true, the mock client is used and no real credentials are needed.

Ingestion Data Flow

sequenceDiagram
    participant G as Microsoft Graph
    participant IW as IngestionWorker
    participant DB as PostgreSQL
    participant TW as TriageWorker
    participant A as Agent (pre-filter then LLM)

    IW->>DB: Load ingestion cursor
    IW->>G: GET /auditLogs/signIns?$skipToken=cursor
    G-->>IW: page of events + nextLink
    IW->>IW: Dedup by event_id + content_hash
    IW->>DB: INSERT sign_in_events (skip dupes)
    IW->>DB: UPDATE ingestion_cursors
    TW->>DB: Fetch unprocessed events (batch)
    TW->>TW: Rule-based pre-filter (pass / flag / ambiguous)
    TW->>A: Ambiguous events only
    A-->>TW: incidents[] + confidence scores
    TW->>DB: INSERT incidents (deduped by actor+category hash)
    TW->>DB: INSERT audit_log

Response Action Flow

sequenceDiagram
    participant A as Agent
    participant PE as Policy Engine
    participant DB as PostgreSQL
    participant AN as Analyst

    A->>PE: evaluateAction(action)
    PE-->>A: AutoExecutable | ApprovalRequired | Prohibited
    alt AutoExecutable
        A->>DB: Execute + INSERT response_actions (completed)
    else ApprovalRequired
        A->>DB: INSERT response_actions (pending)
        AN->>DB: Review + approve
        A->>DB: Execute + UPDATE response_actions (completed)
    else Prohibited
        A->>DB: INSERT audit_log (blocked)
    end

Roadmap — entra-triage

Implementation is driven by Roadrunner. Canonical task definitions with validation commands live in tasks/tasks.yaml. This file is the narrative overview.

Phase 1 — Reconcile & Foundation (ENTRA-001 → 003)

Goal: Align code with the single-tenant DB schema, get every workspace building, and land the Drizzle ORM layer.

  • ENTRA-001 — Reconcile entity Zod schemas with db/migrations/001_initial.sql. Delete tenant.ts and connector.ts (out of scope per ADR 0005); strip tenantId from the remaining 12 entity files; fix db/seed.sql (removes broken INSERT INTO tenants).
  • ENTRA-002 — npm install + npm run build green across all workspaces. Type/import fixes only, no business-logic changes.
  • ENTRA-003 — Drizzle ORM layer in packages/shared/src/db/ (schema.ts, db.ts, migrate.ts). Mirrors every table in the migration exactly.

Exit criteria: npm run build passes across the monorepo; packages/shared exports a working Drizzle client; no tenantId references anywhere in packages/shared/src/entities/.

Phase 2 — Graph Connectors (ENTRA-010, 011)

Goal: Production-grade Graph client + full mock parity for local dev.

  • ENTRA-010 — Real client with @odata.nextLink pagination, 429/503 exponential backoff honoring Retry-After, Zod-validated returns. Covers sign-ins, audit-logs, risky-users, directory, applications.
  • ENTRA-011 — Mock client as a full drop-in: same interface, realistic fixtures, P1/P2 license-tier variants for graceful degradation on ID Protection endpoints.

Exit criteria: MOCK_LICENSE_TIER=P1 MOCK_GRAPH_API=true runs end-to-end without touching real Graph; packages/connectors builds + tests pass.

Phase 3 — Pipeline (ENTRA-020, 030, 031, 032, 040)

Goal: End-to-end ingestion → triage with deterministic pre-filter + LLM agents on the ambiguous slice.

  • ENTRA-020 — Ingestion worker with per-stream cursor checkpointing, content-hash dedup, dead-letter for malformed payloads, pino redaction of PII.
  • ENTRA-030 — Agent base refactor to Anthropic tool use for structured output + mandatory confidence scores (per ADR 0002).
  • ENTRA-031 — SignInTriage + IdentityRisk agents on the new base.
  • ENTRA-032 — DirectoryChange + BaselineReview + ResponsePolicy + Research agents. ResponsePolicyAgent returns ApprovalRequired for all write-type actions in v1 (per ADR 0006 — no AutoExecutable until v2).
  • ENTRA-040 — Triage worker: rule-based pre-filter (PASS / FLAG / AMBIGUOUS — ADR 0003), agent dispatch, incident dedup hash, audit_log writes.

Exit criteria: MOCK_GRAPH_API=true end-to-end produces incidents in the DB; audit_log has a row per incident; dedup prevents second-run duplicates.

Phase 4 — API + UI (ENTRA-050, 051, 060, 061)

Goal: Analyst-facing application — read-heavy API, single-operator UI.

  • ENTRA-050 — tRPC read routes: incidents (list/get/updateStatus/suppress), cases (list/get/create/addIncident/updateStatus), auditLog (read-only). JWT middleware gates all procedures.
  • ENTRA-051 — tRPC action routes: baseline, responseActions, ingestion. responseActions includes propose/approve/reject/markExecutedManually — no execute procedure in v1 (ADR 0006).
  • ENTRA-060 — UI shell: login (OPERATOR_USERNAME/OPERATOR_PASSWORD from .env), dashboard layout with sidebar, dashboard page (risk posture + ingestion health + recent incidents), incidents explorer with filters and detail drawer.
  • ENTRA-061 — UI: baseline review page, approvals page (no Execute button — shows “manual execution” guidance + “Mark executed” action per ADR 0006), settings page (connection status, permission checker, ingestion controls, licensing-gap banner).

Exit criteria: Analyst can log in, view incidents, approve a proposed response action, mark it as manually executed, and see the full lifecycle in the audit log.

Phase 5 — Test Hardening (ENTRA-070, 080, 081, 090)

Goal: Tests, local-dev polish, E2E confidence.

  • ENTRA-070 — Vitest unit tests: policy engine determinism, event parser normalization, dedup hash stability.
  • ENTRA-080 — docker-compose.yml (postgres:16, redis:7) + scripts/dev-setup.sh (check node, compose up, wait for postgres, apply migration, seed, print “Ready”).
  • ENTRA-081 — Integration tests against a live postgres container: ingestion lifecycle (3 events → 3 rows → dedup on re-run), triage lifecycle (anomalous event → incident + audit_log), response-action lifecycle (propose → policy gate ApprovalRequired → approve → markExecutedManually → completed, per ADR 0006).
  • ENTRA-090 — Playwright E2E: login, dashboard, incidents filter + detail, approvals flow, baseline review.

Exit criteria: npm run test green across workspaces; bash scripts/dev-setup.sh produces a working local env from a clean clone; npm run e2e --workspace=apps/web passes all five specs.

Backlog (v2 candidates)

  • Scoped write Graph permissions — User.RevokeSessionsAll first (lowest blast radius), then narrow additions per ADR 0006 upgrade path. Unlocks AutoExecutable classification + tool-driven execute routes.
  • Real Entra credentials and staging environment against a live tenant
  • Multi-tenant support if product expands beyond single-org use (would supersede ADR 0005)
  • Proper auth (Entra ID as IdP, or SSO) replacing v1 env-var login
  • Email/Slack notifications for Critical/High incidents and pending approvals
  • Excel report export (v1 is PDF)
  • Sentinel workspace export
  • On-prem Active Directory support
  • HashiCorp Vault for secret management
  • Anomaly detection pre-filter ML model (v1 is rule-based)
