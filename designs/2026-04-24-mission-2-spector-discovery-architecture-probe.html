Mission 2: Spector Discovery & Architecture Probe

Context

This brief is the direct downstream of AEC Data & RAG Pattern Survey (merged best-of-3).

That brief established three facts with unanimous [3/3] support across parallel research replicates:

  1. Production AEC RAG requires three retrieval primitives — text-RAG for docs, Graph-RAG for BIM, symbol-spotting + OCR for drawings — not one.
  2. Only two substrates realistically support a vendor-neutral multi-tenant firm AI: Autodesk AEC Data Model (GraphQL, beta, ACC-locked) or Speckle (open object graph, Apache 2.0, self-hostable).
  3. Stable object IDs (IFC GlobalId, Revit UniqueId, Speckle id) are the glue that links a chunked spec paragraph to a specific element. Without them, cross-asset RAG degrades to keyword matching.

Spector is a candidate AEC RAG product surface. Mission 2 determines where Spector sits in this architectural space: which substrate, which primitives, which trade-offs, and whether it fits Alex’s multi-tenant firm AI thesis.

Objective

Produce an architectural read on Spector grounded in first-party evidence. Map Spector against the three-primitive framework, the substrate question, and the multi-tenancy posture. Deliver a go / investigate-further / pass recommendation with reasoning.

Probe questions

All seven probes are derived directly from the merged research brief. Answer each from primary sources where possible; flag inference where not.

  1. Ingestion substrate. Does Spector ingest via Speckle, ACC AEC Data Model, direct IFC, native Revit / ArchiCAD connectors, or a mix? This decision constrains everything downstream.
  2. BIM retrieval path. Is the 3D path vector-only, or does it include a graph layer (Neo4j, Memgraph, or equivalent)? Does it support element-parameter queries natively, or only over serialised IFC?
  3. Drawing retrieval. Sheet-level OCR only, or per-region symbol detection? Any ColPali / BLUEPRINT-class pipeline? Does it count objects or just surface sheets?
  4. Multi-tenancy implementation. Project-scoped vector namespaces, row-level security in Postgres, or separate graph DBs per tenant? Can one tenant’s embedding leak into another tenant’s retrieval?
  5. Citation and audit trail. Do responses cite IfcGUIDs / ElementIds / sheet-coordinate anchors? Is there a human-verifiable audit trail per claim? AEC liability requires this; absence is disqualifying for firm deployment.
  6. Cloud lock-in surface. Which features require ACC, iTwin, Procore, or proprietary Spector cloud? Which are portable / self-hostable?
  7. Geometric similarity. Does Spector offer any “find elements like this” primitive, or is it deferred like every other shipping vendor? (The open whitespace flagged in the upstream brief.)

Deliverables

  1. Architecture scorecard: Spector’s posture on each of the seven probes, with source citations.
  2. Substrate map: place Spector on the Autodesk / Speckle / iTwin / hybrid grid.
  3. Fit analysis: does Spector meet the thesis of a multi-tenant firm-wide AEC AI? Where does it fall short?
  4. Risk flags: lock-in dependencies, missing primitives, opacity areas.
  5. Go / investigate / pass recommendation with 3-sentence rationale.

Constraints

  • Cite primary sources (Spector docs, product pages, demos, GitHub, ToS, pricing pages, vendor blog). Flag any inference where primary evidence is unavailable.
  • Use the upstream merged brief as the comparative lens. Do not re-derive the three-primitive framework.
  • Output format: markdown, ≤1,500 words, with substrate map as a table.
  • If Spector is genuinely opaque, say so plainly. A clear “cannot determine from public surfaces” is better than speculation.

Dispatch pattern

Run as 3x best-of-N on research_a (same protocol as upstream). Once research_b’s academic_search is verified functional post engineer fix 07628a75, upgrade subsequent missions to full 3x3 A/B.

Success criteria

  • Every probe answered or explicitly marked “cannot determine.”
  • Substrate and primitive posture clear enough that Alex can decide: buy, partner, build alternative, or ignore.
  • Recommendation is specific, not hedged.
