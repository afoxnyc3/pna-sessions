agent-expert-kit — Spec

Problem

Claude agents have no persistent memory of a domain across sessions. Re-explaining architecture, schemas, conventions wastes tokens and leaks inconsistency. Existing CLAUDE.md files describe how an agent works, not what it knows. There is no durable, versioned layer for domain knowledge that stays in sync with the code.

Proposed Solution

A pna-library skill that scaffolds the Disler “agent expert” triad for any domain. One invocation produces three files in a canonical layout, seeded from current code, YAML-validated, and wired to a cadence for re-sync. The result: a machine-readable mental model the agent reads before acting and revises after.

Non-Goals

  • Not a general documentation generator. Output is structured YAML keyed to an agent’s Q&A workflow, not human-facing docs.
  • Not a replacement for CLAUDE.md. Persona/behaviour stays in CLAUDE.md. This is the knowledge layer.
  • Not multi-repo. Scope is one repo per domain for v1. Cross-repo domains are v2.
  • Not auto-syncing. Re-sync is an explicit command (self-improve-expertise), not a file watcher or git hook in v1.
  • Not generating YAML from nothing. The skill expects source files as input for seeding; will not fabricate architecture.

Acceptance Criteria

Tech Stack

Layer Choice Rationale
Skill format Markdown skill file (SKILL.md + templates/) in pna-library Matches existing library; single distribution mechanism via symlinks.
Templating Plain Markdown/YAML with $VAR substitution + small Python helper No Jinja dependency. Aligns with agent-experts repo style. Python used elsewhere in hooks.
YAML validation PyYAML (yaml.safe_load) Already required by self-improve.md. Zero new deps.
Scope seeding Glob + Read tools (Claude-native) for file collection; Claude completion for YAML extraction No external doc-scraper. Keeps skill portable.
Target integration library skill and skill_roster.yaml symlink pattern Existing distribution path; no parallel system.

Architecture

See ARCHITECTURE.md. High-level:

  • Scaffolder (skill entry) takes inputs and writes three files from templates.
  • Seeder reads scope files, produces initial YAML body via structured prompt.
  • Validator runs trim-to-cap loop and compile check.
  • Registrar updates skill_roster.yaml so the triad becomes discoverable.

Risks

  1. LLM-produced YAML is malformed — Likelihood: M, Impact: M — Mitigation: compile-check gate + up to 2 auto-retry with error message injected into regeneration prompt.
  2. 1000-line cap is too tight for large domains — Likelihood: M, Impact: L — Mitigation: cap is configurable via MAX_LINES variable; document precedent for raising selectively.
  3. Seeded YAML hallucinates architecture not present in code — Likelihood: M, Impact: H — Mitigation: every seeded claim must cite a file path in the scope; post-seed pass rejects unsourced rows.
  4. Symlink conflicts when re-scaffolding — Likelihood: L, Impact: L — Mitigation: prompt before overwrite; keep a .bak of prior expertise.yaml on regenerate.
  5. Domain definition drift (two agents scaffold overlapping “creator” domains) — Likelihood: M, Impact: M — Mitigation: registrar checks for slug collision and suggests merge or rename.

Open Questions

  • Default expertise location: per-agent (~/.claudeclaw/agents/<id>/experts/) or shared (~/.pna/experts/)? Default applied: shared at ~/.pna/experts/<domain>/, symlinked into each consuming agent’s skill path. Rationale: research_a/b A/B only works if both see the same mental model.
  • Re-sync cadence: on-demand (v1 default), cron-scheduled (v2), post-commit hook (v3).
  • Should question.md support multi-domain queries (fanning across experts)? Default applied: no in v1. One question = one domain. Fan-out handled by a separate expertise-ensemble skill if/when needed.

Integration with Existing Skills

  • till-done: called by plan-build-improve between phases as the verification gate.
  • library: registrar writes to library.yaml under a new top-level experts: key; skill_roster.yaml is deepagents routing config and is out of scope. The library list experts subcommand (T2.1b) reads this key. No parallel system.
  • skill-optimizer: target for optimizing SKILL.md once Phase 1 ships. Build an eval harness from Phase 4 dogfood domains.
  • meta-optimize: prune the skill family quarterly; re-rank against usage.

Non-Architecture (explicit)

  • No daemon. No file watcher. No cron in v1. Sync is user-invoked.
  • No cross-repo awareness. One domain = one repo in v1.
  • No web UI. All invocation through Claude Code / principal agents.
