principal-plugin

The Principal & Agent consulting methodology, encoded as a Claude skills plugin.

Status

active — MVP-5 shipped, v1 build in progress (12 skills remaining)

What it is

A composable Claude skills plugin that runs an end-to-end AI consulting engagement: wedge (SaaS audit), discovery, design, governance, delivery, and retrospective. Five skills are production-ready; twelve are backlog. Each engagement reads from and writes to a separate pna-case-base repo that compounds pattern library across engagements.

What it is not

  • Not a hosted application. Deployment surface is Claude Code / Claude.ai, installed via plugin or skill copy.
  • Not a customer-facing agent. Consumer is the operator (Alex) running the engagement.
  • Not a replacement for judgement. Skills enforce sequence and quality bar; they do not replace the consultant.
  • Not a multi-tenant product in v1. Personal operator tooling first; productisation is a later, separate decision.

Quick start

# Install skills into Claude Code
cp -r skills/* ~/.claude/skills/
# Or use the P&A library system
library add principal-plugin

principal-plugin — Spec

Problem

P&A engagements today are manually reproduced each time. Every new client conversation starts from a blank page, and the methodology lives only in Alex’s head. Without encoded methodology, engagements take longer to scope, vary in quality, cannot be delegated to sub-agents, and do not compound across clients. Competitors with similar AI-consulting pitches are entering the market; P&A needs a defensible repeatable delivery asset.

Proposed Solution

Package the full P&A engagement lifecycle as a Claude skills plugin (principal-plugin), with seventeen skills spanning five phases (Wedge, Discovery, Design, Govern, Deliver). Each skill ships as a versioned SKILL.md with a pushy description, inputs, procedure, output template, failure modes, and hand-off to the next skill. A companion repo (pna-case-base) stores anonymised engagement records and extracted patterns; engagement-retrospective writes to it, pattern-recognition reads from it, and it becomes the compounding asset over time. Distribution uses the existing pna-library system and symlinks into ~/.claude/skills/.

Non-Goals

  • Not building a hosted SaaS product. v1 is operator-only tooling.
  • Not replacing human consultant judgement. Skills enforce sequence and quality bar; they do not auto-decide.
  • Not integrating directly with external CRMs, calendars, or invoicing systems in v1.
  • Not supporting multi-tenant use. Single operator, single case base.
  • Not shipping fine-tuned models or custom training. Skills use stock Claude models only.
  • Not adding voice or chat front-ends. Runtime is Claude Code / Claude.ai.
  • Not encoding industry-specific variants (health, finance, legal) in v1. Generic methodology first; vertical packs later.
  • Not a replacement for pna-library. This plugin is consumed via library; it does not replicate library’s distribution logic.

Acceptance Criteria

Tech Stack

Layer Choice Rationale
Skill format anthropics/skills SKILL.md with YAML frontmatter Matches Claude’s native skill loader. Zero custom parser. Interoperable with existing pna-library.
Packaging Flat skills/<name>/ dirs, no build step Skills load as source. Build pipelines add failure modes without value for markdown files.
Distribution pna-library + symlinks into ~/.claude/skills/ Already in use across P&A; avoids inventing a second system. Librarian agent owns this lane.
Case-base storage Separate git repo, markdown files with YAML frontmatter Versioned, diffable, searchable with obsidian-cli and grep; no runtime dependency. Keeps engagement data out of the plugin repo.
Eval harness skill-optimizer + Karpathy binary-assertion loop Alex’s existing standard. Librarian runs loops automatically (Fri 5pm cron already scheduled).
Validation writing-skills skill (linter), critic pass per ADR Catches missing frontmatter, dead hand-offs, non-pushy descriptions before ship.
Voice/style enforcement Per-skill style guide in CLAUDE.md + retro-pass during PR review Lightweight; avoids a custom style-checker that would drift from intent.
Versioning SemVer at plugin level, skill-level version field in frontmatter Consumers pin plugin version; librarian cron enforces.

Architecture

See ARCHITECTURE.md for topology and sequence diagrams. High-level:

  • Plugin repo (this) hosts SKILL.md files, examples, evals, and plugin-level CLAUDE.md.
  • Case-base repo (pna-case-base) hosts engagement records and extracted patterns. Read by pattern-recognition, written by engagement-retrospective.
  • Library system (pna-library) distributes the plugin into operator environments via symlinks.
  • Runtime is Claude Code or Claude.ai; skills load on demand based on description match.

Risks

  1. Case-base bootstrap: no real case data on day 1. pattern-recognition returns empty until ≥3 cases seeded. — Likelihood: H, Impact: M — Mitigation: seed with Spector + two synthetic-but-grounded cases before shipping pattern-recognition.
  2. Skill under-triggering: Claude may not fire skills reliably without aggressive descriptions. — Likelihood: M, Impact: H — Mitigation: enforce pushy-description rule in critic pass; run skill-optimizer loops on each skill before v1 release.
  3. Lifecycle drift: operator fires skills out of phase order, degrading output. — Likelihood: M, Impact: M — Mitigation: each SKILL.md’s When to use it section names prerequisite skills; hand-off prompts at end of every skill reinforce order.
  4. Operator single point of failure: methodology lives in one operator’s head even after encoding. — Likelihood: M, Impact: H — Accepted for v1; mitigation deferred to productisation phase.
  5. Case-base data sensitivity: real engagement records contain client-identifying info. — Likelihood: H, Impact: H — Mitigation: mandatory anonymisation step in engagement-retrospective; case-base repo private; no CI that egresses case content.
  6. Plugin install drift across machines: symlinks break on new laptop or fresh Claude Code install. — Likelihood: M, Impact: L — Mitigation: /bootstrap command idempotent; daily-notes already flag a backup task for ~/.claude/skills/.
  7. 12-skill backlog scope creep: building all 12 skills at once invites half-finished work. — Likelihood: M, Impact: M — Mitigation: roadmap sequences them one-per-week max; critic pass gate per skill.
  8. Competitive leakage: methodology is the moat; publishing the repo publicly surrenders it. — Likelihood: H (if public), Impact: H — Mitigation: repo stays private on GitHub; only anonymised case studies are externally visible.

Open Questions

None. Any remaining ambiguity has been defaulted and documented inline. Specific defaults applied:

  • Case-base repo is private (per Risk 8).
  • Plugin repo is private for v1 (can be flipped to public later if commercialised).
  • Skill distribution via pna-library (not a separate plugin marketplace).
  • No vertical packs in v1; Alex revisits after 3 engagements.

Cases write via engagement-retrospective. Patterns promote from emerging (1 case) to confirmed (≥2 cases). Anti-patterns are single-case by default; duplication strengthens confidence.

Component Responsibilities

  • skills/ — each skill is self-contained: frontmatter, procedure, output template, hand-off. No shared runtime code.
  • docs/adr/ — architectural decisions with context and alternatives. Read before proposing deviations.
  • research/ — source material (xlsx toolkit, original package README). Not part of the runtime contract; kept for traceability.
  • CLAUDE.md (plugin-level) — voice rules, lifecycle sequence, quality bar, operator wisdom. Read when a skill is authored or edited.

Install Surfaces

  1. Direct copy: cp -r skills/* ~/.claude/skills/. Manual, immediate, no dependencies.
  2. Library: library add principal-plugin. Symlinks from pna-library checkout.
  3. Bootstrap: /bootstrap command (to be added in T0.2) does both idempotently.

Non-Components (deliberately excluded)

  • No build step. Markdown is the artefact.
  • No CI for the runtime (skills are interpreted by Claude). CI can be added later for lint and eval regression; not required for v1.
  • No database. Case base is flat files.
  • No web UI. All interaction through Claude Code / Claude.ai.
  • No telemetry. Operator-private tooling.

Dependencies: Phases 1-4 complete (delivery depends on upstream artefacts).

Exit criteria: - Spector engagement produces retrospective committed to case base. - /bootstrap on a fresh Claude Code install pulls the plugin and all 17 skills fire on their triggers in a smoke test. - Pattern library has ≥4 records across ≥2 archetypes. - README includes one public anonymised case link.

SWOT

SWOT — principal-plugin

Architect’s honest read. Written to be useful, not flattering.

Strengths

  1. The methodology is real and scar-tissued. Every skill is written against actual operator experience (272K-event security investigation, 6→400-user rollout, $171K cost takeout). Competitors pitching “AI consulting” skill packs cannot fake this. It shows in the procedure sections and the failure-mode lists.
  2. Skill format is native to the runtime. No custom loader, no build step, no hosted infra. SKILL.md with pushy frontmatter is exactly what Claude expects. Zero integration tax on install.
  3. Sequence is enforced through prose, not code. ADR-0003 chose discipline over orchestration. That keeps skills independently testable and portable. Also means no runtime to break.
  4. Distribution is already solved. pna-library exists; librarian agent already runs grooming cron. Shipping a new plugin is not a new problem.
  5. MVP-5 covers every lifecycle phase. Wedge (saas-decommission), spine (problem-decomposition), taste (agentic-workflow-solutioning), moat (threat-model), compounding (engagement-retrospective). You can run an engagement end-to-end today, not after 17 skills are built.
  6. Threat-model as a gate is an actual moat. Most AI-consulting pitches stop at the architecture diagram. Forcing a STRIDE-for-agents pass before build is a credibility multiplier with CIOs and CTOs.
  7. Voice is consistent. The prose reads like one operator. That makes the deliverables feel like they came from a human consultant, which is the point.

Weaknesses

  1. Case base does not exist. engagement-retrospective and the future pattern-recognition skill both require pna-case-base. Until it is scaffolded and seeded, the compounding loop is theoretical. This is the single biggest v1 blocker.
  2. No evals or reference outputs shipped. Plugin-level CLAUDE.md promises evals/ and examples/ per skill; none exist. This violates your own quality bar as stated.
  3. 12 skills still to build. That’s most of the plugin. MVP-5 is a spine, not a product. Anyone installing today sees gaps in Phase 1, Phase 3, and Phase 4.
  4. Methodology is single-operator. If Alex is hit by a bus, the plugin is the only artefact; it captures procedure but not taste. Encoding taste takes more cycles than one-pass authoring.
  5. No sub-agent integration yet. Principal skills exist alongside ClaudeClaw’s principals (architect, engineer, librarian) but there is no documented hand-off: who fires saas-decommission when a lead arrives? This is operator-driven today; it could be delegated.
  6. Skill triggering is unmeasured. Pushy descriptions are hopeful, not verified. Without a routing log it is hard to know which skills under-fire in live use. Librarian’s skill-optimizer loop is available but not yet applied here.
  7. No public proof yet. Repo is private, no case study is published, no marketing surface beyond Alex’s network. Zero external signal.

Opportunities

  1. Case base becomes the moat over time. Every retro compounds. After 10 engagements the plugin stops being “methodology” and becomes “a consulting firm in a repo”. Very few competitors will build this because it requires doing the engagements.
  2. Anonymised case studies as marketing. Each engagement can produce a public case study (in pna-sessions) linking three or more skills. That becomes the lead magnet the plugin itself cannot be (because it is private).
  3. Vertical packs. Once generic lifecycle is proven, derivative packs (health-principal, finance-principal, legal-principal) are cheap to ship and command premium pricing. ADR-0002 and repo separation make vertical packs additive, not forks.
  4. Productise later. If the methodology proves repeatable, wrapping it as a hosted app for non-technical consultants is a realistic Year-2 move. v1 operator-first keeps optionality.
  5. Training data for downstream agents. Once the case base has records, it feeds future fine-tunes or RAG augmentations of a P&A-specific consulting agent. Data gathered as a side effect of shipping work.
  6. Integration with ClaudeClaw pipeline. saas-decommission could auto-trigger when a prospect replies to outreach with budget objection language. Not v1, but a credible Phase 6+ loop.
  7. Competitive timing. AI-consulting as a category is ~12 months from saturation. Shipping this methodology now, before the category commoditises, is the right moment.

Threats

  1. Anthropic changes skill format or loader. Low probability in the short term, but the plugin’s entire runtime contract is owned by Anthropic. Mitigation: keep skill files format-minimal; avoid reliance on undocumented behaviour.
  2. Competitor publishes a similar methodology publicly. Once the category shape is visible, a competitor could open-source a comparable plugin and undercut the moat. Mitigation: keep the case base private (that is the irreplaceable asset), not the methodology files.
  3. Client data leakage through case base. Any leak would be existential. Mitigation in place (anonymisation, private repo, no CI egress) but the risk is high-impact. Any new tooling that touches the case base must re-run threat-model.
  4. Skill under-firing in real use. If Claude’s description-match heuristics shift, pushy descriptions may stop triggering reliably. Mitigation: skill-optimizer loop on a cron; monitor routing logs once they exist.
  5. Operator bottleneck. Alex is the only person who can currently run the methodology end-to-end with judgement. Every client engagement goes through one calendar. Mitigation deferred to productisation.
  6. Roadmap drag. 12 skills across 5 phases is a lot of solo authoring. If two quarters pass with v1 incomplete, competitors may catch up. Mitigation: librarian agent can author skills to spec; architect reviews. Do not do this solo.
  7. Compliance surface grows faster than the skills. New regulations (EU AI Act, state-level AI laws, sector-specific rulings) continuously expand compliance-scan scope. Mitigation: case-base patterns/compliance/ tracks regs encountered; scan skill updates per engagement, not on a cadence.

The one line

Principal’s strongest asset is not the five skills; it is the case base that does not exist yet. Build pna-case-base next, or the compounding loop is a promise you cannot keep.

ADR-0001 Skill Pack over Hosted App

ADR-0001: Ship Principal as a skill pack, not a hosted application

  • Status: accepted
  • Date: 2026-04-21
  • Deciders: Principal Architect

Context

The Principal methodology could be distributed as (a) a hosted web application, (b) a multi-agent system, (c) a Claude Code skill pack, or (d) a plain PDF methodology document. Each has different handoff characteristics, cost profiles, and maintenance burdens. The consumer is a technical operator (Alex) running engagements in Claude Code; secondary consumers are other P&A sub-agents (architect, engineer, librarian).

Decision

Ship as a flat Claude skill pack: one SKILL.md per capability, distributed through the existing pna-library system, installed into ~/.claude/skills/ via symlink. No hosted runtime, no build step, no custom parser.

Alternatives Considered

  • Hosted web application. Gives a branded UI and non-technical access, but creates ongoing infra ownership, auth, tenant isolation, and support burden. v1 consumer is technical and already in Claude Code. Rejected.
  • Multi-agent system. Would let skills run as separate agents with specialised context. Adds orchestration tax, debugging complexity, and duplicates work already done in ClaudeClaw. Single-agent-with-skills is the simpler pattern until a specific skill proves it needs isolation. Rejected.
  • PDF or Notion methodology doc. Zero integration with runtime; requires human interpretation on every engagement; does not compound. Rejected.

Consequences

Easier: - Zero infra to maintain. - Clean handoff to client operators who already use Claude Code. - Skills compose with other Claude skills the operator already has. - Versioning via git; rollback is trivial.

Harder: - No GUI. Non-technical operators cannot use this directly (accepted; not the target audience for v1). - Skill triggering depends on Claude’s description match; requires pushy descriptions and ongoing tuning. - No central telemetry; have to rely on retrospectives to learn what is and is not working.

ADR-0002: Host the case base in a separate repo

  • Status: accepted
  • Date: 2026-04-21
  • Deciders: Principal Architect

Context

engagement-retrospective writes engagement records and pattern-recognition reads them. These artefacts contain client-identifying information and accumulate across engagements independently of the plugin’s code lifecycle. They are consumed by the plugin at runtime but should not ship inside the plugin repo.

Decision

Create pna-case-base as a separate private GitHub repo. Flat markdown files with YAML frontmatter. Directory schema: cases/, patterns/triggers/, patterns/architectures/, patterns/anti-patterns/, patterns/playbooks/, plus TAXONOMY.md. Plugin skills reference it by path convention, not by import.

Alternatives Considered

  • Inside principal-plugin repo. Simpler to bootstrap but conflates methodology with client data. Any future public release of the plugin becomes risky. Rejected.
  • Database (SQLite or Postgres). Structured, queryable, but adds a runtime dependency, backup story, and egress risk. Markdown grep covers 95% of retrieval needs at zero infra cost. Rejected.
  • Obsidian vault folder. Compatible with operator’s existing tooling but coupled to local-only storage; no GitHub history; harder to share with future collaborators or sub-agents. Rejected.

Consequences

Easier: - Plugin repo stays free of client data; can be made public later if commercialised. - Case base can grow independently; schema evolves via PRs. - Anonymisation is a repo-level policy enforceable at commit time. - Sub-agents (architect, librarian) can reference case-base paths without taking a code dependency on the plugin.

Harder: - Two repos to keep in sync on schema changes. - Cross-repo tooling (e.g. integration tests) requires path conventions the skills must respect. - Bootstrap is a two-step install (/bootstrap handles this, per T0.2).

ADR-0003 Sequential Lifecycle Enforcement

Source: /Users/alex/dev/projects/principal-plugin/docs/adr/0003-sequential-lifecycle-enforcement.md

ADR-0003: Enforce sequential lifecycle through prose and hand-offs, not code

  • Status: accepted
  • Date: 2026-04-21
  • Deciders: Principal Architect

Context

The engagement lifecycle is sequential: saas-decommission before opportunity-discovery before jtbd-discovery, and so on. Skipping phases degrades output. The plugin needs to enforce this order without building a runtime that would couple skills together or require a custom loader.

Decision

Enforce sequence in three places, all prose-level:

  1. Every SKILL.md frontmatter description names preconditions (“ALWAYS use after X has run”).
  2. Every SKILL.md body includes a When to use it section that lists prerequisite skills.
  3. Every SKILL.md ends with a Hand-off section that names the next skill and prompts the operator to fire it.

No runtime gate. Skills remain independently loadable; the sequence is maintained by Claude reading the cues and by operator discipline.

Alternatives Considered

  • Runtime orchestrator. A wrapper that refuses to fire a skill until its prerequisite has produced a recognised output. Adds a runtime component that must be maintained and tested. Violates the skill-pack-only decision from ADR-0001. Rejected.
  • Skill-in-skill composition (skill A calls skill B). Couples skills tightly; makes each skill harder to test in isolation; breaks the “skill is self-contained” mental model. Rejected.
  • No enforcement. Rely on Alex’s memory. Works for one operator; breaks the moment methodology is delegated to sub-agents or another human. Rejected.

Consequences

Easier: - Skills remain plain markdown; no runtime surface. - Operators can deviate when context demands (explicit override is fine; silent skip is the risk). - Claude’s own description match reinforces order when pushy descriptions cite preconditions.

Harder: - No hard gate means operators can skip phases. The cost is output quality; the mitigation is the retrospective, which catches skipped steps after the fact. - Critic pass must verify the hand-off chain (no orphan skills, no cycles) on every new skill PR. - First-time users need to read CLAUDE.md to understand the lifecycle; sequence is not discoverable purely from skill names.

ADR-0004 Authorship Split for Remaining 12 Skills

Source: /Users/alex/dev/projects/principal-plugin/docs/adr/0004-authorship-split-for-remaining-12-skills.md

ADR-0004: Authorship split for the remaining 12 skills

  • Status: accepted
  • Date: 2026-04-24
  • Deciders: Principal Architect (with Alex sign-off)

Context

MVP-5 shipped with five skills (problem-decomposition, agentic-workflow-solutioning, threat-model, saas-decommission, engagement-retrospective). ROADMAP Phases 2-5 add twelve more to complete the lifecycle. Each new skill is one unit of authored work. Two distinct production modes are available:

  • Librarian authors markdown-heavy SKILL.md files: frontmatter description, when-to-use, workflow, evals, reference output. Runs skill-optimizer loop post-authoring. No code outside the SKILL.md.
  • Architect + Engineer is required when the skill bundles executable logic that the SKILL.md alone cannot express: retrieval, scaffolding, tool orchestration, code generation. Architect produces the SPEC; engineer implements to the spec on a feature branch; librarian polishes the SKILL.md wrapper at the end.

Routing every skill through architect+engineer is over-investment for framework-and-rubric skills. Routing every skill through librarian is under-investment for skills whose acceptance criteria require executable behaviour (e.g. “integration test retrieves ≥1 case with similarity score”, “scaffold that runs on a toy scenario”).

Decision

Split the twelve into ten librarian-authored skills and two architect-specced + engineer-implemented skills.

Librarian authors (10)

Skill Phase Why librarian
opportunity-discovery 2 Elicitation rubric; prompts + frontmatter
jtbd-discovery 2 JTBD framework application; prose + templates
data-readiness-5c 2 Five-C checklist; rubric with scoring prompts
systems-thinking 3 Causal-loop/stock-flow framework in prose
impact-effort-matrix 3 2x2 scoring rubric
blast-radius 4 Governance rubric; no external tooling
compliance-scan 4 Framework walk (SHIELD, SOC 2, HIPAA, PCI, GDPR) in prose; v1 is checklist, not automated scanner
adoption-risk 4 Risk rubric; Likert-style scoring
roi-calculator 5 Formula template in markdown; v2 may bolt on a spreadsheet, out of scope for v1
ai-usage-policy 5 Policy template covering access, data handling, kill-switch, incident response

Architect SPEC + Engineer implementation (2)

Skill Phase Why architect+engineer
pattern-recognition 3 Acceptance requires case-base retrieval with similarity reasoning. Needs retrieval strategy decision (grep-over-taxonomy vs embedding vs hybrid), integration with pna-case-base filesystem layout, and an integration test that actually queries and returns a case ID with score. SKILL.md alone cannot satisfy this.
live-agent-prototype 5 Acceptance requires producing a Claude Code skeleton that actually runs. Skill must generate scaffolded files (SKILL.md, evals, example inputs) and validate the scaffold by executing a dry-run. Code generation + validation pipeline.

Alternatives considered

A. All twelve to librarian. Rejected. pattern-recognition and live-agent-prototype have executable acceptance criteria that a markdown-only authoring flow cannot verify. Would produce SKILL.md files that fail their own integration tests.

B. All twelve to architect+engineer. Rejected. Over-investment. Ten of the twelve are rubrics, frameworks, or policy templates. Librarian-authored skill files with skill-optimizer loops have a shorter feedback cycle and are the right tool for markdown artefacts. Using architect+engineer here would add weeks of spec-and-implement overhead for skills where the markdown is the product.

C. Split on phase boundaries (e.g. all Phase 2-3 to librarian, all Phase 4-5 to architect). Rejected. Phase is not a proxy for authoring mode. Phase 3 contains both pattern-recognition (code) and systems-thinking (prose). Phase 4 is pure rubrics, not code.

Consequences

Easier: - Parallelism. Librarian can author 10 skills independent of the two code-bearing tracks. Architect’s scope is smaller and sharper. - Right-sized verification. Librarian uses skill-optimizer + lint; engineer uses test-driven-development + integration tests. - Clear ownership when Clarke kicks back weak work.

Harder: - Two workstreams to track. Mitigation: mission queue per stream, hive_mind logging, weekly cross-check against ROADMAP exit criteria. - pattern-recognition blocks Phase 3 end-to-end dry-run (T3.4). Mitigation: librarian ships systems-thinking and impact-effort-matrix in parallel with architect’s pattern-recognition SPEC so the latter is on the critical path but not the only Phase 3 work. - live-agent-prototype blocks Phase 5 end-to-end dry-run. Mitigation: same pattern; librarian ships roi-calculator and ai-usage-policy while architect+engineer build the prototype generator.

Execution order (recommended, not binding)

  1. Immediately parallel-safe (no Phase 1 case-base dependency): blast-radius, compliance-scan, adoption-risk, ai-usage-policy, roi-calculator.
  2. After Phase 1 seed case-base exists (Librarian): opportunity-discovery, jtbd-discovery, data-readiness-5c, systems-thinking, impact-effort-matrix.
  3. Architect specs in parallel with (1): pattern-recognition SPEC, live-agent-prototype SPEC.
  4. Engineer implements after each SPEC lands: pattern-recognition, live-agent-prototype.

Phase-gating in ROADMAP still holds; this is the intra-phase work ordering.

Hand-off protocol

  • Librarian work: one mission per skill via mission-cli.js --agent principal_librarian. Brief includes target path, acceptance criteria from TASKS.md, and the P&A skill manifest block.
  • Architect work: one mission per SPEC (two SPECs total), output is ~/dev/projects/principal-plugin/skills/<slug>/SPEC.md before any engineer work starts.
  • Engineer work: one mission per implementation, gated on the SPEC being merged. Branch engineer/<skill-slug>, PR into main.
  • Clarke verifies each completion against the relevant skill’s acceptance criteria in TASKS.md before closing the loop.
