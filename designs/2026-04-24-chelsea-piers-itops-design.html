1Overview
What is this

chelsea-piers-itops is a Claude Cowork plugin for the IT Operations team at Chelsea Piers, a sports and entertainment complex in New York City with approximately 300 staff. The plugin extends Claude Code and Claude Cowork with 36 structured IT operations skills, covering five role personas. The plugin contains no executable code — every component is a markdown file or JSON configuration. Institutional knowledge, CP-specific procedures, escalation paths, and connector-conditional logic travel with every Claude session rather than living in a person's head or a SharePoint page nobody reads.

The MCP server layer — outside the plugin directory — provides tool access to vendor systems without official hosted MCP endpoints: Cisco Meraki, NinjaOne RMM, Mimecast email security, and Cerberus (a 3-model risk deliberation system). Official hosted MCP servers handle Microsoft 365 and Atlassian.

Problems solved
Context switchingL1 technicians switch between Jira, Confluence, Intune, and NinjaOne for every single ticket.
No consistent interfaceSysAdmins have no structured workflow for access review, license assignment, and guest provisioning.
Unstructured investigationsSecurity analysts investigate phishing without a structured workflow or consistent output format.
Knowledge in headsEscalation paths, venue network IDs, Jira queues, and runbook locations live in people's heads, not systems.
Manual reportingThe IT Manager assembles the weekly status report manually from Jira exports every Monday morning.
Slow lifecycle opsUser onboarding and offboarding require sequential manual steps across 4+ systems with no consistent checklist.

4MCP Server Layer
Architecture notes

Four custom Python FastMCP servers handle vendor systems with no official hosted MCP endpoint. All support MOCK_MODE=true — tool calls return pre-built fixtures matching live API shapes without credentials. The default .mcp.json ships with mock mode active. Transport is stdio locally and HTTPS/HTTP in cloud deployments. Switching is a single .mcp.json swap; no skill file changes required.

NinjaOne 6 tools · OAuth 2.0 client credentials
  • ninjaone_list_devicesList all managed endpoints with health fields
  • ninjaone_get_deviceFull device record by ID
  • ninjaone_search_devicesSearch by hostname, serial, or user
  • ninjaone_list_alertsOpen alerts with severity and affected device
  • ninjaone_get_patch_statusPatch compliance for device or org
  • ninjaone_get_device_softwareInstalled software inventory
Mimecast 5 tools · API key + app key
  • mimecast_trace_messageTrace message through pipeline by sender/recipient
  • mimecast_list_held_messagesMessages currently in quarantine queue
  • mimecast_get_ttp_url_logsURL click logs from Targeted Threat Protection
  • mimecast_get_attachment_logsAttachment sandbox scan results
  • mimecast_check_senderSender reputation, block status, policy match
Meraki 7 tools · X-Cisco-Meraki-API-Key

  • meraki_find_clientWi-Fi client by MAC, hostname, or IP

5Cerberus Council
3-model parallel deliberation

Cerberus is a structured risk assessment gate for IT operations decisions where a wrong call has meaningful security, compliance, or operational consequences. Three models deliberate in parallel via OpenRouter — each with a fixed perspective. A synthesis pass produces the final verdict with confidence score and any dissenting positions.

claude-3-5-sonnet
Critical Analysis
Structured reasoning, policy alignment, precise language. Checks proposed action against documented CP policy.
openai/gpt-4o
Divergent Thinking
Lateral alternatives, edge cases, second-order effects. Surfaces what was not asked.
x-ai/grok-3
Risk Assessment
Threat modeling, failure modes, adversarial scenarios. Identifies worst-case paths.
Verdicts
APPROVE REJECT ESCALATE DEFER

Each verdict includes: confidence score (0–1), rationale, dissenting positions by model, conditions to proceed, and wiki citations from the Cerberus knowledge base.

Invocation triggers
  • change-requestMedium or High risk classification
  • quarantine-reviewAlways — before any release decision
  • guest-lifecycleWhen granting external access
  • conditional-access-auditWhen recommending a policy change
  • ninjaone-remediateAlways — before running endpoint script
  • email-threat-investigateHIGH confidence threat disposition

8Security Model
Write Gates
change-request required
All Medium/High risk writes. Skill checks that change-request was run in-session before proceeding.
Explicit CONFIRM prompt
Any destructive action. Claude does not proceed unless the user types CONFIRM — never assumed.
Cerberus council
High-stakes policy decisions. cerberus_council verdict must be APPROVE or CONDITIONAL before execution.
Human escalation
ESCALATE verdicts stop Claude entirely. Analyst is instructed to escalate to named human contact.
Data classification rules
  • ✕ Email body — never passed to Claude. Metadata only (sender, recipient, subject, timestamp, verdict).
  • ✕ Passwords / MFA codes — never captured or displayed. Skills provide guided reset steps only.
  • ✕ BitLocker recovery keys — never surfaced. Skills confirm key existence and direct to Entra escrow.
  • ✕ Network device credentials — never in skill output. API key auth at server environment level only.
  • ✕ Credentials in repo — zero. URLs and env var names only in .mcp.json. No secrets in any committed file.
