pna-case-base

The Principal & Agent compounding memory. Engagement records and extracted patterns.

Status

active — schema scaffolded, awaiting seed records

What it is

A private git repo of P&A engagement records and the patterns extracted from them. Every completed engagement writes a case record via the engagement-retrospective skill; the pattern-recognition skill queries this repo to find analog cases for new engagements. Over time, this repo becomes the irreplaceable asset: the methodology plugin (principal-plugin) is reproducible; the case base is not.

What it is not

  • Not a database. Flat markdown, grep-retrievable, git-versioned.
  • Not a public repo. Private, always.
  • Not a dumping ground for raw client data. Anonymisation is mandatory. See docs/ANONYMIZATION.md.
  • Not multi-tenant. One operator, one case base.
  • Not a build target. No CI, no tooling, no indexer in v1.

How writes happen

Only via engagement-retrospective. Never by hand (use the skill even for seed records; this keeps frontmatter consistent).

How reads happen

Via pattern-recognition (when shipped) or ad-hoc grep / obsidian-cli. Markdown + frontmatter is enough for v1 retrieval.

Pattern maturity

  • emerging — 1 citing case. Not yet proven.
  • confirmed — 2+ citing cases. Safe to lean on.
  • retired — deprecated by a later pattern or disproven.

SPEC

pna-case-base — Spec

Problem

The Principal plugin (principal-plugin) ships two skills that require a persistent, queryable store of engagement history: engagement-retrospective writes case records at engagement close, and pattern-recognition (not yet shipped) reads from the store to find analogs for new engagements. Without this store, every engagement starts from a blank page and the methodology does not compound. The data is sensitive (client-identifying), grows slowly (engagement-cadence, not event-cadence), and benefits from human-readable diffs.

Proposed Solution

A private git repo at github.com/afoxnyc3/pna-case-base containing flat markdown files with YAML frontmatter. Five directory buckets under patterns/ (triggers, architectures, anti-patterns, playbooks, plus cases/ for the source-of-truth engagement records). A TAXONOMY.md enumerates the categorical dimensions. An ANONYMIZATION.md defines the mandatory pre-commit policy. Templates under templates/ give engagement-retrospective a stable shape to write against. No database, no indexer, no CI in v1; grep and frontmatter are sufficient retrieval primitives.

Non-Goals

  • Not a database. No SQLite, no Postgres, no vector store in v1.
  • Not an indexer. No embeddings, no BM25, no custom retrieval in v1.
  • Not a public repo. Always private.
  • Not multi-tenant. Single operator.
  • Not a CI-gated repo. No automated tests in v1; anonymisation is enforced at authoring time, not commit time.
  • Not a general-purpose note store. Only P&A engagement records and derived patterns live here.
  • Not the plugin’s install target. Consumers clone this repo alongside the plugin; it is not bundled.
  • Not a source of public case studies. Anonymised case studies are published separately via pna-sessions.

Acceptance Criteria

Tech Stack

Layer Choice Rationale
Storage Flat markdown + YAML frontmatter Human-readable, git-diffable, zero runtime dependency. Matches Obsidian conventions Alex already uses. Any future indexer can be added without a migration.
Hosting Private GitHub repo Versioned, backed up, accessible by CLI tools and sub-agents that already authenticate to GitHub. No separate auth to manage.
Retrieval grep + frontmatter scan Case counts will be in the tens for years. An indexer is premature optimisation. grep hits every file in <200ms at expected scale.
Templates Static markdown files under templates/ engagement-retrospective copies from here to cases/ or patterns/. No template engine needed.
Anonymisation Pre-commit checklist in ANONYMIZATION.md enforced by the skill’s procedure Automated PII scrubbing has false negatives. Making anonymisation a human gate during the retrospective interview is more reliable than regex.
Versioning Git tags per quarter for case-base snapshots Cheap point-in-time reference for audits or engagement disputes.
Access control GitHub repo permissions (owner only in v1) Matches single-operator assumption. Additional collaborators added per engagement basis later.

Architecture

See ARCHITECTURE.md. Data model matches the ER diagram in principal-plugin/ARCHITECTURE.md. Cross-repo contract is path-based: principal-plugin skills reference cases/, patterns/triggers/, etc., via relative path assumptions (operator clones both repos side-by-side under ~/dev/projects/).

Risks

  1. Client data leakage via commit — client name, contact emails, or uniquely identifying metrics pushed to git history. Likelihood: M, Impact: H — Mitigation: ANONYMIZATION.md policy, mandatory pre-commit checklist in retrospective skill, private repo. Residual: git history is forever; a leaked commit requires full repo rotation.
  2. Anonymisation false negatives — operator believes a record is anonymised when it is not. Likelihood: M, Impact: H — Mitigation: ANONYMIZATION.md has explicit prohibited-data list; second-pass review before commit is part of retrospective skill procedure. Accepted residual risk: operator discipline.
  3. Schema drift between plugin and case base — plugin evolves frontmatter fields, case base doesn’t keep up. Likelihood: M, Impact: M — Mitigation: templates/ directory is the source of truth; engagement-retrospective regenerates templates if drift detected; ADR required for any schema change.
  4. Grep retrieval becomes slow — thousands of cases someday. Likelihood: L (years away), Impact: M — Accepted. Add an indexer when counts exceed ~500 cases.
  5. Pattern over-promotion — promoting emerging to confirmed too eagerly from 1.5 cases. Likelihood: M, Impact: M — Mitigation: promotion requires two distinct case citations, enforced in retrospective skill procedure.
  6. Repo loss — laptop dies, GitHub outage during a retrospective. Likelihood: L, Impact: H — Mitigation: GitHub is the durable copy; local clone is cache. Weekly git push is part of the engagement close ritual.
  7. Scope creep — case base becomes a dumping ground for notes, references, marketing drafts. Likelihood: M, Impact: L — Mitigation: Non-Goals list is explicit; README refuses any content that is not an engagement artefact or derived pattern.

Open Questions

None. Defaults applied: - Private repo (always). - Single operator in v1. - No indexer in v1. - Anonymisation enforced at authoring time. - Repo lives under ~/dev/projects/ alongside plugin. - Quarterly tags rather than per-case tags.

Retrieval Flow (v1: grep-based)

sequenceDiagram
    participant S as pattern-recognition skill
    participant FS as Filesystem (pna-case-base)
    participant R as Ranker (skill-local)

    S->>FS: grep frontmatter for archetype / industry / signal
    FS-->>S: candidate file paths
    S->>FS: read full case files
    FS-->>S: case content
    S->>R: score similarity vs query (LLM judge)
    R-->>S: ranked matches
    S-->>S: return top-3 with citations

No persistent index. Case count is expected to stay under 100 for v1; grep latency is not a concern at that scale.

Directory Responsibilities

  • cases/ — the source of truth. Each file is one engagement. Never edit post-commit except for amendments (record in the file itself).
  • patterns/triggers/ — observable signals that indicate a problem archetype. Short (≤200 words).
  • patterns/architectures/ — solution shapes that worked. Includes required tech, guardrails, known gotchas.
  • patterns/anti-patterns/ — approaches that looked right and failed. Single-case fine; duplication strengthens.
  • patterns/playbooks/ — multi-step sequences that worked across ≥2 cases. Higher bar than single patterns.
  • templates/ — starting points copied by engagement-retrospective. Treat as the schema source of truth.
  • docs/ — policies and ADRs. Not consumed by skills.

Cross-Repo Contract with principal-plugin

  • principal-plugin skills assume the case base lives at ../pna-case-base/ relative to the plugin clone, or at $PNA_CASE_BASE if set.
  • Skills read templates from templates/ and write to cases/ or patterns/*/.
  • Any schema change requires an ADR in this repo and a corresponding update to the retrospective skill.

What this repo explicitly is not

  • Not a CI target. No GitHub Actions in v1.
  • Not an indexer host. No sqlite-fts, no embeddings, no vector store.
  • Not a multi-repo federation. One repo; one operator.
  • Not a marketing surface. Public case studies live in pna-sessions, not here.

TAXONOMY

Source: /Users/alex/dev/projects/pna-case-base/TAXONOMY.md

Taxonomy

Controlled enumerations used in case and pattern frontmatter. Adding values is allowed; removing or renaming requires an ADR.

trigger_signals

Observable phrases or situations that indicate a particular problem archetype is in play.

  • budget-objection — “we don’t have budget for AI right now”
  • tool-sprawl — “we have too many tools” / “our software costs are out of control”
  • compliance-wedge — client raises HIPAA, PCI, SOC 2, SHIELD, or GDPR in discovery
  • manual-routing — humans doing inbox triage or ticket routing as a full-time activity
  • expert-bottleneck — one person holds domain knowledge everyone else waits on
  • after-hours-coverage — phone, chat, or email goes unanswered outside business hours
  • data-lake-orphan — client paid for a data warehouse or lake that is unused
  • failed-ai-pilot — client tried AI before with ChatGPT or a vendor and it did not stick
  • shadow-it — individual team members subscribe to tools on personal cards

problem_archetypes

The shape of the underlying problem, independent of solution.

  • knowledge-retrieval — the answer exists; finding it is the work
  • expert-bottleneck — one person gates decisions
  • manual-classification — sorting, tagging, or routing at volume
  • content-drafting — first-draft generation of emails, proposals, reports
  • repetitive-copy-paste — moving data between systems that do not integrate
  • after-hours-coverage — service must run when humans are not available
  • compliance-reporting — periodic evidence-gathering for audits
  • quality-review — checking human work against a rubric

solution_archetypes

The shape of what we built.

  • skill-pack — Claude Code or Claude.ai skills
  • single-call — one-shot API call, stateless
  • workflow — deterministic chain of LLM calls
  • tool-use-workflow — workflow with tool calls, code controls the loop
  • open-agent — model decides trajectory
  • multi-agent — multiple specialised agents with handoff protocol
  • voice-agent — phone / voice channel
  • email-agent — inbox-triggered
  • chat-bot — Slack / Teams channel
  • scheduled-job — cron-triggered periodic process
  • local-inference — MLX or equivalent on operator hardware, privacy-bound

industry_tags

Client industry. Use broad buckets; verticalise only when a third case in the same vertical arrives.

  • msp — managed service provider
  • financial-services
  • health
  • legal
  • professional-services
  • ecommerce
  • saas
  • manufacturing
  • education
  • nonprofit
  • other

data_sensitivity_levels

Classification carried on any data touched by the solution.

  • public — already in the public domain
  • internal — company-internal, not customer-identifying
  • confidential — business-sensitive, competitive
  • restricted — PII, PHI, PCI, or regulated financial data

engagement_outcomes

Single word status at close of engagement.

  • shipped-adopted — delivered and client is actively using it
  • shipped-dormant — delivered but not in daily use
  • killed-pre-build — scoped and killed before code was written
  • killed-mid-build — started and stopped before delivery
  • client-churned — client ended relationship during or shortly after engagement
  • handed-off — delivered and transferred to client’s internal team

pattern_maturity

Applies to pattern records, not cases.

  • emerging — 1 citing case
  • confirmed — ≥2 citing cases from distinct engagements
  • retired — superseded or disproven; keep for history, do not cite

Anonymisation Policy

Non-negotiable. Every file committed to this repo passes this checklist. A single leaked commit requires repo rotation; git history is forever.

Redaction rules

Before committing any case file, replace the following with pseudonymous or categorical substitutes:

Raw Replace with
Client legal name Pseudonym matching client_anon in frontmatter (e.g. “NorthMSP-01”)
Employee names Role only (e.g. “CFO”, “Lead Engineer”, “Ops Manager”)
Email addresses Redacted or categorical (“info@client” → “contact-email”)
Phone numbers Redacted
Specific dollar amounts ≥ $50k Round to nearest $10k and label as approximate
Customer lists, counts < 50 Redact; use industry bracket instead
Proprietary product names owned by the client Generic category name
Physical addresses Redact; region only if relevant
Internal project codenames Redact
Contract numbers, invoice numbers Redact

Dollar amounts between $10k and $50k stay as rounded figures; under $10k is fine verbatim.

Prohibited data (never commit, even redacted)

  • Authentication credentials (API keys, passwords, tokens) — treat as incident if encountered.
  • PHI (health) or PCI (cardholder) data of any form.
  • SSNs, EINs, tax IDs.
  • Security findings that name an unpatched exploitable vulnerability on a client system.
  • Anything the client explicitly marked confidential in writing.

If a record’s content cannot be written without including prohibited data, do not write the record. Capture the lesson as a pattern fragment instead, with no case attribution.

Pre-commit checklist

Ran as part of engagement-retrospective before file commit:

Breach protocol

If a commit is made that includes prohibited or insufficiently anonymised data:

  1. Stop work. Do not push if the push has not yet happened.
  2. If already pushed: git log the exposed commit, note the SHA, and escalate to Alex immediately. Do not attempt a force-push rewrite alone.
  3. Full repo rotation is the default remediation: create a fresh private repo, copy sanitised content, delete the old repo via GitHub UI (not just archive).
  4. Record the incident in docs/incidents/YYYY-MM-DD-<slug>.md with root cause, scope, and process change.

Why anonymisation happens at authoring time, not at commit time

Automated PII regex scans have false negatives (unusual names, non-US phone formats, contextual identifiers like “their biggest client is X”). A human pass during retrospective catches contextual identifiers regex cannot. ADR-0002 records this trade-off. The checklist above is the compensating control.

ADR-0001 Flat Markdown over Database

Source: /Users/alex/dev/projects/pna-case-base/docs/adr/0001-flat-markdown-over-database.md

ADR-0001: Flat markdown over a database

  • Status: accepted
  • Date: 2026-04-21
  • Deciders: Principal Architect

Context

The case base stores engagement records and extracted patterns. Data volume is engagement-cadence (tens of records per year), not event-cadence. Consumers are Claude skills and ad-hoc operator grep. Human-readable diffs matter; the operator regularly reviews cases by eye.

Decision

Flat markdown files with YAML frontmatter, hosted in a private git repo. No database, no indexer, no build step in v1. Retrieval uses grep and frontmatter scan.

Alternatives Considered

  • SQLite + FTS5. Would give fast full-text retrieval and structured queries. Adds a runtime dependency, a schema migration story, and loses git-diffability. Premature optimisation at the expected scale (<100 records for v1). Rejected.
  • Postgres + pgvector. Full-featured store with embedding retrieval. Adds hosting cost, auth surface, and a backup story. Massively over-scoped for a single-operator case base. Rejected.
  • Obsidian vault folder. Operator already uses Obsidian. Would give good local ergonomics but couples the store to one machine; no native git versioning; harder to cite from sub-agents. Rejected.
  • Notion / Airtable. Vendor lock-in, no offline access, poor fit for skill consumption. Rejected.

Consequences

Easier: - Zero infra. git clone is the install. - Diffs are meaningful; reviewing changes is a normal git workflow. - Sub-agents (architect, librarian, engineer) can read by path without any new integration. - Anonymisation review happens in a plain editor, not a DB UI.

Harder: - No structured queries. SELECT * WHERE industry = 'msp' becomes a grep pattern. - No foreign-key enforcement between cases and patterns; citations are prose-validated. - Retrieval cost scales linearly; trigger in ROADMAP Phase 5 flips to an indexer if needed.

ADR-0002 Anonymisation at Authoring Not Commit

Source: /Users/alex/dev/projects/pna-case-base/docs/adr/0002-anonymisation-at-authoring-not-commit.md

ADR-0002: Anonymisation enforced at authoring time, not commit time

  • Status: accepted
  • Date: 2026-04-21
  • Deciders: Principal Architect

Context

Case records contain client-identifying information that must be removed before commit. The options for enforcement are: (a) automated pre-commit hook with regex-based PII detection, (b) CI job that fails the build on unexpected patterns, (c) a human checklist enforced by the skill that authors the file.

Decision

Enforce anonymisation at authoring time via a mandatory checklist in docs/ANONYMIZATION.md. The engagement-retrospective skill runs this checklist before emitting the final commit. No automated regex gate; no CI.

Alternatives Considered

  • Pre-commit regex hook. Catches email addresses, phone numbers, common PII. Fast, cheap. Fails on non-US phone formats, unusual name spellings, and contextual identifiers (“their biggest competitor is X”). False negatives would create a misleading sense of safety. Rejected as a primary control; can be added later as a belt-and-braces layer.
  • CI PII scanner. Higher quality than simple regex (e.g. Presidio, gitleaks). Still has false negatives on contextual identifiers. Adds a required-checks gate on a single-operator repo, which is friction without much benefit. Rejected for v1.
  • Post-commit periodic scan. Reviewer sweeps the repo monthly looking for leaks. Too late; git history is already permanent. Rejected.

Consequences

Easier: - Anonymisation happens while the record is still being edited; fixes are cheap. - Contextual identifiers (client sentence patterns, named competitors) get caught by human judgement, which regex misses. - Skill owns the policy; updating the policy updates the skill and the checklist in one place.

Harder: - Single point of failure is operator discipline. If the checklist is skipped, there is no second line of defence. - No automated record of which files passed the check. Mitigation: anonymisation_checked: true flag in case frontmatter; skill refuses to write the file until the flag flips. - Reversible leaks require repo rotation, not history rewrite. Cost is high but predictable.

Follow-up trigger

If the case base grows past 50 records or a second operator is added, revisit this decision. Automated gates become higher-value at that point.
