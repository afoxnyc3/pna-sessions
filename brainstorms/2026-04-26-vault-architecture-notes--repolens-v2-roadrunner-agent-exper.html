Repolens v2

AI-native repository intelligence tool. Ingests a local git repo, classifies and scores every file, builds token-efficient context bundles, and runs AI tasks against that context. Replaces pasting entire codebases into chat windows.

Stack

Python 3.12 + SQLite + Anthropic SDK + Typer CLI. Local-first. No cloud dependencies for MVP.

Architecture

repolens/
├── ingestion/      scan repo, apply gitignore + custom filters
├── classification/ rules-based file type + importance scoring
├── summarization/  AI summaries at file, directory, repo scope (cached by content hash)
├── context/        token-budgeted context bundle assembler
├── ai/             Anthropic client, prompt templates, task executor
├── db/             SQLite schema + data access layer
└── cli/            Typer CLI entrypoint

Pipeline: ingest → classify → summarize → context → run

Key decisions

Decision Choice ADR
Summarization strategy Per-file (not batch) [[ADR-001]]
Default token budget 32K (override: --budget or REPOLENS_TOKEN_BUDGET) [[ADR-002]]
FastAPI layer Deferred — CLI first [[ADR-003]]

Repolens v2 — Build Log & Learnings

Per-task notes from the mission-driven build. Each entry records what was built, what broke, and what to carry forward.

T01 — Project scaffold

Status: ✅ Done Issue: Engineer returned “no output” — work was partially done but T01 never marked done, venv not set up, nothing committed. Root cause: Mission-cli result capture timed out or agent context ran out before returning. Fixes applied: - setuptools.backends.legacy:build → setuptools.build_meta (invalid backend) - Added tests/test_scaffold.py (pytest exited 5 with no tests — needed at least one) - Updated T01 validation commands to use .venv/bin/ prefixes - Added venv activation instructions to CLAUDE.md

Learning: Always add at least one real test in the scaffold so pytest exits 0, not 5. Venv prefix discipline needs to be set from task 1, not retrofitted.

T02 — SQLite schema + migration runner

Status: ✅ Done Quality: 25 tests. WAL mode + FK enforcement enabled at init. Idempotent. schema_version seeded. Notable: Engineer added _ensure_row_factory as a shared helper — good pattern, picked up by later tasks. Learning: Good practice to enable WAL + FKs at init time, not leave it to callers.

T03 — Data access layer

Status: ✅ Done Quality: 38 tests, 12 functions, zero raw SQL outside the module. Notable: update_run auto-sets completed_at on terminal status — nice defensive default. file_paths JSON-encoded in DB, decoded on read. Upsert is PATCH semantics (existing columns not in kwargs untouched). Notes for future tasks: FK pragma is per-connection — PRAGMA foreign_keys=ON needed after each sqlite3.connect() if cascade behaviour required.

T04 — File filters

Status: ✅ Done Quality: 20 tests. Clean four-function interface. Notable: is_binary defaults to True on unreadable files — good safe default. .git/ hardcoded as always-ignored regardless of .gitignore. Learning: Safe defaults on error > silent failures. Never trust the caller to handle missing files gracefully.

T05 — Directory scanner

Status: ✅ Done Quality: 8 tests. Scans this repo at 55 files. Notable: Used os.scandir not os.walk — correct for efficiency. sha256 per file. Learning: os.scandir is meaningfully faster than os.walk on large trees. Worth the extra code.

T06 — CLI ingest command

Status: ✅ Done Notable: Running ingest twice on same path updates rather than duplicates (get_repo check + upsert handles idempotency). DB at ~/.repolens/repolens.db auto-created on first use. Learning: CLI commands that touch the DB should always be idempotent. User will run them more than once.

T07 — File classifier + scorer

Status: ✅ Done Notable: other category captured 27/60 files on first classify run — .pyc, egg-info, lock files. Score ties at 1.0 for shallow __init__.py files (entry-point boost + depth=0). Learning: other being high is expected on a fresh Python project with build artifacts. Consider adding generated classification for .pyc/__pycache__ in the scanner or filter layer rather than letting them through to classification.

T08 — CLI classify command

Status: ✅ Done Output on this repo (60 files): core=18, other=27, test=7, docs=4, config=4. Learning: Score differentiation improves with depth — shallow files all score 1.0 due to entry-point boost. The high other count is an upstream filter gap, not a classifier bug.

T09 — AI client wrapper

Status: ✅ Done Quality: 16 tests, all mocked (no real API calls in tests). Notable: ValueError raised before SDK touches missing key — correct. Returns (str, int, int) tuple consistently. Learning: Always validate API keys before instantiating the client. Fail fast with a clear message, not an SDK stack trace.

T10 — Prompt templates

Status: ✅ Done (engineer returned no output — second occurrence) Quality: Code landed correctly. 4 pure string functions, no I/O. Learning (meta): “No output” pattern now seen twice. Likely agent context runs out after writing code but before writing the result string. Not blocking but worth investigating the mission-cli result buffer.

T11 — File summarizer with caching

Status: ✅ Done Quality: Cache hit/miss verified. Content capped at 200k chars (~50K tokens) before API call. Notable: Cache key is (repo_id, scope='file', relative_path) + content_hash match. Hash mismatch triggers re-summarisation automatically. Learning: Cache-by-hash is the right pattern for AI summaries. Never re-summarise unchanged files.

T12 — Directory + repo summarizer

Status: ✅ Done Quality: 28 new tests. Multi-repo isolation verified (repo_id is part of cache key). Notable: _REPO_TARGET_PATH = '' exported as constant — keeps the magic string in one place. Learning: Constants for magic DB values (like empty-string repo target) should be module-level exports, not inline literals scattered across callers.

T14 — Token counter

Status: ✅ Done Quality: 12 tests. tiktoken working. Fallback verified. Pricing table correct. Pricing encoded: opus-4-5 ($15/$75), sonnet-4-5 ($3/$15), haiku-3-5 ($0.80/$4). Unknown model → sonnet pricing. Learning: Always encode a fallback for token counting. tiktoken can fail on unusual Python versions.

T15 — Context packager

Status: ✅ Done Quality: 22 tests. Budget enforcement at 1.05x tolerance confirmed. Notable: list_summaries_by_scope() added to repository.py to keep SQL inside db/ — correct layering. Edge cases: file too large → summary fallback, file missing from disk → summary fallback, file with no summary and no disk → silently skipped. Learning: The “silently skip” vs “error” decision for missing files matters in batch operations. Skipping is right here — a context bundle with some gaps is better than a hard failure.

T17 — Task executor + run logging

Status: ✅ Done (engineer returned no output — third occurrence) Issue: Engineer imported get_run from repository.py but function didn’t exist — broke test collection for entire test suite. Fix: Added get_run(conn, run_id) -> dict | None to repository.py. Natural function, should have been in T03 scope. Quality: Run lifecycle (running → done/failed) verified. completed_at auto-set on terminal status (from T03’s update_run design). Learning: When an engineer imports a function that doesn’t exist, it’s more likely a spec gap than engineer error. get_run is obviously needed alongside create_run and update_run. Specs should include all CRUD pairs.

T13 — CLI summarize command

Status: ✅ Done Quality: 55 tests passing. --scope file|dir|repo|all, --force, token tracking, cost report all implemented. Design: Progress printed per-file during run. Token + cost summary at end. --force passes through to summarizer cache bypass. Learning: Good example of a CLI command that’s essentially a loop over existing primitives — the summarizers did the heavy lifting, T13 just wired them together with a clean interface.

T16 — Context exporter + CLI context command

Status: ✅ Done Quality: New tests added to test_context.py. export_markdown and export_json as pure functions (no DB/API side effects). Design: Token count + file list go to stderr; formatted bundle content goes to stdout (or --output file). Clean separation lets the output be piped. Learning: Stderr for diagnostics, stdout for content. Simple rule that makes CLI tools composable.

T19 — CLI status, list, runs commands

Status: ✅ Done Quality: All three commands implemented and validated. Graceful empty-table handling. Design: list shows repo table, status shows per-repo breakdown (file count, category breakdown, summary coverage %, last scan), runs shows recent run history with tokens + cost. Learning: These three commands are the “health check” surface for the tool — getting them right early is worth it. Users need fast answers to “what have I indexed?” and “what did that last run cost?”

Meta: “No output” pattern

Seen on T01, T10, T17. Engineer completes the work (code lands, validated) but mission result string is empty. Hypothesis: agent context window exhausted after implementation, before writing result. Not blocking — work always lands correctly. But wastes a review cycle. Mitigation in place: “Report back with your result” added to all prompts. Seems to reduce but not eliminate. TODO: Investigate mission-cli result capture timeout. May need to increase buffer or add a checkpoint mid-task for large implementations.

roadrunner-cli

Deterministic agentic loop for Claude Code. Python owns control. Claude owns execution. Hooks enforce completion.

What it does

Runs Claude Code through a task queue (tasks/tasks.yaml) where: - Every task must pass its validation_commands before it can be marked done - The Stop hook blocks Claude from exiting when tasks remain - The PreCompact hook snapshots state before context window resets - The SessionStart hook injects roadmap state after compaction restarts

Architecture

tasks/tasks.yaml          ← task queue (todo → in_progress → done)
roadrunner.py             ← controller: start, validate, complete, block
.claude/settings.json     ← wires hooks to Claude Code events
hooks/stop_hook.sh        ← blocks exit when tasks remain
hooks/session_start_hook.sh  ← injects snapshot context on session start
hooks/precompact_hook.sh  ← snapshots state before compaction
hooks/post_write_hook.sh  ← surfaces ruff/yaml lint feedback

10 ADRs documenting real bugs found and fixed: line-anchored completion signal, in-progress awareness, retry storm prevention, atomic writes, absolute hook paths, structured trace logging, dead hook cleanup, validation timeout, state schema versioning, hook→Python entry point unification.

Status

102 tests. Active roadmap: ROAD-001 to ROAD-009.

Bootstrap (6/6 done). Now developing the project itself with its own tasks.yaml. Loop battle-tested via repolens-v2 build — drove 18+ tasks across the full pipeline.

Agent Experts Pattern — Lessons Learned

Headline

Agents forget between sessions. The fix is a persistent YAML mental model per domain, read before acting, validated against code after acting, re-synced on a schedule. Code stays the source of truth. The YAML is a cache of how to think about the code. That’s the whole pattern.

Everything else in disler/agent-experts is scaffolding around this idea.

Why this matters for P&A

  • research_a/b A/B test: each research agent can accumulate domain expertise files over sessions. Better expertise → better outputs. Measurable delta between frameworks.
  • creator-stack: first real consumer. Codify “what good creator content looks like for Alex” as an expertise file so the stack doesn’t drift each invocation.
  • Clarke delegation: expertise files sit next to principals’ CLAUDE.md as the domain layer. Persona prompt says how you work; expertise file says what you know.

The 7 patterns (ranked by leverage)

1. Domain triad

Every domain gets three files:

experts/<domain>/
├── expertise.yaml       # the mental model (canonical, versioned)
├── question.md          # read-only Q&A against the YAML
└── self-improve.md      # rewrite YAML to match current code

Clean separation of storage, read path, write path. No other files. No drift risk between “docs” and “notes” and “reference”.

2. Strict line cap

MAX_LINES = 1000. Self-improve loops an iterative trim until under cap. Trim priority:

  1. Verbose descriptions → terse
  2. Redundant examples → one canonical each
  3. Low-priority edge cases → drop

The cap is what keeps the YAML a working reference instead of documentation sprawl. No cap, no memory; just docs.

3. Compile-check as validation gate

End of every self-improve run:

python3 -c "import yaml; yaml.safe_load(open('expertise.yaml'))"

Cheap correctness floor. Broken YAML never ships. Cost: milliseconds. Value: eliminates an entire class of silent failures.

4. Plan → Build → Improve orchestrator

plan_build_improve.md chains three Task-tool subagents:

  1. /experts:<domain>:plan [prompt] → path_to_plan
  2. /build [path_to_plan] → build report
  3. /experts:<domain>:self-improve true → drift report

Load-bearing rule: do not stop between steps. Parent holds the thread, children do clean work with fresh context.

5. Fresh-context subagent rule

Every subagent brief stands alone. No implicit context. Same discipline Clarke uses when delegating to principals. Confirms the universal law: delegation quality = brief quality.

6. Sub-agent contract has teeth

Frontmatter: - description states when to delegate (not just what) - tools list is minimal and comma-separated

Body: exactly 4 sections — Purpose / Instructions / Workflow / Report. No optional sections, no ceremony. Meta-agent enforces this shape; violations get regenerated.

7. Single-file build agent

build-agent.md implements one file at a time from a detailed spec. Parallelizable out of the box. Matches our atomic-task rule in TASKS.md. A spec with N atomic tasks = N parallel build-agents.

Supporting concepts (from the TAC-13 video, zTcDwqopvKE)

  • Three atoms of meta-agentics: meta-prompt (prompts writing prompts), meta-agent (agents building agents), meta-skill (skills creating skills).
  • Act → Learn → Reuse: every task improves the mental model for next time. Compounding per domain.
  • Code is source of truth, expertise is mental model: never let them compete. Expertise re-reads code, not the other way around.
  • Scale-out pattern: fire N parallel experts against the same question when you need confidence.

Open questions

  • Where do expertise files live? Per-agent .claude/skills/<agent>/experts/<domain>/, or shared at ~/.pna/experts/<domain>/ so research_a and research_b see the same mental model?
  • Do we git-track expertise.yaml? (Yes for code-linked domains, no for ephemeral/personal. Default: yes.)
  • Self-improve cadence — on-demand, post-commit hook, or nightly cron?
