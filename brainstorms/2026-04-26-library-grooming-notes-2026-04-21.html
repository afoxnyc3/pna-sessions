Pre-Mission Librarian Check-in — Pattern Spec

The Problem

Agents launch into missions without knowing what skills are available to them. They either re-invent workflows already encoded in SKILL.md files, or use inferior ad-hoc approaches when a purpose-built skill exists. The library is not consulted. Armor is not equipped.

The Vision

Before any agent starts a mission, it checks in with the library: 1. Receives a curated list of skills relevant to the task 2. Confirms which are installed and ready 3. Loads them into its active context before proceeding

Same principle as a soldier checking kit before deployment. You know what you’re carrying before the mission starts.

Implementation Options

Option A — Clarke injects skill manifest into every mission brief (lowest friction)

Clarke’s mission brief template includes a ## Available Skills section, auto-populated from library.yaml filtered by the agent’s domain and task type. No agent code change required. Works with current mission-cli.js dispatch.

Pros: No changes to agent CLAUDE.md files. Immediate. Cons: Clarke must maintain relevance mapping (which skills apply to which agents). Static — doesn’t adapt to task specifics.

Implementation: Amend Clarke’s CLAUDE.md briefing template. Add a per-principal skill manifest (pre-filtered list) injected into every mission prompt.

Option B — preflight skill at session start (most general)

A new preflight skill each agent runs at session start. The skill: 1. Reads the incoming task/mission description 2. Queries library.yaml for skills matching keywords in the task 3. Returns a ranked list: “For this mission, consider loading: [skill-a, skill-b, skill-c]” 4. Agent loads them before proceeding

Pros: Task-aware, dynamic, works for any agent. Cons: Requires every agent’s CLAUDE.md to include a session-start step. Adds one round-trip before work begins.

Implementation: Author preflight SKILL.md. Add “Run /preflight before starting any mission” to each principal’s CLAUDE.md.

Option C — sandbox orchestrator selects skill set from library.yaml (most powerful)

When Alex ships the sandbox orchestrator (dev sandbox / E2B), it reads library.yaml and selects a skill loadout for each mission before spawning the agent. The agent is initialized with the correct skills already in context.

Pros: Fully automated. Agent never needs to ask — it just has the right tools. Cons: Requires sandbox orchestrator to exist (future work). Most complex.

Implementation: Sandbox orchestrator reads library.yaml, scores skills against mission description, passes selected skill content in the agent’s initial context window.

Recommended Sequencing

  1. Now: Option A — Clarke injects a static per-principal skill manifest into mission briefs. Zero new code. Immediate improvement.
  2. Near-term: Option B — preflight skill authored, added to principal CLAUDE.mds as a session-start gate.
  3. When sandbox orchestrator ships: Option C — fully automated, orchestrator-driven skill selection.

Toolbox Concept

Each principal should have a named toolbox — a curated 8-12 skill list that is always available for their domain. The toolbox is the baseline; the pre-flight check supplements it with task-specific additions.

# example: ~/.claudeclaw/agents/principal_engineer/toolbox.yaml
