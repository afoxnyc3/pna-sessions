Data Flows

Ingestion Pipeline
1
IngestionWorker
Load cursor from ingestion_cursors by stream_type
2
IngestionWorker → Graph
Paginate /auditLogs/signIns with $skipToken — handle @odata.nextLink
Retry on 429 with Retry-After. Exponential backoff on 503.
3
IngestionWorker
Dedup by event_id + content_hash — skip known events
4
IngestionWorker → PostgreSQL
INSERT new events, UPDATE ingestion_cursors
Dead-letter on malformed payloads — log, don't crash.
5
TriageWorker
Fetch unprocessed events, run rule-based pre-filter
PASS → suppress  |  FLAG → direct incident  |  AMBIGUOUS → agent
6
Agent → TriageWorker
Incidents[] + confidence scores returned via tool use
Dedup against open incidents by hash(actor + category + target) in 24h window.
7
TriageWorker → PostgreSQL
INSERT incidents + audit_log entry per incident
Response Action Flow
1
Agent (ResponsePolicyAgent)
Calls evaluateAction(action) in packages/policy
2
Policy Engine
Returns decision against the action approval matrix
AutoExecutable  |  ApprovalRequired  |  Prohibited
3a
AutoExecutable path
Execute immediately → INSERT response_actions (completed)
E.g. open_case, attach_note, suppress_duplicate, mark_urgent
3b
ApprovalRequired path
INSERT response_actions (pending) → appears in Approval Center UI
E.g. disable_user, revoke_session, remove_role_assignment
4b
Analyst
Reviews diff + risk context → Approve / Reject
5b
System → Graph
Execute approved action → UPDATE status completed
3c
Prohibited path
Throw PolicyViolationError → INSERT audit_log (blocked)
E.g. bulk_disable_users, delete_application — blocked in v1.

Agent Definitions

SignInTriageAgent
sign-in
In: sign_in_events batch, user metadata
Out: incidents[] with confidence scores, MFA anomaly flags, recommended actions
IdentityRiskAgent
risk
In: risk_events (risky users, risky sign-ins)
Out: urgency classification, whether CA/MFA is sufficient or escalation needed
DirectoryChangeAgent
directory
In: directory_audit_events clustered by actor + time window
Out: incident clusters, blast radius estimate, MITRE tactic tags
BaselineReviewAgent
baseline
In: baseline_controls catalog + Graph-collected evidence
Out: scored posture report, prioritized gaps, remediation steps
ResponsePolicyAgent
policy
In: proposed action + risk context
Out: policy decision via assertActionPermitted() — auto / approval / prohibited + rationale
ResearchAgent
research
In: incident bundle, entity identifiers
Out: evidence-backed narrative citing Microsoft docs, for analyst review

07

Architecture Decision Records

ADR-0001
Monorepo — Turborepo + npm workspaces
Apps and packages share types via a single workspace. Type changes in pkgs/shared surface as compile errors in all consumers immediately. Rejected: separate repos (too much friction), Nx (more complexity than needed).
ADR-0002
Anthropic SDK direct — no Pi framework
Spec cited Pi ecosystem but those references link to unstable/unclear repos. Direct Anthropic SDK with tool use gives full control. No external agent framework dependency in v1.
ADR-0003
Rule-based pre-filter before agent invocation
At volume, running an LLM per event is cost-prohibitive. Pre-filter classifies pass / flag / ambiguous. Only ambiguous events reach the agent. Target: under 10% of events require LLM reasoning.
ADR-0004
Env-var secret management in v1
Graph credentials (client_id, client_secret) live in .env only. DB stores non-secret runtime state. Migration path to HashiCorp Vault in v2 requires no schema change.
ADR-0005
Single-tenant architecture — no MSP multi-tenancy
Initial deployment is Chelsea Piers or dev Entra tenant. Multi-tenancy (tenants + connectors tables, tenant_id FKs, Redis namespace) is straightforward to add in v2 without breaking the current schema.
